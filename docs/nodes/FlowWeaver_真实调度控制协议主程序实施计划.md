# FlowWeaver 真实调度控制协议主程序实施计划

更新时间：2026-07-07

## 文档定位

本文用于承接已经落地的预览控制节点，规划后续真实分支、真实循环和真实子工作流执行所需的主程序级协议。

当前控制节点已经能输出统一控制状态表，但 `actual_control` 固定为 `false`。真实调度不能靠某个节点 handler 私自修改执行路径，必须先补工作流定义、DAG、ready queue、controller、运行状态和测试协议。

## 当前稳定边界

预览控制状态表字段保持稳定：

```text
signal_type
signal_status
source_node_id
target_node_id
target_anchor
condition_result
selected_branch
action
actual_control
reason
details
```

当前已覆盖的预览节点：

| 节点 | 当前行为 |
|---|---|
| `JumpAnchorNode` | 输出锚点声明计划 |
| `UnconditionalJumpNode` | 输出跳转计划 |
| `ConditionalJumpNode` | 读取条件状态表并输出分支计划 |
| `LoopStartNode` | 输出循环计划 |
| `LoopJudgeNode` | 输出循环判断计划 |
| `SubWorkflowNode` | 输出子工作流计划 |

## 总体原则

1. 真实控制能力属于工作流调度协议，不属于单个节点业务逻辑。
2. 普通数据边和控制边需要在定义层分开表达，避免把控制语义塞进普通端口连接。
3. 节点 handler 继续只负责产出业务结果或控制状态，路径选择由 controller 统一解释。
4. `WorkflowDag` 不能直接放开任意环，循环必须有结构化边界、最大次数和可解释的退出状态。
5. 子工作流需要父子运行关系和输入输出映射，不能只靠一个表引用暗中启动。
6. 预览状态表继续保留，作为调试、审阅和未来真实控制的解释来源。

## 阶段一：工作流定义协议

目标：让 workflow definition 能描述控制边，而不是只靠普通 `connections` 表达。

建议新增或扩展的定义：

| 能力 | 说明 |
|---|---|
| `control_edges` | 描述条件边、跳转边、循环边、子工作流边 |
| `edge_kind` | `data` / `control` / `condition` / `loop` / `subworkflow` |
| `branch_key` | true、false、default、continue、end 等分支名 |
| `target_anchor` | 控制边可引用锚点 |
| `loop_id` | 循环起点和判断节点之间的结构化绑定 |
| `subworkflow_ref` | 子工作流模板或工作流定义引用 |

验收点：

| 项目 | 标准 |
|---|---|
| 定义校验 | 能区分数据端口错误和控制边错误 |
| 向后兼容 | 旧 `connections` 继续按普通 DAG 执行 |
| 锚点校验 | 可检查重复锚点、缺失锚点和不可达目标 |

## 阶段二：DAG 与控制图拆分

目标：保持普通数据依赖仍为 DAG，同时允许受控控制结构。

建议拆成两层：

| 层级 | 职责 |
|---|---|
| 数据 DAG | 保持无环，用于普通节点依赖和输入表准备 |
| 控制图 | 记录分支、跳转、循环和子工作流关系 |

循环不能直接放开为普通图环，应先落成结构化循环块：

```text
LoopStartNode -> 子流程主体 -> LoopJudgeNode
LoopJudgeNode continue -> LoopStartNode
LoopJudgeNode end -> 后续节点
```

验收点：

| 项目 | 标准 |
|---|---|
| 普通 DAG | 旧流程拓扑排序结果不变 |
| 控制图 | 能独立校验控制目标和循环边界 |
| 循环保护 | 必须有最大迭代次数和退出条件 |

## 阶段三：ready queue 与 controller

目标：让调度器统一解释控制结果，而不是节点自己调度下游。

需要补充的运行状态：

| 状态 | 说明 |
|---|---|
| `SKIPPED` | 未选中分支或控制路径不进入 |
| `WAITING_CONTROL` | 等待控制结果决定是否进入 |
| `CONTROLLED_READY` | 控制结果已允许进入 ready |

controller 需要处理：

1. 节点成功后读取控制状态表或控制事件。
2. 根据 `selected_branch` 选择可进入的控制边。
3. 将未选分支节点标记为可解释的跳过状态。
4. 重新计算 workflow 完成条件，接受合法跳过后的完成。
5. 保持普通数据依赖必须满足后才能进入执行。

验收点：

| 项目 | 标准 |
|---|---|
| 分支选择 | true/false/default 路径只执行被选路径 |
| 下游跳过 | 未选路径不会卡住 workflow 完成 |
| 错误定位 | 控制结果无效时能指向源节点和目标边 |

## 阶段四：真实循环执行

目标：在结构化循环块内支持受控重复执行。

需要新增的运行记录：

| 字段 | 说明 |
|---|---|
| `loop_id` | 循环实例标识 |
| `iteration_index` | 当前迭代序号 |
| `max_iterations` | 最大迭代次数 |
| `loop_state` | running、ending、ended、failed |
| `current_input_ref` | 当前轮输入快照 |
| `exit_reason` | 条件结束、达到上限、失败等 |

执行规则：

1. 每轮使用明确输入快照，避免上游表在循环中被隐式覆盖。
2. `LoopJudgeNode` 的 continue/end 结果只由 controller 解释。
3. 达到最大次数时必须退出并给出明确原因。
4. 循环体失败时按工作流失败策略处理，不静默进入下一轮。

验收点：

| 项目 | 标准 |
|---|---|
| 最大次数 | 能阻止无限循环 |
| 结果可查 | 每轮输入、输出和退出原因可追踪 |
| 普通节点隔离 | 非循环流程不受影响 |

## 阶段五：真实子工作流执行

目标：让 `SubWorkflowNode` 能通过统一协议启动并汇总子工作流。

需要补充：

| 能力 | 说明 |
|---|---|
| 父子 `WorkflowRun` 关系 | 记录 parent_run_id、parent_node_run_id、child_run_id |
| 输入映射 | 将父节点输入表映射为子工作流入口 |
| 输出映射 | 将子工作流输出回填为父节点输出 |
| 状态汇总 | 子运行成功、失败、取消后映射到父节点状态 |
| 资源隔离 | 子运行中转表和父运行中转表边界清晰 |

第一版真实子工作流可以先限制：

1. 不支持任意深度嵌套。
2. 不允许子工作流直接修改父流程的普通 DAG。
3. 输出只支持显式命名结果。

验收点：

| 项目 | 标准 |
|---|---|
| 父子关系 | API 可查询父节点和子运行关系 |
| 输入输出 | 映射错误能在启动前发现 |
| 失败传播 | 子运行失败能让父节点进入明确失败状态 |

## 阶段六：迁移与兼容

目标：让已有工作流不受真实控制协议引入影响。

执行顺序：

1. 先只增加定义字段和校验，不改变旧运行语义。
2. 增加预览节点到真实控制的显式开关或 workflow 级能力标记。
3. 新协议工作流单独走控制调度路径。
4. 等控制调度测试稳定后，再允许 UI 暴露真实控制能力。

兼容要求：

| 项目 | 标准 |
|---|---|
| 旧流程 | 没有控制边的流程执行结果不变 |
| 预览节点 | 仍可作为普通节点输出状态表 |
| API | 节点定义和表数据读取接口保持兼容 |

## 性能与耦合评估

| 项目 | 影响 |
|---|---|
| 普通表节点性能 | 无直接影响 |
| 预览控制节点性能 | 仍为小状态表输出，影响很低 |
| 真实分支 | controller 需要维护路径状态，影响中等 |
| 真实循环 | 取决于迭代次数和表快照大小，影响较高 |
| 真实子工作流 | 取决于子运行数量、输入输出映射和运行隔离，影响较高 |
| 主程序耦合 | 会增加调度核心复杂度，但应集中在通用控制协议，不进入具体节点 handler |

## 测试矩阵

| 类型 | 覆盖内容 |
|---|---|
| 单元测试 | 定义校验、控制图构建、循环边界 |
| 集成测试 | 条件分支、未选路径跳过、循环退出、子工作流成功和失败 |
| 回归测试 | 旧普通 DAG、预览控制节点、运行内写入节点 |
| API 测试 | 控制边定义、父子运行查询、表引用读取 |
| 压力测试 | 大表循环、多分支、多子工作流并发 |

## 下一步落点

当前最稳的下一步不是打开真实调度，而是先做协议设计和测试骨架：

1. 定义 `control_edges` 草案和校验错误码。
2. 为 `WorkflowDag` 增加数据图与控制图拆分的设计文档或空实现测试。
3. 为 controller 路径状态写最小状态机测试。
4. 在真实执行开启前，继续保持所有预览控制节点 `actual_control=false`。
