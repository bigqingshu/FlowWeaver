# 节点方案：SubWorkflowNode

更新时间：2026-07-05

## 基本信息

DataFlowKit 来源：`core.group` / 节点组 / 子工作流

FlowWeaver 暂定类型名：`SubWorkflowNode` / `CompositeNode`

节点方向：流程控制 / 工作流组合

优先级：P4

当前状态：规划中，代码未实现

## 要解决的问题

将一组子节点封装为一个可复用的子工作流或复合节点。

用户看到的能力：在一个节点中配置子流程输入、输出和内部节点列表。

不解决的内容：第一版不支持任意嵌套循环，不把 DataFlowKit 的组内 `transit_tables` 直接搬进 FlowWeaver。

## 输入输出

输入：一个或多个 `TableRef`，以及输入字段映射。

输出：子流程最终输出 `TableRef`，或命名输出集合。

## 配置项草案

| 字段 | 含义 |
|---|---|
| `group_name` | 组名 |
| `nodes` | 子节点定义 |
| `input_source_type` | 输入来源 |
| `input_fields` / `input_mapping` / `input_defaults` | 输入字段映射 |
| `missing_input_policy` | 缺失输入处理 |
| `transit_scope` | 中转数据隔离策略 |
| `allow_loop_nodes` | 是否允许循环节点 |
| `main_output_mode` | 主输出模式 |
| `save_to_transit` / `output_transit_name` | 命名输出 |
| `save_to_sqlite` | 外部保存，后置 |

## 数据契约

子工作流必须有明确输入快照和输出引用。

如果映射为父子 `WorkflowRun`，需要记录父 run、父 node run、子 run 的关系。

## 执行模式

普通运行：后置规划。

预览运行：需要定义是否只运行子流程上游闭包。

支持取消：必须支持，取消父节点应能取消子流程。

支持进度：必须支持，子流程进度需要折算或汇总。

## 副作用与确认

副作用与外部资源说明：改变运行结构；子节点副作用由子节点自身说明。

是否需要用户确认：后置讨论；如果子流程包含外部写入节点，必须确认。

## 主程序交互边界

不推荐让 `WorkflowRunProcess` 直接识别组节点并解释子节点。

推荐先补通用父子 run 或复合节点执行能力，由节点通过标准上下文创建子运行。

不依赖 UI 专用组编辑器作为执行前提。

## 运行记录

结果摘要建议包含子节点数、子 run id、输入映射摘要、输出引用摘要。

`RuntimeEvent` 需要记录子流程开始、完成、失败和取消。

## 验收方式

子流程可读取父节点输入并输出结果。

父节点失败时错误可指向子节点。

取消父流程时子流程被取消或进入可解释状态。

## 实现前置依赖

需要父子 `WorkflowRun` 关系。

需要子流程输入/输出映射协议。

需要复合节点进度汇总规则。

## 简要模板补齐

节点名称：节点组与子工作流。

节点定位：流程控制 / 工作流组合。

优先级：P4。

当前状态：规划中，代码未实现。

要解决的问题：见上文“要解决的问题”章节。

用户看到的能力：见上文“用户看到的能力”描述。

第一版不解决的内容：见上文“不解决的内容”描述。

注册参数：
- node_type：SubWorkflowNode
- node_version：1.0
- plugin_id：core
- provider_type：builtin
- category：流程控制
- ui_visibility：visible
- enabled：规划期为 false；实现和验收后再按节点成熟度设为 true。
- display_name：节点组与子工作流
- config_schema：沿用本文“配置项草案”，后续落到统一 config_schema。
- input_ports：按循环或子工作流配置接收 TableRef、循环上下文或子工作流输入绑定。
- output_ports：输出循环上下文、当前项 TableRef、子工作流结果或控制状态。
- implementation_ref：builtin.SubWorkflowNode（暂定内部执行入口，后续实现时绑定真实实现；不对普通 UI 暴露）。

输入说明：见上文“输入输出”章节；第一版按 input_ports 约束接收数据。

输出说明：见上文“输入输出”章节；输出必须使用标准引用或标准运行摘要。

配置说明：见上文“配置项草案”；配置只描述节点自身能力，不绑定具体 UI 控件。

数据流转方式：通过标准端口和运行上下文传递循环/子工作流状态；不在 WorkflowRunProcess 中增加节点专用回跳分支，需接入通用循环或动态调度能力。

是否支持取消：必须支持；每轮或每批次检查取消，并保证循环状态可解释。

是否支持进度上报：必须支持，记录当前轮次、总轮次、当前项和中止原因。

节点反馈：反馈循环初始化、每轮进度、子工作流阶段、完成、失败、取消和是否达到上限。

节点心跳：必须保留低频心跳，每轮或每批次刷新 NodeRun.last_heartbeat。

心跳查询方式：通过 NodeRun.last_heartbeat、status、progress、current_stage、error 和 RuntimeEvent 查询节点活动状态。

后台极简模式：后台极简模式保留循环开始、每批次低频进度、结束、失败、取消、心跳和最终摘要，避免记录每个子节点的重复细节。

外部资源与副作用：沿用本文“副作用与确认”章节；权限、审计、字段级追踪不写入默认节点方案。

性能影响等级：高。

主要性能消耗点：循环状态维护、重复调度、子工作流执行、结果合并和恢复状态记录。

失败场景：循环边界配置错误、最大次数超限、循环状态不可恢复、子工作流失败或取消后状态不一致。

失败提示：提示 loop_id、当前轮次、失败子节点、最大次数、恢复状态和建议缩小循环范围。

验收方式：沿用本文“验收方式”章节，并补充校验运行记录、取消、进度、心跳和后台极简模式是否符合本节约束。
