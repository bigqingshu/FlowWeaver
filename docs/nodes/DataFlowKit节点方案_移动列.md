# 节点方案：ReorderColumnsNode

更新时间：2026-07-05

## 基本信息

DataFlowKit 来源：`core.move_columns` / 移动列

FlowWeaver 暂定类型名：`ReorderColumnsNode`

节点方向：数据处理 / 表结构变换

优先级：P0

当前状态：规划中，代码未实现

## 要解决的问题

调整输入表字段顺序，输出字段顺序符合配置的新表。

用户看到的能力：拖拽或填写字段顺序，运行后得到列顺序调整后的表。

不解决的内容：不改字段名，不删除字段，不修改外部表结构。

## 输入输出

输入：一个 `TableRef`。

输出：一个新的 `TableRef`。

输出行数不变，单元格值不变，只改变字段顺序。

## 配置项草案

| 字段 | 含义 |
|---|---|
| `order` | 目标字段顺序 |
| `missing_policy` | 配置中字段不存在时跳过、警告或报错 |
| `unlisted_policy` | 未列出字段保留到末尾、丢弃或报错 |

## 数据契约

节点根据输入 schema 和 `order` 计算输出 schema，再按输出字段顺序重组每一行。

第一版建议默认保留未列出字段到末尾，避免配置遗漏导致数据丢失。

## 执行模式

支持普通运行和预览运行。

支持取消：建议支持。

支持进度：低优先级，大表可按批次上报。

## 副作用与确认

副作用与外部资源说明：无外部副作用，只写本次运行输出。

是否需要用户确认：不需要。

## 主程序交互边界

节点不直接访问 UI 排序状态，UI 只负责把目标顺序写入 config。

节点通过通用运行上下文读取输入表和发布输出表。

不要求主程序新增节点专用判断。

## 运行记录

结果摘要建议包含输出字段顺序、未列出字段数、缺失字段警告。

`NodeRun.current_stage` 可记录“计算字段顺序”“发布输出表”。

## 验收方式

字段顺序按配置输出。

未列出字段按默认策略保留到末尾。

字段不存在时按 `missing_policy` 返回预期结果。

## 实现前置依赖

需要字段顺序配置 schema。

需要通用表格节点执行入口。

是否支持取消：支持；短耗时场景可在批处理边界检查取消，大表处理时按行或分批检查。

是否支持进度上报：支持低频进度上报，至少记录当前阶段、已处理数量和结果数量。

节点反馈：通过 NodeRun、RuntimeEvent 和结果摘要反馈开始、运行中、完成、失败、取消；摘要包含输入数量、输出数量、跳过数量和错误数量。

节点心跳：通常不强制主动心跳，开始和结束状态即可；大表结构复制时可按批次刷新 NodeRun.last_heartbeat。

心跳查询方式：通过 NodeRun.last_heartbeat、status、progress、current_stage、error 和 RuntimeEvent 查询节点活动状态。

后台极简模式：保留开始、结束、失败、取消和最终结果摘要；关闭逐行日志和高频 UI 刷新，大表场景保留低频进度和心跳。

外部资源与副作用：沿用本文“副作用与确认”章节；权限、审计、字段级追踪不写入默认节点方案。

性能影响等级：低。

主要性能消耗点：主要是字段结构调整和表头元数据处理，数据复制量随表宽变化。

失败场景：输入表不可读、字段不存在、配置不完整、数据类型不符合预期。

失败提示：提示缺失字段、非法配置项或失败阶段，并给出可修正的字段名、配置名和样例值。

验收方式：沿用本文“验收方式”章节，并补充校验运行记录、取消、进度、心跳和后台极简模式是否符合本节约束。
