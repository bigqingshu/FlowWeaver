# FlowWeaver 节点方案简要撰写模板

更新时间：2026-07-06

## 文档用途

本模板用于后续逐个讨论、规划和审核节点方案。

使用时直接复制“节点方案模板”部分，按当前节点情况填写即可。

当前模板只写节点大方向、节点契约和实现边界，不写具体代码实现。

## 填写原则

- 先写用户能用它解决什么问题，再写输入、输出和配置。
- 节点只描述自己的能力，不把业务逻辑写进主程序。
- 节点不得直接依赖 `EngineHost`、桌面 UI、主程序全局服务或 `RuntimeStore` 内部实现。
- 权限、审计、字段级追踪暂时不写入默认节点方案。
- `implementation_ref` 只表示内部执行入口，不是用户配置，也不对普通 UI 暴露。
- `implementation_path` 不再使用。

## 栏目说明

### 节点名称

写用户能看懂的节点名称，例如“高级筛选”“批量替换”“获取文件列表”。

### 节点定位

说明这个节点属于哪类能力，例如：

```text
数据输入
数据处理
数据输出
流程控制
运行辅助
插件扩展
```

### 优先级

建议使用：

```text
P0：基础能力，很多节点会依赖它。
P1：常用主线能力，优先实现。
P2：提升配置、预览或调试体验。
P3：提升稳定性、资源控制或后台运行能力。
P4：高级扩展，后续再讨论。
```

### 当前状态

建议使用：

```text
未实现
部分实现
已实现
需要补充
后置讨论
```

### 注册参数

这里对应统一节点注册表的第一版 12 个核心参数。

| 参数 | 通俗说明 |
| --- | --- |
| `node_type` | 系统识别节点用的唯一类型名。 |
| `node_version` | 节点版本。 |
| `plugin_id` | 节点来源，内置节点也可以有固定来源标识。 |
| `provider_type` | 节点来源类型，例如内置、开发测试、插件。 |
| `category` | 节点分类，用于列表分组和筛选。 |
| `ui_visibility` | 是否在普通 UI 中显示。 |
| `enabled` | 当前是否启用。 |
| `display_name` | 用户看到的名称。 |
| `config_schema` | 节点配置结构。 |
| `input_ports` | 节点需要哪些输入。 |
| `output_ports` | 节点会产出哪些输出。 |
| `implementation_ref` | 内部执行入口引用，由执行器用来找到节点实现。 |

### 输入说明

写清楚节点需要什么数据进来。

例如：

```text
需要一个表数据输入。
可以没有输入，由节点自己产生数据。
需要多个输入表。
需要文件路径或外部资源引用。
```

### 输出说明

写清楚节点会产出什么结果。

例如：

```text
输出一张新表。
输出修改后的表引用。
输出文件列表。
输出判断结果。
不输出数据，只影响流程走向。
```

### 配置说明

只写用户需要设置什么，不写具体控件实现。

例如：

```text
选择列名。
填写匹配条件。
选择写入方式。
填写目标路径。
选择失败时是否继续。
```

### 数据流转方式

说明节点通过什么方式拿数据、交数据。

建议写成：

```text
通过标准输入引用读取数据。
通过标准输出引用交付结果。
不直接读写主程序内部 Store。
不直接控制工作流调度。
```

### 节点反馈

说明节点运行过程中会反馈哪些信息。

建议写成：

```text
状态变化：开始、运行中、完成、失败、取消。
进度信息：百分比、当前阶段、已处理数量。
结果摘要：输出数量、命中数量、跳过数量、错误数量。
错误信息：失败原因和用户可修正的配置项。
```

节点反馈应走通用运行通道，例如 `NodeRun`、`RuntimeEvent` 和节点结果摘要，不应写入权限审计或主程序专属结构。

### 节点心跳

说明节点是否需要主动心跳。

建议写成：

```text
短耗时节点：不强制主动心跳，开始和结束状态即可。
长耗时节点：必须保留低频心跳。
循环节点：每轮或每批次保留低频心跳。
外部程序节点：必须保留心跳，用于判断外部程序是否长时间无响应。
插件节点：建议必须支持心跳，避免插件卡住时不可判断。
```

心跳只表示“节点任务还在运行边界内活动”，不表示业务结果已经成功。

### 心跳查询方式

说明用户或 UI 后续怎么查看节点是否还在活动。

当前建议写成：

```text
通过节点运行状态查询 `NodeRun.last_heartbeat`。
通过节点运行状态查询 `status`、`progress`、`current_stage` 和 `error`。
通过 RuntimeEvent 查看节点进度和关键运行事件。
```

第一版可以只记录最近心跳时间。后续如需要，再补充 `healthy`、`stale`、`lost` 这类计算后的健康状态。

### 后台极简模式

说明后台运行时是否可以少记录、少刷新、少上报。

建议写成：

```text
保留开始、结束、失败和最终结果摘要。
关闭逐行日志、详细中间状态和高频进度刷新。
长耗时节点可保留低频进度。
心跳不建议关闭，只降低频率。
```

### 外部资源与副作用

说明节点是否会影响工作流外部的东西。

建议使用：

```text
无外部副作用。
只读取外部资源。
写入外部文件或数据库。
启动外部程序。
拉起其他工作流。
```

### 性能影响等级

建议使用：

```text
低：普通表结构处理，数据量不大。
中：会扫描较多行、较多列或多个输入。
高：会读写大文件、访问外部程序、批量复制数据或长时间循环。
```

### 失败提示

失败提示要让用户知道下一步怎么处理。

建议写成：

```text
哪项配置不正确。
哪个输入不存在。
哪个外部资源无法访问。
处理到哪一步失败。
用户可以如何修正。
```

## 最小填写示例

```text
节点名称：高级筛选

节点定位：数据处理

优先级：P1

当前状态：待规划

要解决的问题：让用户按条件筛选表数据，不需要手动写复杂逻辑。

用户看到的能力：选择输入表、设置筛选条件、输出筛选后的新表。

第一版不解决的内容：不做跨数据库查询，不做权限审计。

注册参数：
- node_type：AdvancedFilterNode
- node_version：1.0
- plugin_id：core
- provider_type：builtin
- category：数据处理
- ui_visibility：visible
- enabled：true
- display_name：高级筛选
- config_schema：筛选条件配置
- input_ports：一个表输入
- output_ports：一个表输出
- implementation_ref：内部执行入口，后续实现时填写

输入说明：读取一个上游表数据引用。

输出说明：输出筛选后的新表引用。

配置说明：选择列、比较方式、比较值、多个条件关系。

数据流转方式：通过标准输入引用读取，通过标准输出引用返回。

是否支持取消：支持，按行或分批检查取消状态。

是否支持进度上报：支持低频进度。

节点反馈：反馈开始、当前筛选阶段、已处理行数、输出行数、失败原因。

节点心跳：大表筛选时保留低频心跳；小表筛选可只依赖开始和结束状态。

心跳查询方式：通过 `NodeRun.last_heartbeat`、`progress`、`current_stage` 和 RuntimeEvent 查看。

后台极简模式：只保留开始、结束、失败和结果摘要。

外部资源与副作用：无外部副作用。

性能影响等级：中。

主要性能消耗点：大表逐行筛选。

失败场景：列不存在、条件配置不完整、输入表不可读。

失败提示：提示缺失的列名或配置项。

验收方式：给定输入表和条件后，输出结果行数与预期一致。

后续扩展：支持复杂条件分组和条件模板复用。
```
