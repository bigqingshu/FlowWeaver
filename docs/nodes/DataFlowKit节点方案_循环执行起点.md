# 节点方案：LoopStartNode

更新时间：2026-07-05

## 基本信息

DataFlowKit 来源：`core.loop_start` / 循环执行起点

FlowWeaver 暂定类型名：`LoopStartNode`

节点方向：流程控制 / 循环

优先级：P4

当前状态：后端预览版已实现。已进入默认注册表，输出统一控制状态表；不改变 DAG 执行路径，不执行真实循环。

## 要解决的问题

第一版用于生成循环计划，记录输入行数、最大循环次数和计划迭代次数。

用户看到的能力：配置循环 ID、来源类型、循环字段、最大循环次数和当前项输出名，并产出可预览的循环计划。

不解决的内容：当前 FlowWeaver DAG 第一版不直接支持 program counter 式回跳，因此本节点不执行真实循环，不重复调度下游节点。

## 输入输出

输入：可选单个 `TableRef`。

输出：一个 `status` 控制状态表。

## 配置项草案

| 字段 | 含义 |
|---|---|
| `loop_id` | 循环 ID |
| `source_type` | 当前表、SQLite 表、命名表 |
| `fields` | 循环项字段 |
| `flag_field` | 执行标志字段 |
| `init_flag_mode` | 初始标志策略 |
| `boundary_mode` | 数据边界 |
| `reference_field` | 参考字段 |
| `current_table_name` | 当前循环项输出名 |
| `output_current_as_table` | 是否输出当前项表 |
| `max_loop_count` | 最大循环次数 |

## 数据契约

第一版不创建循环运行状态，不输出当前循环项表，只输出统一控制状态表。

```text
signal_type=loop_plan
signal_status=planned
target_anchor=<loop_id>
action=declare_loop_plan
actual_control=false
```

## 执行模式

普通运行：生成循环计划状态表。

预览运行：记录输入行数、最大循环次数和计划迭代次数。

支持取消：必须支持。

支持进度：必须支持，按当前循环次数上报。

## 副作用与确认

副作用与外部资源说明：第一版不改变运行结构，无外部写入。

是否需要用户确认：后置讨论；循环次数大或包含外部写入节点时必须确认。

## 主程序交互边界

不应让当前 DAG 调度器临时增加节点专用回跳逻辑。

应先设计通用循环/动态调度能力，再接入本节点。

## 运行记录

结果表包含 `loop_id`、总项数、最大循环次数、计划迭代次数和 `actual_control=false`。

RuntimeEvent 需要记录循环开始、每轮进度、循环结束和异常中止。

## 验收方式

后端能注册 `LoopStartNode`。

运行后能输出一行 `loop_plan` 控制状态表。

`loop_id` 为空时返回验证错误。

## 实现前置依赖

预览版已无需 DAG 循环或动态调度设计。

真实循环仍需要循环状态持久化或可恢复策略。

节点名称：循环执行起点。

节点定位：流程控制 / 循环。

优先级：P4。

当前状态：后端预览版已实现，真实循环语义后置。

要解决的问题：见上文“要解决的问题”章节。

用户看到的能力：见上文“用户看到的能力”描述。

第一版不解决的内容：见上文“不解决的内容”描述。

注册参数：
- node_type：LoopStartNode
- node_version：1.0
- plugin_id：core
- provider_type：builtin
- category：流程控制
- ui_visibility：visible
- enabled：后端预览版已可注册使用；真实循环仍后置。
- display_name：循环执行起点
- config_schema：沿用本文“配置项草案”，后续落到统一 config_schema。
- input_ports：可选 `in`。
- output_ports：`status`，输出统一控制状态表。
- implementation_ref：builtin.LoopStartNode（暂定内部执行入口，后续实现时绑定真实实现；不对普通 UI 暴露）。

输入说明：见上文“输入输出”章节；第一版最多接收一个输入引用，仅用于统计预览计划。

输出说明：见上文“输入输出”章节；输出必须使用标准引用或标准运行摘要。

配置说明：见上文“配置项草案”；配置只描述节点自身能力，不绑定具体 UI 控件。

数据流转方式：只输出标准控制状态表；不在 WorkflowRunProcess 中增加节点专用回跳分支，真实循环需接入通用循环或动态调度能力。

是否支持取消：必须支持；每轮或每批次检查取消，并保证循环状态可解释。

是否支持进度上报：必须支持，记录当前轮次、总轮次、当前项和中止原因。

节点反馈：反馈循环初始化、每轮进度、子工作流阶段、完成、失败、取消和是否达到上限。

节点心跳：必须保留低频心跳，每轮或每批次刷新 NodeRun.last_heartbeat。

心跳查询方式：通过 NodeRun.last_heartbeat、status、progress、current_stage、error 和 RuntimeEvent 查询节点活动状态。

后台极简模式：后台极简模式保留循环开始、每批次低频进度、结束、失败、取消、心跳和最终摘要，避免记录每个子节点的重复细节。

外部资源与副作用：沿用本文“副作用与确认”章节；权限、审计、字段级追踪不写入默认节点方案。

性能影响等级：高。

主要性能消耗点：循环状态维护、重复调度、子工作流执行、结果合并和恢复状态记录。

失败场景：循环边界配置错误、最大次数超限、循环状态不可恢复、子工作流失败或取消后状态不一致。

失败提示：提示 loop_id、当前轮次、失败子节点、最大次数、恢复状态和建议缩小循环范围。

验收方式：沿用本文“验收方式”章节，并补充校验运行记录、取消、进度、心跳和后台极简模式是否符合本节约束。
