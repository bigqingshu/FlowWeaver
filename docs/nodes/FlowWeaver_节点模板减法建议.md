# FlowWeaver 节点模板减法建议

> 文档用途：在按节点规划核心模板逐个分析节点前，先明确哪些能力当前阶段保留，哪些能力后置，哪些能力暂时移出当前阶段。
> 使用原则：先保证节点能稳定新增、能配置、能运行、能失败可查，再考虑高级调试、血缘、插件和复杂运行策略。

## 一、减法后的总体原则

当前阶段不要把节点体系一次做成完整平台生态。

先保留最能支撑新增节点落地的能力：

- 节点怎么接入。
- 节点怎么配置。
- 节点怎么读取输入。
- 节点怎么产出输出。
- 节点怎么取消。
- 节点怎么上报基础进度。
- 节点失败后怎么给用户看得懂的提示。
- 节点是否有副作用、是否访问外部资源、性能大概重不重。

暂时不追求：

- 完整血缘。
- 单元级追踪。
- 完整插件生态。
- 复杂后台运行策略。
- 复杂断点和二分调试。

## 二、当前阶段优先保留

### 1. 通用节点实现接口

通俗说明：给所有后续节点一个统一接入口。

作用：

- 后续新增节点不用每次改主执行器分流逻辑。
- 节点可以按统一方式实现输入、输出、执行、失败处理。
- 方便测试和验收。

建议：保留，优先级最高。

### 2. NodeRuntimeContext

通俗说明：节点运行时拿到的统一工具箱。

第一版只需要保留：

- 读取输入。
- 发布输出。
- 检查取消。
- 上报粗粒度进度。
- 返回错误摘要。

不需要第一版就做复杂资源、血缘、完整调试接口。

建议：保留，但做薄。

### 3. 节点方向分类

通俗说明：标明节点属于输入、处理、输出、共享、调试、资源访问等哪一类。

作用：

- 节点多了以后便于查找。
- UI 可以按分类展示。
- 后续规划时能看出节点生态是否偏科。

建议：保留，成本低。

### 4. 节点描述

通俗说明：一句话告诉用户这个节点干什么。

作用：

- 节点目录里更容易理解。
- 用户不用靠内部类型名猜用途。

建议：保留，成本低。

### 5. 是否支持取消

通俗说明：声明这个节点运行中能不能被停止。

作用：

- 长时间节点需要优先支持取消。
- 后台运行时可以更安全地停止任务。
- UI 可以显示用户是否能取消。

建议：保留。

### 6. 是否支持进度上报

通俗说明：声明这个节点会不会告诉系统自己跑到哪一步了。

建议分级：

```text
不需要：执行很快
粗粒度：开始、处理中、完成
详细：分阶段、百分比、关键计数
```

建议：保留，但默认只要求粗粒度。

### 7. 副作用等级

通俗说明：说明节点会不会修改外部东西。

建议分级：

```text
NONE：无副作用
READ_EXTERNAL：只读外部资源
WRITE_RUNTIME：只写本次运行数据
WRITE_EXTERNAL：会写外部文件、数据库或系统
CALL_EXTERNAL：会调用外部服务或程序
```

作用：

- 判断是否需要用户确认。
- 判断是否允许自动重试。
- 判断后台运行时风险。

建议：保留。

### 8. 性能影响等级

通俗说明：标明节点大概重不重。

建议分级：

```text
低
中
高
极高
```

作用：

- 帮助后续决定是否默认预览。
- 帮助用户识别慢节点。
- 给后台运行策略留基础。

建议：保留，但只做人工声明，不做复杂资源估算。

### 9. 外部资源与副作用声明

通俗说明：节点提前说清楚自己是否会访问或修改工作流外部资源。

第一版只需要支持：

```text
无外部资源
只读外部资源
写入外部文件或数据库
调用外部程序或服务
需要用户确认的副作用
```

建议：保留，先做简单声明；复杂授权、审批和安全策略后置到节点方案或独立能力。

### 10. 用户友好失败提示

通俗说明：节点失败时，不只给底层异常，而是给用户能理解的话。

第一版建议每个节点至少整理：

- 配置错误提示。
- 输入缺失提示。
- 外部资源不可用或用户未确认提示。
- 输出失败提示。
- 取消或超时提示。

建议：保留。

## 三、当前阶段暂时后置

### 1. 后台极简运行策略

说明：文档中可以继续保留“后台极简是否适用”的判断，但程序层面的 `BACKGROUND_MINIMAL` 策略后置。

后置原因：

- 需要工作流级策略、节点级覆盖、事件降级、详细记录降级等配套。
- 现在先保证节点能稳定新增和运行更重要。

### 2. 节点级记录覆盖

说明：例如整个工作流默认最少记录，但某个慢节点单独开详细记录。

后置原因：

- 需要运行历史分析和策略合并。
- 当前先保留基础 NodeRun 状态即可。

### 3. 完整 NodeResultModel 落地

说明：协议里有 warnings、metrics、diagnostics、change_set_summary、side_effect_summary 等字段。

当前阶段建议：

- 主链路先保留成功、失败、输出引用、错误摘要。
- 复杂诊断结果后置。

### 4. DataRef

说明：当前主线以 TableRef 为主，DataRef 可用于未来承载文件、文本、对象等非表数据。

后置原因：

- 当前节点主线如果以表类处理为主，TableRef 已够用。
- DataRef 会扩大数据模型范围。

### 5. ChangeSet

说明：记录节点对数据结构、字段、值域造成的变化。

当前阶段建议：

- 先不做完整 ChangeSet。
- 如有需要，只做轻量摘要。

### 6. 组节点完整运行模型

说明：把一组节点封装为可复用节点，并拥有自己的输入、输出、状态和生命周期。

后置原因：

- 依赖普通节点体系稳定。
- 会引入嵌套调度和资源边界问题。

### 7. 插件生命周期

说明：插件节点如何安装、启动、复用、取消、销毁。

后置原因：

- 当前先做好内置节点和统一节点接口。
- 插件生态会明显增加复杂度。

## 四、当前阶段建议移出

### 1. 单元级追踪

说明：追踪每个单元格从哪里来、怎么变。

移出原因：

- 存储量和索引成本高。
- 对第一阶段新增节点不是必需。

### 2. 完整字段血缘

说明：完整追踪每个字段经过哪些节点产生和变化。

移出原因：

- 比节点级输入输出关系复杂很多。
- 可以等节点体系稳定后再设计。

### 3. 数据二分定位

说明：长流程出错时自动反复重跑来定位问题。

移出原因：

- 需要缓存、局部重跑、运行快照配套。
- 当前阶段成本偏高。

### 4. 条件断点

说明：预览运行时满足某个条件就暂停。

移出原因：

- 更像高级调试器功能。
- 不影响节点基础运行。

### 5. FULL / DEBUG 默认详细诊断

说明：默认记录非常详细的调试信息、诊断信息和运行细节。

移出原因：

- 后台运行性能压力大。
- 当前只保留 RuntimeEvent、NodeRun 状态和必要错误信息。

## 五、减法后的第一版节点定义建议

后续如果要让程序字段和节点模板更贴合，第一版只建议补这些轻量字段：

```text
category
description
tags
supports_cancel
supports_progress
side_effect_level
performance_level
external_resource_summary
```

其中：

```text
category：节点方向分类
description：一句话说明
tags：搜索和筛选标签
supports_cancel：是否支持取消
supports_progress：是否支持进度上报
side_effect_level：副作用等级
performance_level：性能影响等级
external_resource_summary：外部资源与副作用摘要
```

暂时不要第一版加入：

```text
background_record_profile
node_record_override
field_lineage_mode
cell_trace_mode
full_change_set
plugin_lifecycle
group_node_lifecycle
```

## 六、减法后的开发顺序建议

建议顺序：

```text
1. 补节点定义轻量元数据
2. 补通用 NodeRunner 接口
3. 补薄版 NodeRuntimeContext
4. 把现有内置节点迁移或适配到统一接口
5. 按节点规划核心模板逐个分析新节点
6. 选择 P0 / P1 节点进入实现
```

这样做的好处：

- 不会一开始就陷入高级调试和复杂运行策略。
- 后续新增节点有统一入口。
- 节点目录和配置体验可以逐步增强。
- 后台极简、节点级记录覆盖、血缘追踪可以等节点体系稳定后再进入。

## 七、最终保留清单

当前阶段真正保留：

```text
通用节点实现接口
薄版 NodeRuntimeContext
节点方向分类
节点描述
节点标签
是否支持取消
是否支持进度上报
副作用等级
性能影响等级
外部资源与副作用声明
用户友好失败提示
```

当前阶段明确后置：

```text
后台极简运行策略
节点级记录覆盖
完整 NodeResultModel
DataRef
ChangeSet
组节点完整运行模型
插件生命周期
```

当前阶段先移出：

```text
单元级追踪
完整字段血缘
数据二分定位
条件断点
FULL / DEBUG 默认详细诊断
```
