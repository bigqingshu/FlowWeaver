# FlowWeaver 工作流启动节点想法草案

> 文档用途：记录“工作流拉起工作流”的节点想法，供后续继续讨论和取舍。
> 当前状态：想法草案，不进入实现，不涉及代码改动。
> 核心原则：平台只负责拉起工作流；外部程序是否拉起，由具体节点根据运行来源自行判断。

## 一、想法来源

后续工作流变多后，可能会出现这种需求：

```text
一个工作流完成某些准备工作
-> 拉起另一个工作流继续处理
-> 下游工作流根据自己的节点逻辑继续运行
```

这个能力可以让复杂流程拆成多个小工作流，避免所有逻辑都挤在一个巨大工作流里。

## 二、当前决定

当前只考虑：

```text
拉起另一个工作流
```

暂不考虑由平台统一拉起外部程序。

外部程序相关逻辑交给具体节点自己判断和执行。

## 三、为什么不让平台统一拉起程序

平台统一拉起外部程序会带来很多额外复杂度：

- 程序路径从哪里来。
- 参数怎么传。
- 程序是否安全。
- 是否允许后台自动执行。
- 程序卡住后怎么处理。
- 取消工作流时是否终止程序。
- 程序退出码怎么解释。
- 日志怎么收集。

这些问题每类程序都不同，不适合当前阶段由平台统一处理。

所以更简单的方向是：

```text
平台负责工作流之间的启动关系
节点负责自己是否需要启动外部程序
```

## 四、核心机制

建议给每次工作流运行增加“运行来源”概念。

例如：

```text
MANUAL：用户手动启动
WORKFLOW_TRIGGER：由其他工作流启动
PROGRAM_TRIGGER：由外部程序启动，先预留
SCHEDULED：定时启动，先预留
```

同时记录父级来源：

```text
parent_workflow_run_id
parent_node_run_id
```

这样下游节点可以知道：

```text
这次运行是谁启动的
是不是另一个工作流拉起的
是不是外部程序已经启动后才进入的
```

## 五、节点如何利用运行来源

节点可以根据运行来源决定是否拉起外部程序。

例如：

```text
如果 run_origin == MANUAL：
    节点可以提示用户确认是否启动外部程序

如果 run_origin == WORKFLOW_TRIGGER：
    节点可以按后台逻辑运行，避免重复弹窗

如果 run_origin == PROGRAM_TRIGGER：
    说明外部程序可能已经启动了工作流
    节点可以不再重复启动同一个程序
```

这个设计可以避免重复拉起。

例如：

```text
程序 A 启动工作流 X
工作流 X 里有节点原本会启动程序 A
```

如果没有运行来源判断，可能会重复启动程序 A。

有运行来源后，节点可以判断：

```text
本次工作流已经来自程序 A
-> 不再启动程序 A
```

## 六、工作流启动节点第一版设想

节点方向：

```text
工作流控制
```

节点名称暂定：

```text
启动工作流节点
```

第一版只做：

```text
选择目标工作流
用户手动确认
启动目标工作流
记录 run_origin = WORKFLOW_TRIGGER
记录 parent_workflow_run_id
记录 parent_node_run_id
返回 child_workflow_run_id
```

第一版不做：

```text
自动拉起外部程序
自动等待子工作流完成
自动读取子工作流输出
复杂参数传递
循环依赖自动处理
自动重试
后台无确认执行
```

## 七、用户确认

第一版必须要求用户手动确认。

原因：

- 防止误启动大量工作流。
- 防止工作流之间形成循环。
- 防止后台无感知地启动额外任务。

确认内容可以包括：

```text
将要启动哪个工作流
由哪个节点触发
是否会创建新的 WorkflowRun
是否只启动不等待
```

## 八、父子关系

启动后建议形成父子关系：

```text
父 workflow_run
父 node_run
子 workflow_run
```

这样后续可以追溯：

```text
这个工作流是谁拉起的
这个子工作流属于哪个父流程
哪个节点触发了它
```

## 九、第一版返回结果

第一版可以只返回：

```text
child_workflow_run_id
target_workflow_id
target_workflow_revision_id
started_at
```

是否需要把它包装成 TableRef，后续再讨论。

当前草案倾向于：

```text
先作为节点运行结果摘要记录
不强行进入表格数据流
```

## 十、风险点

### 1. 循环启动

例如：

```text
工作流 A 启动 B
工作流 B 又启动 A
```

第一版可以先通过用户确认控制，不做复杂自动检测。

后续再考虑最大启动深度或禁止直接循环。

### 2. 大量启动

如果一个节点在循环或批量处理中启动很多工作流，会造成运行压力。

第一版建议：

```text
一次节点运行只允许启动一个目标工作流
```

### 3. 父子失败关系

第一版建议：

```text
父工作流只负责启动成功
子工作流失败不反向影响父工作流
```

也就是说第一版“不等待子工作流完成”。

### 4. 后台自动执行

第一版不建议后台无确认启动其他工作流。

后台场景可以后续单独设计允许策略。

## 十一、后续可能扩展

后续可讨论：

```text
父工作流等待子工作流完成
子工作流失败后影响父节点状态
向子工作流传递参数
读取子工作流共享输出
限制最大启动深度
工作流启动白名单
工作流启动审计详情
PROGRAM_TRIGGER 来源正式接入
SCHEDULED 来源正式接入
```

## 十二、外部程序生命周期边界

外部程序生命周期不交给 EngineHost。

正常情况下，由节点自己管理它拉起的外部程序。

异常情况下，由 `WorkflowRunProcess` 级 Windows Job Object 做兜底清理。

这个边界的目标是：

```text
保持工作流进程隔离
避免主程序介入工作流内部程序细节
避免工作流被终止后留下孤儿外部程序
```

建议职责划分：

```text
EngineHost：
只管理工作流运行进程，不直接管理节点拉起的外部程序。

WorkflowRunProcess：
作为本次工作流的故障边界，持有或关联本次工作流的 Windows Job Object。

节点：
负责正常启动、状态判断、退出码、日志摘要、超时、取消和用户友好错误。

Windows Job Object：
只做兜底清理，用于工作流取消、超时、崩溃或节点来不及释放外部程序的情况。
```

推荐后续原则：

```text
需要被 FlowWeaver 管理生命周期的外部程序，必须通过节点运行上下文启动。
```

也就是说，未来如果节点需要启动外部程序，理想入口不是让节点随意调用系统启动命令，而是通过类似：

```text
context.launch_process(...)
```

由这个入口负责：

```text
记录 launch_session_id
记录 pid
关联当前 workflow_run_id
关联当前 node_run_id
加入 WorkflowRunProcess 级 Job Object
提供取消和超时清理能力
```

这样可以形成两层保障：

```text
节点内部主动管理：
负责正常路径下的启动、等待、状态判断、日志和错误提示。

WorkflowRunProcess 级 Job Object：
负责异常路径下的进程树兜底清理。
```

该机制不要求第一版马上实现，只作为后续外部程序节点或程序型业务节点的边界原则。

## 十三、当前结论

这个方向可以保留。

当前阶段建议只做最小想法：

```text
平台只拉起工作流
节点自行判断是否拉起外部程序
WorkflowRun 记录运行来源和父子关系
第一版必须用户确认
第一版只启动，不等待
```

这个设计简单、可解释，也能为后续工作流组合和程序联动留下空间。
