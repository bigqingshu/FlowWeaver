# FlowWeaver 节点规划核心模板

> 文档用途：后续逐个分析、规划和拆解节点时套用。
> 使用原则：只写单个节点本身需要说明的内容，不写全局运行策略、不写 UI 页面落点、不写具体代码实现。
> 适用对象：新增节点、已有节点补全、节点能力升级、节点验收方案。
> 核心边界：节点与主程序不能有任何直接关联。节点不得直接调用、持有、依赖或反向控制 EngineHost、桌面 UI、主窗口状态、全局服务对象或主程序内部实现。

## 一、填写原则

- 先说明节点要解决什么问题，再说明输入、输出和配置。
- 能用通俗语言说明的，不先写技术细节。
- 节点必须只通过标准数据引用、节点配置、运行上下文和结果协议与平台交互。
- 节点不得直接访问主程序对象、UI ViewModel、EngineHost 内部服务、全局 Store 实例或主程序生命周期。
- 节点需要启动外部程序时，也应由节点运行上下文提供受控入口，不得让 EngineHost 直接管理节点内部程序。
- 每个节点都要能判断是否适合后台极简运行。
- 每个节点都要说明是否有副作用、是否访问外部资源、失败时怎么提示。
- 性能只写影响等级和主要消耗点，不展开全局调度策略。
- 验收标准要能直接变成后续测试或人工检查清单。

## 二、节点与主程序隔离原则

这是节点体系的核心概念。

节点不能直接关联主程序。

也就是说，节点不应该知道：

```text
EngineHost 怎么启动
桌面 UI 当前在哪个页面
MainWindowViewModel 里有什么状态
主程序内部服务对象怎么组织
RuntimeStore 的具体实例从哪里来
主程序如何管理工作流进程
```

节点只应该知道：

```text
自己的配置
自己的输入引用
自己的输出要求
自己的运行上下文
自己的取消状态
自己的进度上报入口
自己的错误返回方式
```

推荐边界：

```text
主程序：
负责工作流管理、调度、进程监督、运行事件记录和 UI 展示。

WorkflowRunProcess：
负责本次工作流运行、节点调度、节点任务提交和故障边界。

节点：
只负责处理自己的输入、配置、输出和错误，不直接控制主程序。
```

如果节点需要平台能力，只能通过受控接口：

```text
NodeTask
NodeRuntimeContext
TableRef / DataRef
NodeResult
RuntimeEvent
```

禁止方向：

```text
节点直接调用 EngineHost
节点直接操作桌面 UI
节点持有主程序全局服务
节点直接改 RuntimeStore 内部状态
节点绕过运行上下文启动并托管外部程序
节点反向控制工作流调度器
```

这样做的目的：

- 保持节点可测试。
- 保持主程序稳定。
- 保持工作流进程隔离。
- 避免节点把 EngineHost 变成大管家。
- 避免后续插件或外部程序污染主程序生命周期。
- 让节点失败时只影响自己的运行边界，而不是拖垮主程序。

## 四、字段说明

### 节点名称

写节点的明确名称。可以先用中文工作名，后续实现时再补充内部类型名。

### 节点方向

说明节点属于哪类能力，例如：

```text
数据输入
数据处理
数据输出
共享复用
运行辅助
调试检查
资源访问
```

### 优先级

建议使用：

```text
P0：节点基础能力或其他节点依赖它
P1：常用业务主线需要它
P2：提升配置、预览、调试体验
P3：提升安全、稳定性、资源控制
P4：高级生态或远期扩展
```

### 当前状态

建议使用：

```text
未实现
已实现
部分实现
需要补充
后置讨论
```

### 要解决的问题

用普通话说明这个节点为什么需要存在。重点写用户现在遇到的实际问题。

### 用户看到的能力

说明用户使用这个节点时能完成什么动作，不写内部代码怎么做。

### 不解决的内容

明确本节点第一版不做什么，避免范围越写越大。

### 输入

写节点需要什么输入。可以包括：

```text
是否需要上游 TableRef / DataRef
是否允许没有输入
是否支持多个输入
输入是否要求固定版本
输入数据的基本要求
```

### 输出

写节点会产出什么。可以包括：

```text
是否输出 TableRef / DataRef
是否输出多个结果
是否输出给下游节点
是否可发布或复用
失败时是否保留输出
```

### 配置项

写用户需要配置什么。只写配置含义，不写具体控件实现。

建议格式：

```text
配置名：
是否必填：
默认值：
说明：
```

### 数据契约

说明节点如何遵守 FlowWeaver 的数据传递规则。重点写：

```text
输入引用怎么读取
输出引用怎么发布
是否需要版本固定
是否需要原子发布
是否依赖共享表
是否会产生中间结果
```

### 执行模式

说明节点大概怎么运行。建议使用：

```text
快速节点
普通节点
长时间节点
外部资源节点
高风险节点
后台友好节点
调试专用节点
```

### 是否支持取消

说明节点运行中是否能响应取消。长时间节点必须优先考虑取消。

### 是否支持进度上报

说明节点是否需要上报进度。

建议：

```text
不需要：执行很快
粗粒度：开始、处理中、完成
详细：分阶段、百分比、关键计数
```

### 是否有副作用

说明节点是否会修改外部状态，例如写文件、写数据库、调用外部系统、发送消息等。

### 外部资源与副作用声明

说明节点是否会访问或修改工作流外部资源。最少写：

```text
无外部资源
只读外部资源
写入外部文件或数据库
调用外部程序或服务
需要用户确认的副作用
```

### 后台极简模式是否保留

说明后台最省性能运行时，这个节点是否仍然启用。

建议：

```text
保留
保留但降级
默认关闭
不适用
```

### 后台极简模式保留内容

说明后台极简运行时保留哪些最小行为，例如只保留状态、错误摘要、原子发布。

### 后台极简模式关闭内容

说明后台极简运行时关闭哪些高成本行为，例如预览、详细指标、字段追踪、完整调试日志。

### 性能影响等级

建议使用：

```text
低
中
高
极高
```

### 性能消耗点

说明主要消耗来自哪里，例如：

```text
读取大量数据
写入大量结果
频繁上报进度
访问外部系统
重复执行
生成预览
保存调试记录
```

### 性能控制方式

写控制成本的办法，例如：

```text
抽样
分页
限制行数
限制字段数
限制重试次数
异步写日志
后台模式关闭详细记录
```

### 失败场景

列出最常见的失败原因。

### 失败提示

写用户应该看到的通俗提示，不直接暴露底层异常。

### 验收方式

写这个节点怎样算完成。至少包含：

```text
正常输入能成功
错误输入能失败并提示
取消时能退出
后台极简模式能运行或正确跳过
输出能被下游识别
```

### 后续扩展

写第一版之后可以再加什么，但不要塞进第一版范围。

## 五、最小示例

```text
节点名称：示例节点

节点方向：数据处理

优先级：P1

当前状态：未实现

要解决的问题：用户需要把上游数据做一次标准处理，交给下游继续使用。

用户看到的能力：选择上游结果，配置处理规则，运行后得到新的结果。

不解决的内容：第一版不做高级调试、不做单元级追踪、不做复杂自动重试。

输入：一个上游 TableRef。

输出：一个新的 TableRef，成功后原子发布。

配置项：处理规则、是否保留原字段、失败时是否停止。

数据契约：读取固定输入版本，输出成功后发布新引用，失败时不发布半成品。

执行模式：普通节点。

是否支持取消：支持。

是否支持进度上报：粗粒度。

是否有副作用：无外部副作用。

外部资源与副作用声明：无外部资源，只写本次运行结果。

后台极简模式是否保留：保留。

后台极简模式保留内容：状态、错误摘要、输入输出引用、原子发布。

后台极简模式关闭内容：预览、详细调试日志、字段级追踪。

性能影响等级：中。

性能消耗点：读取和写入数据。

性能控制方式：限制预览，后台模式不生成预览缓存。

失败场景：输入为空、配置无效、输出写入失败。

失败提示：节点处理失败，请检查输入数据和处理规则。

验收方式：正常输入成功；错误配置失败并提示；取消可退出；输出可被下游识别。

后续扩展：增加详细预览、质量检查、字段级摘要。
```
