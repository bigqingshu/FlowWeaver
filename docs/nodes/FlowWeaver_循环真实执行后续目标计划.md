# FlowWeaver 循环真实执行后续目标计划

更新时间：2026-07-08

## 目标定位

本文用于承接 `FlowWeaver_循环真实执行大白话实施方案.md` 后续执行。

当前已经完成的是循环真实执行的后端地基：

- 循环区域协议和校验。
- `LoopRun` / `LoopIterationRun` / `LoopIterationTableRef` 存储与迁移。
- 串行循环状态推进骨架。
- 恢复状态检查 helper。
- 循环运行只读后端查询接口。

还没有完成的是主程序真实调度接入：

- 还不会在节点成功后读取控制状态表并真正创建下一轮。
- 还不会为每一轮物化当前轮输入表。
- 还不能让同一个循环体节点在不同轮次留下独立运行归属。
- 完成判断、失败、取消、恢复还没有接入主流程。

本目标计划只处理后端，不修改前端 UI。

## 硬边界

1. 不碰前端 UI 文件。
2. 不把循环回线塞进普通 `connections`。
3. 不把主程序写成 `if node_type == LoopJudgeNode`。
4. 主程序只解释通用控制协议。
5. 旧工作流默认仍保持预览语义，真实循环必须显式启用。
6. 第一版只支持串行循环，不支持并行轮次。
7. 第一版不支持嵌套循环，不支持循环体内真实子工作流。
8. 不靠拼接 `node_instance_id` 表示轮次。
9. 不直接把循环字段塞进 `TableRef`。
10. 每完成一个阶段后，必须补对应测试。

## 性能与耦合优化补充约束

本节作为后续实现时的补充硬边界。后续阶段可以按实际代码状态微调落点，但不能突破这些方向。

### 约束一：当前轮输入默认懒物化

循环每一轮不要提前批量生成所有未来输入表。

默认只记录：

```text
source_table_ref_id
row_index / row_range / selector
materialized_input_table_ref_id
```

只有当某一轮真的要派发循环体入口节点时，才物化当前轮输入表。

这样可以避免：

```text
1 万行输入 -> 1 万张提前生成的小表
```

也可以避免循环中途取消或失败后，留下大量永远不会使用的中间表。

### 约束二：不一次性创建全部未来轮次

第一版串行循环只创建当前需要的轮次。

允许：

```text
第 1 轮结束并判断 continue -> 创建第 2 轮
```

不允许：

```text
循环启动时按 max_iterations 一次性创建 100 轮
```

这样可以减少无效记录，也能让失败、取消、恢复逻辑更简单。

### 约束三：循环体节点记录按需创建

循环继续时，不要一次性创建循环体内所有节点的未来运行记录。

第一版建议只创建当前轮入口节点或当前轮必要节点，后续仍按普通依赖推进。

如果后续需要预创建整轮节点，也必须保证：

- 不创建未来轮次节点。
- 重复恢复不会重复创建同一轮节点。
- 未进入的节点不会影响工作流完成判断。

### 约束四：控制解释层必须独立

控制结果解释不能散落到多个主流程文件里。

主流程只做类似：

```text
node succeeded -> try interpret control signal -> apply control decision
```

控制解释层内部负责：

- 读取节点输出表。
- 解析通用控制字段。
- 判断 `actual_control`。
- 调用循环推进器。

主流程不能知道某个具体节点类型，也不能直接解析 `LoopJudgeNode` 配置。

### 约束五：完成判断独立成 evaluator

工作流完成判断不要继续堆在 controller 内部。

建议形成独立判断层：

```text
WorkflowCompletionEvaluator
```

它统一判断：

- 普通节点是否完成。
- 循环总记录是否终态。
- 已创建轮次是否终态。
- 是否还有 ready / queued / running 的节点或轮次任务。

这样后续子工作流、批处理、局部预览也能复用同一类完成判断思路。

### 约束六：恢复查询必须走索引

恢复逻辑不能每次全量读取所有轮次再取最后一条。

需要优先支持这些查询：

```text
按 loop_run_id 查询最新 iteration
按 loop_run_id 查询 RUNNING iteration
按 loop_run_id + iteration_index 查询指定轮次
按 loop_iteration_id + role 查询表引用
```

涉及的表必须保留或补齐索引：

```text
loop_iteration_runs(loop_run_id, iteration_index)
loop_iteration_runs(loop_run_id, status)
loop_iteration_table_refs(loop_iteration_id, role)
```

如果发现现有 RuntimeStore 查询会变成全量扫描，应先补存储接口和测试，再接入恢复流程。

### 约束七：事件默认摘要级

循环可能跑很多轮，不能默认把每个内部动作都写成事件。

默认只记录摘要事件：

```text
循环开始
每轮开始
每轮结束
循环结束
循环失败
循环取消
达到最大次数
```

详细事件只在诊断配置打开时记录，例如：

```text
每轮节点详细进度
每轮输入输出表摘要
每轮判断详情
```

后台快速运行时，应优先减少事件量。

### 约束八：第一版优先最小关联表

完整 `execution_scope` 是更通用的方向，但改动面较大。

第一版如果要降低风险，可以优先使用最小关联表：

```text
loop_iteration_node_runs
```

先解决循环轮次和节点运行记录的归属问题。

等循环真实执行稳定后，再评估是否抽象成完整：

```text
execution_scope
```

不能为了追求一次性通用化，把主流程、NodeRun、API 和 UI 同时大改。

### 约束九：出口 ready 仍走统一依赖评估

循环结束后，不能简单写成：

```text
loop ended -> 直接把某个出口节点改 READY
```

更稳的方式是把“循环区域已终态”作为依赖条件之一，再交给统一 ready 判断。

这样可以避免出口节点还有其他普通上游未完成时被提前执行。

### 约束十：真实执行开关最后打开

只有在以下能力都完成后，才允许打开真实循环：

- 轮次节点归属。
- 当前轮输入物化。
- 控制结果解释。
- 下一轮节点创建。
- 出口放行。
- 完成判断。
- 失败取消收口。
- 恢复幂等。

在此之前，`control_protocol.mode=enabled` 和 `loop_region.enabled=true` 继续保持拒绝是合理保护。

## 总体路线

最稳路线是：

```text
先解决一轮里的节点归属
再解决一轮输入表
再接入控制结果解释
再升级完成/失败/取消/恢复
最后打开真实循环协议开关
```

这样可以避免主程序先开始回跳，但运行记录和输入表还说不清楚。

## 阶段一：循环轮次内节点运行归属

### 要解决的问题

现在 `NodeRun` 仍然基本按：

```text
workflow_run_id + node_instance_id
```

理解为一个工作流运行中，一个节点实例对应一条主要运行记录。

真实循环需要同一个节点在不同轮次执行多次，如果不先解决归属问题，后续会出现：

- 第 1 轮和第 2 轮节点记录混在一起。
- `get_node_run_for_instance` 查询不稳定。
- UI/API 无法回答某一轮里哪个节点失败。
- 恢复时不知道当前轮节点是否跑完。

### 解决方案

优先引入通用执行作用域或轮次节点关联，而不是直接在 `NodeRun` 上硬塞循环字段。

推荐方向：

```text
execution_scope
scope_type: root / loop_iteration / subworkflow
parent_scope_id
```

第一版可以先做最小实现：

```text
loop_iteration_node_runs
loop_iteration_id
node_run_id
node_instance_id
role/status
```

如果暂时不改 `NodeRun` 查询模型，也要保证能按轮次查到节点运行记录。

### 验收标准

- 能记录某个 `NodeRun` 属于哪个 `LoopIterationRun`。
- 同一个 `node_instance_id` 可以在不同轮次有可区分记录。
- 普通非循环节点的查询行为不变。
- 有重复关联幂等保护。
- 有跨 workflow_run 关联拒绝测试。

## 阶段二：当前轮输入表物化

### 要解决的问题

现在循环记录里可以保存：

```text
input_table_ref_id
input_selector
```

但还没有真正按 selector 生成当前轮输入表。

如果循环体节点直接读整张原始表，就无法实现：

```text
第 1 轮处理第 1 行
第 2 轮处理第 2 行
```

### 解决方案

实现一个后端输入物化 helper。

第一版只支持按行：

```text
source_table_ref_id
row_index
materialized_input_table_ref_id
```

执行规则：

1. 根据 `input_selector.row_index` 读取源表一行。
2. 生成一张当前轮输入小表。
3. 将生成的 `TableRef` 绑定到 `LoopIterationTableRef(role=INPUT)`。
4. 如果已经存在同轮输入表，重复调用应返回已有结果。

### 验收标准

- 能把源表某一行物化成当前轮输入表。
- 能通过 `loop_iteration_table_refs` 查到该输入表。
- 重复物化同一轮不会生成多份输入表。
- 空表或越界行返回清晰错误。
- 不默认复制整张源表。

## 阶段三：控制结果解释接入节点成功路径

### 要解决的问题

现在 `LoopJudgeNode` 输出的控制状态表只是数据，不会影响调度。

真实循环需要在判断节点成功后：

```text
读取控制状态表
确认 actual_control=true
解释 selected_branch
决定继续下一轮或结束循环
```

### 解决方案

在节点成功后的后端路径中增加控制结果解释入口。

注意：入口只看通用控制协议字段：

```text
signal_type
selected_branch
actual_control
target_anchor
details
```

不能根据具体节点类型写死逻辑。

接入点建议在 `advance_after_node_success` 或其上层增加一个小的控制协议处理层：

```text
节点成功
记录普通节点成功
读取该节点最新成功结果输出表
如果输出控制信号且 actual_control=true
调用循环推进器
否则继续普通 DAG ready 恢复
```

### 验收标准

- 预览控制表 `actual_control=false` 不产生调度副作用。
- 非 `loop_decision` 信号不触发循环推进。
- `continue_loop` 会尝试创建下一轮。
- `end_loop` 会结束循环。
- 重复处理同一结果不会多创建一轮。
- 不出现节点类型硬编码。

## 阶段四：循环下一轮节点创建与派发

### 要解决的问题

即使循环状态推进器创建了下一轮记录，主程序也还不会让循环体节点在下一轮执行。

### 解决方案

基于阶段一的轮次节点归属，为下一轮创建循环体节点运行记录。

第一版规则：

```text
第 N 轮完成并判断 continue
创建第 N+1 轮
创建循环体节点运行记录
把循环体入口节点置为 READY
后续仍按普通依赖推进
```

### 验收标准

- `continue_loop` 后能创建下一轮循环体入口节点。
- 同一轮内节点仍按普通 DAG 依赖顺序执行。
- 不影响循环外普通节点运行。
- 达到最大次数时不再创建下一轮节点。

## 阶段五：循环结束后放行出口

### 要解决的问题

循环结束后，循环后续节点还不知道可以继续跑。

### 解决方案

循环区域需要明确出口节点或出口依赖规则。

当循环进入终态：

```text
ENDED
MAX_ITERATIONS_REACHED
```

主程序恢复循环出口或后续节点的 ready 判断。

### 验收标准

- `end_loop` 后循环出口节点可以进入 READY。
- 达到最大次数后按配置进入出口或失败。
- 循环未终态时出口节点不能提前 READY。
- 普通 DAG 完成判断不被提前触发。

## 阶段六：工作流完成判断升级

### 要解决的问题

现在工作流完成判断主要看固定 `NodeRun` 是否全部成功。

真实循环需要额外判断：

- 循环总记录是否终态。
- 已创建轮次是否都终态。
- 是否还有运行中/等待中轮次节点。

### 解决方案

升级完成判断为：

```text
普通非循环节点满足完成条件
所有 LoopRun 处于终态
所有已创建 LoopIterationRun 处于终态
没有 READY / QUEUED / RUNNING / LONG_RUNNING 的节点任务
```

第一版先只支持串行单层循环。

### 验收标准

- 循环结束前工作流不会成功。
- 循环结束且后续节点完成后工作流成功。
- 达到上限时按设计状态完成或失败。
- 没有循环的旧工作流完成行为不变。

## 阶段七：失败和取消收口

### 要解决的问题

循环体节点失败时，不能只标记普通节点失败，还要说清楚：

- 哪个循环失败。
- 第几轮失败。
- 哪个节点失败。
- 后续轮次是否停止创建。

取消也一样，不能取消后继续创建下一轮。

### 解决方案

第一版失败规则：

```text
循环体任一节点失败 -> 当前轮失败
当前轮失败 -> 循环失败
循环失败 -> 按工作流失败策略处理
```

取消规则：

```text
工作流取消 -> 当前运行中轮次取消
循环取消 -> 不再创建后续轮次
```

### 验收标准

- 循环体节点失败会标记当前轮 FAILED。
- 循环总记录标记 FAILED。
- 记录 `failed_node_run_id`。
- 工作流取消时循环总记录和当前轮进入 CANCELLED。
- 取消后不会创建下一轮。

## 阶段八：恢复逻辑接入主流程

### 要解决的问题

程序崩溃或重启后，需要知道循环处在什么状态。

典型情况：

- 判断节点成功，但还没创建下一轮。
- 下一轮创建了一半。
- 当前轮运行中但进程丢失。
- 循环已经终态。

### 解决方案

将 `inspect_serial_loop_state` 接入恢复流程。

恢复策略：

| 状态 | 处理 |
|---|---|
| NOT_STARTED | 可重新启动第 1 轮 |
| ACTIVE_ITERATION_RUNNING | 交给现有进程/节点丢失处理 |
| WAITING_FOR_DECISION | 幂等补推进 |
| LOOP_TERMINAL | 不再创建轮次 |
| BLOCKED_BY_FAILED_ITERATION | 收口循环失败 |
| INCONSISTENT_STATE | 标记错误或进入人工诊断 |

### 验收标准

- 重复恢复不会多创建轮次。
- 判断结果重复处理不会多推进。
- 终态循环不会被恢复成运行中。
- 运行中轮次丢失有清晰错误记录。

## 阶段九：打开真实循环开关

### 要解决的问题

当前校验仍拒绝：

```text
control_protocol.mode=enabled
loop_region.enabled=true
```

这是合理的保护。只有主调度接入完成后，才能打开。

### 解决方案

在前面阶段通过测试后，再修改校验：

```text
允许 enabled
但继续拒绝嵌套循环
拒绝循环体内真实子工作流
拒绝不支持的 input_mode
```

旧流程仍默认 preview。

### 验收标准

- `preview` 工作流行为不变。
- `enabled` 工作流可以通过校验。
- 不支持结构给出清晰校验错误。
- 真实执行只在显式 enabled 时发生。

## 阶段十：完整集成测试矩阵

### 必测场景

| 类型 | 场景 |
|---|---|
| 成功 | 单轮后 end_loop |
| 成功 | 多轮 continue 后 end_loop |
| 上限 | 一直 continue 直到 max_iterations |
| 输入 | 按行物化当前轮输入表 |
| 失败 | 循环体节点失败 |
| 失败 | 判断节点失败 |
| 取消 | 当前轮运行中取消 |
| 恢复 | 判断成功后崩溃，恢复后只创建一轮 |
| 恢复 | 重复处理同一判断结果 |
| 协议 | actual_control=false 无副作用 |
| 协议 | 非法 selected_branch 拒绝 |
| 完成 | 循环未结束时工作流不完成 |
| 完成 | 循环结束且后续节点完成后工作流完成 |

## 最终验收标准

第一版真实循环完成时，应满足：

```text
识别一个标准循环区域
创建循环总记录
创建第 1 轮记录
准备第 1 轮输入表
执行循环体
读取判断结果
continue 时创建下一轮
end 时跳出循环
达到最大次数时停止
循环结束后执行后续节点
失败和取消能按轮次收口
恢复后不会重复创建轮次
API 能按循环和轮次查记录
旧预览流程不受影响
```

## 建议下一个 Codex 目标

建议下一轮目标从阶段一和阶段二开始：

```text
只改后端，实现循环轮次内节点运行归属与当前轮输入表物化；
完成后提交并推送；
暂不接入真实回跳主调度；
不修改前端 UI。
```

这样下一阶段做完后，真实回跳接入主流程会更稳。
