# 节点方案：ConditionFlagNode

更新时间：2026-07-07

## 基本信息

DataFlowKit 来源：`core.condition_check` / 条件判断节点

FlowWeaver 暂定类型名：`ConditionFlagNode`

节点方向：流程控制 / 条件结果

优先级：P4

当前状态：第一版已实现。当前后端已注册 `ConditionFlagNode`，支持作为普通结果节点输出条件状态表；不改变 DAG 执行路径，不驱动跳转、循环或条件边。

## 要解决的问题

根据输入表状态或字段值生成一个条件结果，供后续条件跳转或条件边使用。

用户看到的能力：配置条件表达式，运行后得到 TRUE/FALSE 或自定义标志。

不解决的内容：第一版可只输出条件结果，不直接驱动调度跳转。

## 输入输出

输入：一个 `TableRef`。

输出：一个条件状态 `TableRef`。第一版不透传输入表，不写运行上下文私有标志。

## 配置项草案

| 字段 | 含义 |
|---|---|
| `flag_name` | 条件标志名 |
| `condition_type` | 表行数、字段是否存在、字段值 |
| `field` | 字段 |
| `operator` | 操作符 |
| `value` | 比较值 |
| `value_source` | 比较值来源，固定值、当前行字段值 |
| `value_field` | 字段值来源 |
| `aggregation` | 聚合方式：`any`、`all`、`first`、`count` |
| `case_sensitive` | 是否区分大小写 |
| `true_value` / `false_value` | 成立和不成立输出值 |

## 动态值来源补充

条件判断节点需要支持字段对字段判断，而不是只支持字段对固定值判断。

实际使用中可能需要：

```text
判断字段 A 是否等于字段 B。
判断字段 A 是否包含字段 B 当前行值。
判断字段 A 是否大于由字段 B 和字段 C 计算出的值。
```

第一版值来源已支持：

```text
固定值：value 来自配置。
字段值：value 来自当前行指定字段。
```

指定单元格和表达式值可后置，用于复合计算。

## 数据契约

条件结果已落为标准 `TableRef`，不写进进程内 dict。

当比较值来自字段时，条件判断已明确聚合语义：任意行满足、全部行满足、首行判断或命中计数。

当前状态表字段：

| 字段 | 含义 |
|---|---|
| `flag_name` | 条件标志名 |
| `condition_type` | 条件类型 |
| `aggregation` | 聚合方式 |
| `result` | `true` / `false` |
| `true_value` | 成立输出值 |
| `false_value` | 不成立输出值 |
| `output_value` | 按判断结果选择的最终输出值 |
| `matched_count` | 命中行数 |
| `total_rows` | 输入总行数 |
| `details` | JSON 摘要 |

如果后续驱动调度，需要与条件边模型对齐。

## 执行模式

支持普通运行和预览运行。

支持取消：低优先级。

支持进度：当前沿用节点执行器基础运行状态；第一版不额外上报节点内细粒度进度。

## 副作用与确认

副作用与外部资源说明：无外部副作用；只生成条件结果。

是否需要用户确认：不需要。

## 主程序交互边界

节点不直接要求调度器跳转。

第一版已作为数据/结果节点实现，后续如需驱动路径选择，再接入通用条件边或控制信号协议。

## 运行记录

结果状态表包含条件类型、判断结果、输出标志、命中数、总行数和 JSON 摘要。

RuntimeEvent 可记录条件判断结果。

## 验收方式

表行数、字段是否存在、字段值条件均可正确判断。

字段值条件支持固定值和字段值两类比较值来源。

字段不存在时按条件类型返回错误或 FALSE。

结果可被后续节点读取。

当前测试覆盖：行数 true/false、字段存在 true/false、固定值比较、当前行字段比较、`any` / `all` / `first` / `count` 聚合、缺字段、缺配置和非法操作符。

## 后续调度前置依赖

第一版的条件结果引用已通过状态 `TableRef` 满足。

如果后续要驱动调度，还需要条件边模型或统一控制信号协议。

要解决的问题：见上文“要解决的问题”章节。

用户看到的能力：见上文“用户看到的能力”描述。

- node_type：ConditionFlagNode

- plugin_id：core

- display_name：条件判断节点
- config_schema：已落到统一 config_schema。

- output_ports：`status`，输出条件状态 `TableRef`。
- implementation_ref：builtin.ConditionFlagNode。

输入说明：见上文“输入输出”章节；第一版要求一个输入表。

输出说明：见上文“输入输出”章节；当前输出使用标准表引用。

配置说明：见上文“配置项草案”；配置只描述节点自身能力，不绑定具体 UI 控件。

数据流转方式：只产出标准条件结果表；真实跳转必须由通用调度能力解释，不让主程序为单个跳转节点写专用分支。

是否支持取消：支持；通常为短耗时节点，执行前后检查取消即可。

是否支持进度上报：当前使用节点基础运行状态；细粒度扫描进度后置。

节点反馈：反馈条件求值、判断结果、命中数量、失败和取消。

节点心跳：短耗时控制节点不强制主动心跳；复杂条件或等待调度时可刷新 NodeRun.last_heartbeat。

心跳查询方式：通过 NodeRun.last_heartbeat、status、progress、current_stage、error 和 RuntimeEvent 查询节点活动状态。

后台极简模式：后台极简模式保留开始、判断结果、失败和取消，不记录高频细节。

外部资源与副作用：沿用本文“副作用与确认”章节；权限、审计、字段级追踪不写入默认节点方案。

性能影响等级：低到中。

主要性能消耗点：字段值条件需要扫描输入表；行数和字段存在条件成本较低。

失败场景：条件配置不完整、字段不存在、比较值来源字段不存在、非法操作符、不可比较的数据类型。

失败提示：提示缺失的条件字段、非法操作符、缺少比较值或不可比较的数据类型。

验收方式：沿用本文“验收方式”章节，第一版已补后端集成测试和节点定义 API 测试。
