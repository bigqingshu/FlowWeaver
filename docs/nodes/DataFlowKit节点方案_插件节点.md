# 节点方案：PluginNode

更新时间：2026-07-07

## 基本信息

DataFlowKit 来源：`core.plugin` / `plugin.<plugin_id>` / 插件节点

FlowWeaver 暂定类型名：`PluginNode`

节点方向：插件生态 / 外部扩展

优先级：P4

当前状态：已部分实现。当前后端已注册 `PluginNode`，支持配置 schema、状态表输出和 `plugin_manifest` 执行前校验；真实插件发现、加载和执行 runner 尚未打开。

## 要解决的问题

允许外部插件以节点形式接入工作流，提供内置节点之外的输入、处理或输出能力。

用户看到的能力：从插件目录加载插件节点，填写插件参数，运行后得到插件输出。

不解决的内容：第一版不直接兼容 DataFlowKit 全部插件协议，不默认允许插件任意写外部资源。

## 输入输出

输入：取决于插件声明，可为 `TableRef`、DataRef、文件路径或参数。

输出：取决于插件声明，可为 `TableRef`、DataRef、文件、日志或状态摘要。

## 配置项草案

| 字段 | 含义 |
|---|---|
| `plugin_id` | 插件 ID |
| `plugin_version` | 插件版本 |
| `params` | 插件参数 |
| `input_bindings` | 输入绑定 |
| `output_bindings` | 输出绑定 |
| `plugin_manifest` | 插件清单草稿，用于当前阶段执行前校验，后续可替换为正式插件发现器读取结果 |
| `execution_mode` | 进程内或外部进程，后置 |
| `allow_external_actions` | 是否允许外部副作用，后置 |
| `enable_execute` | 是否请求真实执行；当前即使开启也只进入执行前检查，不调用插件 runner |

## 数据契约

插件必须通过清单声明输入、输出、配置 schema、副作用和运行方式。

插件输出必须转成 FlowWeaver 标准结果引用和结果摘要。

## 执行模式

普通运行：当前阶段只做执行前检查并输出状态表，不实际运行插件代码。

预览运行：插件需声明是否支持；不支持时应拒绝而不是假执行。

支持取消：外部进程插件必须支持取消或超时清理。

支持进度：插件可通过标准 RuntimeEvent 或 NodeTask 进度通道上报。

## 副作用与确认

副作用与外部资源说明：取决于插件，可能读取文件、写数据库、调用外部程序或网络。

是否需要用户确认：高风险插件必须确认；插件清单应声明副作用等级。

## 主程序交互边界

主程序不理解插件业务逻辑。

插件由通用插件运行器加载和执行，不能直接调用 `EngineHost`、UI 或 `RuntimeStore` 内部结构。

不继承 DataFlowKit 的权限审计体系，改用副作用声明、用户确认和运行记录。

## 运行记录

结果摘要建议包含插件 ID、版本、输入输出摘要、日志摘要、外部动作摘要。

RuntimeEvent 应记录插件启动、进度、日志、完成、失败和取消。

## 验收方式

插件清单可被加载并生成节点定义。

参数校验失败时不会执行插件。

外部进程插件超时或取消时能清理。

## 实现前置依赖

需要插件清单协议。

需要插件依赖隔离和外部进程运行策略。

需要副作用声明和用户确认机制。

## 简要模板补齐

节点名称：插件节点。

节点定位：插件生态 / 外部扩展。

优先级：P4。

当前状态：已部分实现，当前只做执行前清单校验和状态表输出，真实插件执行后置。

要解决的问题：见上文“要解决的问题”章节。

用户看到的能力：见上文“用户看到的能力”描述。

第一版不解决的内容：见上文“不解决的内容”描述。

注册参数：
- node_type：PluginNode
- node_version：1.0
- plugin_id：插件清单声明的 plugin_id；内置插件桥接节点可用 core。
- provider_type：plugin 或内置插件运行器声明的来源类型。
- category：插件生态
- ui_visibility：visible
- enabled：规划期为 false；实现和验收后再按节点成熟度设为 true。
- display_name：插件节点
- config_schema：沿用本文“配置项草案”，后续落到统一 config_schema。
- input_ports：按插件清单和 input_bindings 声明，可接收 TableRef、DataRef、文件路径或普通参数。
- output_ports：按插件清单声明，输出必须转成标准 TableRef、DataRef、文件引用或状态摘要。
- implementation_ref：builtin.PluginNode（暂定内部执行入口，后续实现时绑定真实实现；不对普通 UI 暴露）。

输入说明：见上文“输入输出”章节；第一版按 input_ports 约束接收数据。

输出说明：见上文“输入输出”章节；输出必须使用标准引用或标准运行摘要。

配置说明：见上文“配置项草案”；配置只描述节点自身能力，不绑定具体 UI 控件。

数据流转方式：插件只通过清单、参数和标准绑定交换数据，不直接调用 EngineHost、桌面 UI 或 RuntimeStore 内部结构。

是否支持取消：必须支持超时和取消；外部进程插件取消时需要清理子进程或临时资源。

是否支持进度上报：插件应通过标准 RuntimeEvent 或节点进度通道上报阶段、日志摘要和完成状态。

节点反馈：当前已反馈插件清单状态、ID/版本匹配、输入输出绑定、必填参数、外部动作门控、执行就绪、跳过原因和校验错误；后续真实 runner 再补插件加载、执行阶段、标准日志摘要、输出摘要、失败、超时和取消。

节点心跳：建议强制保留心跳，避免插件卡住时不可判断；外部进程插件必须刷新 NodeRun.last_heartbeat。

心跳查询方式：通过 NodeRun.last_heartbeat、status、progress、current_stage、error 和 RuntimeEvent 查询节点活动状态。

后台极简模式：后台极简模式保留插件 ID、版本、开始、结束、失败、取消、低频日志摘要、心跳和输出摘要。

外部资源与副作用：沿用本文“副作用与确认”章节；权限、审计、字段级追踪不写入默认节点方案。

性能影响等级：高。

主要性能消耗点：插件加载、依赖初始化、外部进程通信、插件自定义计算或外部资源访问。

失败场景：插件清单无效、依赖缺失、参数校验失败、插件超时、输出不符合协议或外部资源访问失败。

失败提示：提示插件 ID、版本、缺失依赖、失败阶段、超时状态和插件返回的可修正配置项。

验收方式：沿用本文“验收方式”章节，并补充校验运行记录、取消、进度、心跳和后台极简模式是否符合本节约束。
## 后续扩展
