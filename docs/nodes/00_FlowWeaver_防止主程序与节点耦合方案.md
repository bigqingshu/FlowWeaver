# FlowWeaver 防止主程序与节点耦合方案

更新时间：2026-07-06

## 文档定位

这是节点体系规划的 00 级别边界文档。

当前项目还处于开发阶段，允许已有实现存在临时耦合，但后续新增节点和整理节点模板时，需要先遵守这里的方向，避免主程序因为节点越来越多而越来越重。

## 核心原则

节点与主程序不能有直接关联。

主程序只负责工作流生命周期、节点调度、通用运行上下文、状态记录、事件转发和通用数据流转。具体节点做什么、怎么处理数据、是否启动外部程序、需要哪些节点内部资源，都应该由节点自己声明和实现。

主程序不应该持续增加类似“如果是某某节点，就走某某逻辑”的判断。

默认主程序权限/审计模块已经移除并后置。主程序不再负责权限申请、授权句柄、AuditEvent、审计 API 或审计页面。节点访问文件、外部程序、网络或其他资源时，应在节点方案内说明，由节点自身实现和管理。

## 当前已发现的耦合点

| 主要依赖点 | 当前情况 | 耦合程度 | 后续处理 |
| --- | --- | --- | --- |
| 内置节点注册表 | 主程序仍通过 `default_registry` 暴露内置节点定义，例如测试表、过滤、共享表发布/读取、延迟测试和故障测试节点。 | 中 | 短期保留；后续改成节点清单、节点包或插件注册，不让主程序长期硬写节点清单。 |
| Workflow 校验 | 校验 workflow 里的 `node_type`、端口和连接是否合法。 | 低 | 应保留；这是主程序需要知道的节点契约，不是节点业务逻辑。 |
| NodeTask 分发 | 主程序按 `node_type`、配置和输入引用创建任务并分发。 | 低 | 应保留；这是通用执行接口，不应加入节点专属逻辑。 |
| WorkflowRunProcess 执行器选择 | 当前仍会按内置表节点、共享表节点选择对应执行器。 | 中 | 后续收口到统一节点执行入口或执行器工厂，避免新增节点时继续修改运行进程。 |
| 内置节点执行器 | 当前仍有内置表节点、共享表节点、故障测试节点和 Fake 默认执行器。 | 中 | 短期作为核心自带节点和测试能力保留；后续真实节点应逐步移到独立节点包/插件包，测试执行器与正式路径分开。 |
| TableRef / SharedPublication | 节点通过表引用和共享发布传递数据，主程序负责存储和查询。 | 低 | 应保留；这是当前数据中转核心，不属于耦合问题。 |
| RuntimeEvent / NodeRun | 节点进度、状态、失败和日志走 RuntimeEvent、NodeRun 或节点结果。 | 低 | 应保留；这是调试模式和后台运行都需要的基础诊断能力。 |
| UI 节点定义读取 | 桌面端读取节点定义、配置 schema、节点显示名和节点目录信息。 | 中 | 可保留；但 UI 不应写死具体节点，后续依赖节点定义动态生成展示和配置入口。 |
| UI 示例/模板工作流 | 当前模板里仍会出现具体内置节点类型。 | 中 | 后续移动到模板注册表或示例包，避免 `MainWindowViewModel` 长期硬编码节点。 |
| 权限/审计 | 权限句柄、AuditEvent、审计 API、审计页面已移除。 | 已解除 | 不再作为默认主程序依赖点；如后续需要，作为独立能力重新讨论。 |
| 外部程序/外部资源 | 当前不交给主程序统一管理，节点自己负责。 | 低 | 后续保持节点自治；异常兜底放在 WorkflowRunProcess 级别，而不是 EngineHost。 |
| 测试用内置节点 | 集成测试仍依赖内置节点跑正式链路。 | 中 | 测试可保留；后续可用测试节点包或 fixture registry 降低主程序感知。 |

## 00 级别执行方案

### 第一优先级：保留通用节点契约

主程序应继续保留 `NodeDefinition`、`NodeTask`、`NodeRun`、`RuntimeEvent`、`TableRef`、`SharedPublication` 等通用契约。

这些是工作流运行和数据中转的公共语言，不等同于主程序依赖具体节点。

### 第二优先级：统一节点执行入口

主程序不应持续按具体节点类型分发执行器。

目标是让主程序只知道“我要执行一个节点任务”，至于这个任务由哪个节点实现处理，由节点注册体系或执行器工厂解决。

### 第三优先级：节点显示能力元数据化

节点是否显示、是否测试节点、是否推荐给普通用户，应该写在节点定义元数据里。

主程序和 UI 不再通过具体节点名称判断节点展示方式。

### 第四优先级：UI 默认走通用节点定义

UI 添加节点、展示节点、编辑配置时，优先依赖后端节点定义和配置 Schema。

专用编辑器可以保留，但它应该是可选增强。没有专用编辑器的节点，也应该能通过通用配置表单使用。

### 第五优先级：模板和示例从主 UI 中拆出

示例工作流可以继续存在，但不应该长期写死在主窗口 ViewModel 里。

后续应改成模板注册表、示例包或节点包自带模板，让 UI 只负责展示和选择模板。

### 第六优先级：拆分开发测试节点和正式节点

延迟测试、故障测试、Fake 执行器等开发辅助能力应和正式节点路径区分开。

目标是让正式运行链路不依赖测试节点特殊判断，测试能力只在测试或开发模式下启用。

### 第七优先级：数据接口预留非表格类型

当前继续以表格数据为主，但不要把表格当成永久唯一数据形态。

后续节点如果产生文件、文本、图片、外部程序输出或其他结构化结果，应通过通用数据引用表达，主程序只负责保存引用、类型元数据和预览入口。

## 开发阶段暂定规则

1. 新增节点时，尽量不要修改 EngineHost、WorkflowRunProcess、MainWindowViewModel 这类主程序核心文件。
2. 如果必须修改主程序，需要先判断是不是缺少通用接口，而不是直接为某个节点加特殊判断。
3. 节点可以依赖主程序提供的通用上下文，例如 `NodeTask`、`NodeExecutionContext`、数据引用、RuntimeEvent/NodeRun 和节点结果上报接口。
4. 节点不能直接依赖主程序内部对象，例如 EngineHost、桌面 UI、MainWindowViewModel、RuntimeStore 内部实现或主程序生命周期。
5. 外部程序生命周期不交给 EngineHost。正常情况下由节点自己管理，异常情况下由 WorkflowRunProcess 级 Windows Job Object 兜底清理。
6. 默认主程序不再内置权限/审计。节点涉及外部资源、外部程序或副作用时，先写入节点方案，等节点体系大部分讨论完成后再决定是否需要统一能力。

## 当前阶段结论

当前权限/审计这条强耦合已经解除。剩余主要耦合集中在内置节点注册、执行器选择、测试节点与正式路径混用、UI 示例模板写死节点，以及表格优先的数据预览上。

这些问题在开发阶段可以接受，但需要作为后续节点体系整理的第一批减法目标。核心方向是：主程序提供通用能力，节点通过统一接口声明和运行，避免主程序随着节点数量增加而不断膨胀。
