# FlowWeaver 节点方案撰写规划

更新时间：2026-07-05

## 文档定位

本文件用于指导后续逐个撰写节点方案。

当前阶段只做节点规划，不要求代码已经实现，也不要求主程序马上补齐全部字段。

后续每个节点方案应先成为清晰的设计事实源，再决定是否进入实现。

## 当前前提

1. 主程序正在移除默认权限审计模块。
2. 后续节点规划暂时不围绕权限句柄、权限授权、权限撤销、AuditEvent 展开。
3. 主程序只保留工作流生命周期、节点调度、NodeRun 状态、RuntimeEvent、数据引用和基础数据中转。
4. 节点方案需要先遵守低耦合方向，避免把具体节点能力写进 EngineHost、WorkflowRunProcess、桌面 UI 或全局 Store。
5. 节点是否访问文件、数据库、网络、外部程序或其他资源，由节点方案自身说明，不由主程序统一授权。

## 总体原则

后续节点方案的核心问题不是“主程序给不给权限”，而是：

```text
这个节点解决什么问题
它需要什么输入
它产出什么输出
它需要哪些配置
它是否会影响外部世界
它如何失败得可理解
它怎样不让主程序认识自己的业务细节
```

每个节点方案都应尽量只描述节点自身契约，不写全局调度策略，不写 UI 页面布局，不写具体代码实现。

## 字段说明

### 内部类型名（暂定）

用于提前约定未来可能的 `node_type`，例如：

```text
FilterRowsNode
SelectColumnsNode
PublishSharedTablesNode
StartWorkflowNode
```

当前只是规划名，不代表代码已经实现。

### 副作用与外部资源说明

用于替代旧模板中的“需要的权限”。

建议使用：

```text
无外部副作用：只读取输入引用，只写本次运行输出。
只读外部资源：读取文件、数据库、网络接口等。
写入运行数据：只写 FlowWeaver 本次运行产生的数据。
写入外部资源：写文件、写数据库、调用外部系统。
启动外部程序：节点会启动或控制外部进程。
```

这一栏只描述事实和风险，不要求主程序统一授权。

### 是否需要用户确认

用于处理高风险节点。

建议使用：

```text
不需要：普通数据处理节点。
建议确认：会读取外部资源或消耗较高。
必须确认：会写外部资源、启动其他工作流、启动外部程序。
后置讨论：需要结合后台运行策略再决定。
```

第一版不做复杂授权体系，但可以在方案中明确哪些节点不适合后台无感运行。

### 主程序交互边界

这一栏用于防止耦合。

应写清楚节点只通过哪些标准能力与平台交互，例如：

```text
读取 NodeTask.config。
读取 input_refs。
发布 output_refs。
通过运行上下文检查取消。
通过运行上下文上报粗粒度进度。
返回 NodeTaskResult。
必要状态进入 RuntimeEvent 或 NodeRun.error。
```

同时应明确禁止事项，例如：

```text
不直接调用 EngineHost。
不直接操作 MainWindowViewModel。
不直接持有 RuntimeStore 实例。
不要求 WorkflowRunProcess 为本节点增加专用 if 分支。
不要求 UI 为本节点提供专用编辑器作为可用前提。
```

### 运行记录

用于替代旧的审计记录。

建议只写：

```text
NodeRun.status
NodeRun.progress
NodeRun.current_stage
NodeRun.error
RuntimeEvent 中的必要状态变化
节点结果中的 output_refs / error 摘要
```

不要写 AuditEvent、权限审计详情或权限授权记录。

### 实现前置依赖

用于明确当前代码还没具备的框架条件。

常见写法：

```text
无：当前表格节点主链路已经能承接。
需要通用 NodeRunner 接口。
需要薄版 NodeRuntimeContext。
需要节点定义元数据 category / description / tags。
需要 DataRef 或非表格数据预览。
需要 WorkflowRun 运行来源和父子关系。
需要用户确认机制。
```

这样可以先写规划，不把未实现能力误认为已经完成。

## 防耦合写作红线

每个节点方案都应避免以下表述：

```text
让 EngineHost 针对该节点做特殊处理。
让 WorkflowRunProcess 判断该 node_type 后走专用逻辑。
让 MainWindowViewModel 直接识别该节点。
让主程序为该节点维护内部资源生命周期。
让节点直接读写 RuntimeStore 内部结构。
让节点直接控制桌面 UI。
让节点依赖权限句柄或 AuditEvent。
```

如果某个节点确实需要主程序能力，应写成通用前置能力，而不是节点专用能力。

例如：

```text
不推荐：
WorkflowRunProcess 如果发现 StartWorkflowNode，就直接启动子工作流。

推荐：
补通用 NodeRunner / NodeRuntimeContext 能力，StartWorkflowNode 通过标准上下文请求创建子 WorkflowRun。
```

## 当前主程序能力假设

后续节点方案可以假设当前框架已经或正在具备：

```text
节点定义注册
节点目录只读 API
节点 config_schema
工作流节点实例 config
DAG 调度
NodeTask
NodeRun 状态
RuntimeEvent
TableRef 输入输出
共享表发布和读取基础能力
Avalonia 通用节点目录和基础配置表单
```

后续节点方案不应假设当前已经具备：

```text
通用 NodeRunner 接口
完整 NodeRuntimeContext
节点分类元数据
统一副作用等级字段
统一性能等级字段
DataRef 完整落地
后台极简运行策略
工作流启动来源和父子 run 关系
用户确认机制
主程序权限授权和 AuditEvent
```

## 推荐撰写顺序

### 第一批：现有核心节点补方案

先为已有节点补齐方案，形成样板。

```text
GenerateTestTableNode
FilterRowsNode
PublishSharedTablesNode
ReadSharedTablesNode
```

目的：

```text
验证模板是否好用。
把当前已存在能力讲清楚。
形成后续新节点的写作参考。
```

### 第二批：P0 / P1 表格处理节点

优先规划最容易复用当前 TableRef 主链路的节点。

建议候选：

```text
选择字段节点
重命名字段节点
筛选行节点补强
排序节点
限制行数 / 抽样节点
去重节点
字段计算节点
聚合节点
表连接节点
```

这些节点多数只读输入 TableRef、写本次运行输出，外部副作用低，适合作为第一批新增节点规划对象。

### 第三批：数据输入输出节点

这些节点更容易涉及外部资源，应在方案中重点写清副作用和用户确认。

建议候选：

```text
CSV 读取节点
CSV 写出节点
Excel 读取节点
Excel 写出节点
数据库读取节点
数据库写入节点
HTTP 读取节点
```

当前阶段不需要主程序统一权限，只需要节点方案说清楚：

```text
访问什么资源。
是否只读。
是否写外部状态。
是否需要用户确认。
失败时怎样提示。
后台运行是否适合。
```

### 第四批：运行控制和工作流组合节点

这类节点需要更多主程序通用能力，不建议第一批实现，但可以提前规划。

建议候选：

```text
启动工作流节点
等待工作流节点
人工确认节点
条件分支节点
停止 / 跳过后续节点
```

其中“启动工作流节点”应继续沿用已有草案方向：

```text
第一版只启动一个目标工作流。
必须用户确认。
记录 run_origin = WORKFLOW_TRIGGER。
记录 parent_workflow_run_id / parent_node_run_id。
第一版不等待子工作流完成。
第一版不自动读取子工作流输出。
```

### 第五批：外部程序和高风险节点

这些节点容易牵涉进程生命周期、取消、日志、退出码和后台风险，建议后置。

建议候选：

```text
运行外部程序节点
调用脚本节点
打开本地应用节点
自动点击 / 桌面自动化节点
```

规划时应强调：

```text
EngineHost 不直接管理外部程序。
节点负责正常生命周期。
WorkflowRunProcess 级别只做异常兜底清理。
后续需要 context.launch_process 之类的通用入口。
```

## 节点方案完成标准

一份节点方案算完成，至少应回答：

```text
用户为什么需要这个节点。
节点第一版做什么，不做什么。
输入和输出是否清楚。
配置项是否清楚。
数据契约是否能落到 TableRef / DataRef / 结果摘要。
是否有外部副作用。
是否需要用户确认。
失败提示是否能让用户知道怎么处理。
主程序是否需要新增节点专用判断。
当前实现还缺什么通用前置能力。
验收方式是否能转成测试或人工检查清单。
```

## 当前结论

可以开始提前撰写节点方案。

但节点方案应以“低耦合、无默认主程序权限审计”为前提：

```text
主程序不再作为权限管家。
节点不直接依赖主程序内部对象。
主程序不为每个节点增加专用分支。
节点通过标准输入、配置、输出、运行上下文和结果协议表达能力。
副作用、外部资源和用户确认先写入节点方案，不进入主程序权限审计链路。
```

建议下一步先按 V2 模板补齐现有四个核心节点方案，再进入新增 P0 / P1 表格处理节点规划。
