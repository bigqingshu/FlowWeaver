# FlowWeaver UI-COMPACT-0 工作流页面精简与通知日志浮层边界清单

## 1. 阶段定位

本阶段只做语义、界面边界和后续实施顺序确认，不改 Avalonia 代码，不改变后端 API，不调整工作流运行语义。

目标是减少工作流页面里的说明文字和调试信息占比，让主界面优先显示：

- 工作流列表
- 当前工作流定义摘要
- 新增节点入口
- 选中节点配置
- 工作流节点顺序
- 数据预览表格
- 必要的短状态和错误入口

详细说明、长日志、历史事件、低频高级信息应收进按钮、浮层、折叠面板或完整日志页。

## 2. 当前页面现状

当前 `WorkflowPage` 的主布局为：

- 左侧：`WorkflowListView`
- 中间：`WorkflowSummaryView`
- 右侧：`WorkflowNodeListView`
- 底部：`WorkflowDataPreviewView`

当前已存在的可复用基础：

- 连接区已在 `WorkflowSummaryView` 内默认折叠。
- 完整日志页已存在 `LogsAuditPage`，包含 RuntimeEvent 和 AuditEvent。
- 工作流节点列表已开始向紧凑卡片收口。
- 操作禁用原因已通过 `ToolTip.Tip` 部分承载。

当前占用主界面空间较多的信息：

- 节点移动语义说明。
- 线性链路状态说明。
- 工作流定义加载、校验、保存等长状态消息。
- 数据预览的说明、空状态和错误消息。
- 连接列表的详情。
- 日志和审计事件只能进入完整日志页查看，工作流页缺少最近事件入口。

## 3. 精简原则

主界面只保留直接影响当前操作的短文本。

说明文字优先移动到：

- 信息按钮。
- Tooltip。
- Flyout。
- 高级折叠区。
- 完整日志页。

状态信息分三层：

- 主界面短状态：例如“草稿未保存”“校验失败”“连接 1 条”。
- 通知浮层：例如“节点已删除”“预览完成”“保存失败”。
- 完整日志：RuntimeEvent、AuditEvent、WebSocket 流和历史事件。

错误展示分级：

- 阻断当前操作的错误保留在相关区域内。
- 非阻断成功、提示、普通进度变化进入通知浮层。
- 可追溯事件进入最近日志面板和完整日志页。

## 4. 主界面信息收纳清单

| 当前信息 | 建议处理 | 第一版位置 |
| --- | --- | --- |
| 节点移动语义说明 | 从常驻文本改为信息按钮 | 节点操作区标题右侧 |
| 线性链路状态 | 保留短状态，详细解释进入信息按钮 | 节点操作区 |
| 批量选择数量 | 保留短文本 | 节点操作区 |
| 工作流定义加载成功消息 | 非错误时进入通知浮层或短状态 | 通知浮层 |
| 工作流定义错误消息 | 保留区域内错误，另进入通知浮层 | 工作流定义摘要 |
| 校验成功消息 | 进入通知浮层，主界面只显示校验状态 | 通知浮层 |
| 校验失败消息 | 保留短错误，详情可展开 | 节点操作区或草稿高级区 |
| 连接列表 | 默认折叠 | 连接区 |
| 连接数量 | 保留 | 连接区标题 |
| 数据预览空状态说明 | 改成短状态 | 数据预览标题或表格占位 |
| 数据预览错误 | 保留短错误，详情进入通知或日志 | 数据预览区 |
| 运行/预览完成消息 | 进入通知浮层和最近日志 | 通知浮层、最近日志 |
| RuntimeEvent/AuditEvent | 不常驻主界面完整列表 | 最近日志、完整日志页 |

## 5. 通知浮层边界

建议新增统一通知浮层，视觉上接近桌面版 Snackbar/Toast，不使用系统级新窗口。

第一版建议放在 Shell 或 WorkflowPage 内的 Overlay 层，位置优先为右下角；如果和数据预览冲突，则改为右上角。

通知模型建议包含：

- `NotificationKey`
- `Kind`: `Info` / `Success` / `Warning` / `Error`
- `Title`
- `Message`
- `UpdatedAt`
- `IsOpen`
- `IsSticky`
- `AutoDismissAfter`

更新规则：

- 相同 `NotificationKey` 的消息更新时，只替换浮层内部文字，不重新播放弹入动画。
- 新 `NotificationKey` 第一次出现时播放一次轻量动画。
- 成功消息自动消失。
- 普通提示可自动消失。
- 错误消息默认常驻，用户点击关闭后消失。
- 高频运行事件不逐条弹动画，只更新“最近运行事件”通知内容。

第一版接入消息范围：

- 连接成功或失败。
- 工作流详情加载失败。
- 校验成功或失败。
- 保存成功或冲突失败。
- 新增、复制、删除、移动节点成功或失败。
- 数据预览完成或失败。
- 运行启动、完成、失败。

暂不接入：

- 每一条 RuntimeEvent 明细弹出通知。
- 每一条 AuditEvent 明细弹出通知。
- 长堆栈、完整 JSON、完整审计上下文。

## 6. 最近日志面板边界

建议新增“最近事件”小面板，放在 `WorkflowSummaryView` 的连接区下方。

用途：

- 让用户在工作流页面看到最近一条或少量事件。
- 减少进入完整日志页的频率。
- 作为运行、预览、保存、连接状态变化的轻量反馈区。

第一版形态：

- 标题：`最近事件`
- 右侧按钮：`查看全部`
- 常驻显示最近 1 条事件。
- 可展开显示最近 5 条事件。
- 点击“查看全部”跳转或提示进入完整日志页。

事件来源建议：

- RuntimeEvent 流中的最新事件。
- UI 操作结果消息。
- 数据预览结果消息。
- 工作流运行结果消息。

第一版不做：

- 复杂过滤。
- 审计详情展开。
- 大量日志虚拟列表。
- 日志搜索。

## 7. 连接区处理边界

连接区继续默认折叠。

建议保留：

- 标题：`连接`
- 数量：例如 `1 条连接`
- 展开/收起按钮
- 展开后显示连接 ID 和源目标端口

建议不放在主视图常驻：

- 手动连接编辑表单。
- 删除连接按钮。
- 连接校验的长说明。

删除节点、移动节点产生的连接变化，优先通过通知浮层和最近事件提示：

- `已同步移除 2 条连接`
- `已重排线性连接`

## 8. 信息按钮边界

信息按钮只用于解释低频规则，不作为主要操作入口。

第一批建议收纳：

- 上移/下移对连接的影响。
- 批量删除会同步移除相关连接。
- 运行到选中节点的预览语义。
- 草稿 JSON 是高级入口。
- 连接区默认折叠的原因。

信息按钮展示方式：

- 短说明使用 Tooltip。
- 多段说明使用 Flyout。
- 不新开系统窗口。

## 9. 后续实施顺序建议

### UI-COMPACT-1 节点操作区说明收口

只改 View 层和结构测试。

- 移除常驻的节点移动语义长文本。
- 保留短状态。
- 增加信息按钮承载详细说明。

### UI-COMPACT-2 通知浮层模型与 Host 骨架

只做最小 ViewModel、View 和测试。

- 新增通知状态。
- 新增显示、更新、关闭方法。
- 不接入所有业务消息。

### UI-COMPACT-3 工作流编辑消息接入通知浮层

接入新增、复制、删除、移动、校验、保存相关消息。

- 成功消息自动消失。
- 失败消息常驻。
- 相同消息键只更新文本，不重复动画。

### UI-COMPACT-4 最近事件面板

在连接区下方增加最近事件小面板。

- 默认显示最近一条。
- 可展开最近五条。
- 完整日志页保留。

### UI-COMPACT-5 数据预览与运行消息瘦身

压缩数据预览区说明文字。

- 表格常驻。
- 空状态短文本。
- 运行和预览结果进入通知浮层和最近事件。

### UI-COMPACT-6 总体验收复核

复核：

- 主界面说明文字是否明显减少。
- 关键错误是否仍然可见。
- 成功消息是否不会占据主界面。
- 通知浮层是否不会重复动画。
- 最近事件是否能辅助定位运行问题。

## 10. 验收基线

UI-COMPACT 阶段完成后，应满足：

- 工作流页面主要空间用于列表、节点配置和数据表。
- 普通说明文字不再常驻占用主界面。
- 用户执行操作后能看到即时反馈。
- 高频消息不会重复弹出动画。
- 错误不会被通知自动消失误吞。
- 完整日志能力仍通过日志页保留。
- 连接区仍是高级辅助信息，不重新变成主要编辑区。

## 11. 明确不做

本阶段和后续 UI-COMPACT 第一轮不做：

- 更换 UI 框架。
- 改后端事件协议。
- 改工作流运行语义。
- 实现自由画布。
- 实现拖拽连线。
- 实现复杂日志搜索。
- 实现系统级通知。
- 实现多窗口日志管理器。
