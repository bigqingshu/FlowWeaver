# FlowWeaver 表数据流与节点资源解耦边界清单

## 目标

明确后续节点扩展时的主程序边界：主程序只管理工作流中的表数据流，不直接管理文件位置、数据库连接、接口地址、SQL 语句或其他外部资源细节。

## 核心结论

- 主程序统一以 `TableRef` 作为节点之间的数据传递对象。
- 连接只表达节点之间的表依赖关系，不负责复制表数据。
- 文件、SQL、接口、内存表等资源来源由节点内部处理。
- UI 默认面向用户展示节点顺序、节点配置和数据预览，不把连接作为主要操作入口。
- 后续新增节点应优先通过后端节点注册和通用配置 Schema 接入，避免为每个节点单独改主程序。

## 主程序负责

- 保存工作流定义、节点列表和节点配置。
- 保存节点之间的连接关系，用于表达表数据依赖。
- 调度节点执行，并把上游节点的 `output_refs` 传给下游节点的 `input_refs`。
- 记录 `WorkflowRun`、`NodeRun`、`NodeTaskResult` 和 `TableRef`。
- 提供统一的数据预览入口，根据 `TableRef` 查询表结构、行数和样例数据。

## 节点负责

- 解释自身配置。
- 读取文件、访问 SQL、调用接口或生成内存表。
- 将外部资源或计算结果转换为表数据。
- 输出一个或多个 `TableRef`。
- 对输入表执行筛选、转换、合并、写回、发布等节点自身逻辑。

## 不建议主程序负责

- 不直接保存或解释文件路径的业务含义。
- 不直接保存或解释数据库连接串、SQL 语句和表映射规则。
- 不直接管理接口地址、鉴权参数和请求体结构。
- 不直接理解每类节点的业务配置语义。
- 不把多端口作为第一阶段 UI 主概念暴露给用户。

## 当前主程序观察

当前主程序已经基本满足“只编排表引用”的方向：

- `WorkflowDefinition` 保存节点、连接和节点 `config`，不解释具体业务配置。
- 连接只校验节点和端口是否存在、必填输入是否已连接、DAG 是否有环。
- 调度 READY 节点时，主程序读取上游节点成功结果里的 `output_refs`，再传给下游节点的 `input_refs`。
- `NodeTaskResult` 记录的是 `output_refs`，不是表数据本体。
- `TableRef` 已包含 `provider_id`、`storage_kind`、`logical_table_id`、`opaque_handle`、schema 和 capabilities，具备承载不同表来源的基础。

当前缺口也应保持在 provider/节点边界解决：

- 现有数据预览 API 仍只支持 `SQLiteRuntimeTableProvider` 和 `RUNTIME_SQL`。
- 还没有 SQL 映射节点，也没有外部 SQL provider 路由。
- 后续若支持外部 SQL 表预览，应补 provider 注册/路由能力，而不是让主程序直接理解 SQL 节点。

## 性能与复制边界

连接和 `TableRef` 本身不应触发表数据复制：

```text
连接：表达依赖关系
TableRef：表达表引用和读取能力
真实数据：由 provider 或节点管理
```

可能发生复制的位置应集中在明确边界：

- 节点处理需要产出新结果表时。
- 跨 provider 且无法下推计算时。
- 保存、导出、发布或 checkpoint 时。
- 外部资源需要转成工作流内部可读表时。

应尽量避免复制的位置：

- 节点之间通过连接传递数据时。
- SQL 节点只是映射外部表或查询结果时。
- 数据预览只读取分页、样例或 schema 时。
- 中间结果仍处于临时计算链路、尚未要求保存时。

## SQL 映射方向

SQL 相关能力应做成节点：

1. SQL 节点在自身配置中维护本地数据库配置、SQL 查询或表映射规则。
2. SQL 节点执行后输出 `TableRef`。
3. 后续节点只看到输入是一张表，不需要知道它来自 SQLite、外部数据库、视图或查询结果。
4. 主程序只保存节点配置和 `TableRef`，不解释 SQL 资源细节。

目前尚未实现 SQL 节点。本阶段只固化边界，不提前改主程序或引入 SQL provider。

SQL 节点的目标是零复制映射：

```text
SQL 节点配置物理地址 / SQL / 表映射
-> 输出 ExternalSqlTableRef
-> 后续节点通过 provider 读取或下推计算
```

后续是否复制由下游能力决定：

- SQL 表接 SQL 可下推节点时，可以继续生成查询引用或视图引用。
- SQL 表接通用 Python 行处理节点时，可能需要读取数据并产出新表。
- SQL 表接保存节点时，保存动作本身会写出一份结果。

主程序不应因为 SQL 节点存在而理解数据库连接串、SQL 文本或物理表地址。

## SQL 节点映射处理原则

“节点映射”应理解为 SQL 节点内部把物理资源映射成工作流表结果：

```text
物理数据库 / SQL / 物理表
-> SQL 节点内部映射
-> TableRef
-> 工作流连接传递 TableRef
```

主程序只看到：

- 节点实例：`node_instance_id`、`node_type`、`node_version`、`config`。
- 节点结果：`output_refs`。
- 表引用：`TableRef`。

主程序不应看到或解析：

- 数据库连接串。
- SQL 文本的业务含义。
- 物理库名、物理表名和字段映射规则。
- 认证信息、文件路径或接口地址。

SQL 节点映射信息的推荐放置位置：

- 普通映射参数放在 SQL 节点自己的 `config` 中。
- 敏感连接信息优先放在节点/provider 自己管理的 profile 中，工作流里只保存引用 ID。
- `TableRef.logical_table_id` 表示工作流内部可读的逻辑表名。
- `TableRef.provider_id` 指向负责读取它的 provider。
- `TableRef.opaque_handle` 只作为 provider 私有句柄，主程序保存但不解释。
- `TableRef.schema` 和 `schema_fingerprint` 用于预览、校验和后续节点理解字段结构。

后续最小落地顺序应是：

1. 新增 SQL 节点定义和配置 Schema。
2. SQL 节点执行后产出只读 `TableRef`，先不要求复制数据。
3. 增加外部 SQL provider 的读取能力。
4. 数据预览 API 从单一 SQLite provider 过渡到 provider 路由。
5. 再考虑筛选、排序、选择列等下推能力。

暂不建议：

- 不在工作流顶层新增 SQL 资源管理模型。
- 不让连接保存 SQL 映射规则。
- 不为 SQL 节点单独改 UI 主结构。
- 不为了 SQL 节点提前引入多端口 UI。
- 不在未实现 provider 前承诺外部 SQL 表一定可预览。

## 内存表方向

内存表也应通过 `TableRef` 接入：

1. 内存表节点或 provider 负责维护真实数据位置。
2. `TableRef.storage_kind` 或 `provider_id` 标识它由哪个 provider 读取。
3. 数据预览和后续节点通过统一接口读取。
4. 是否落盘、是否溢写、是否转换为运行期 SQL 表，由节点或 provider 决定。

`TableRef` 本身不是数据库，也不是缓存系统。查询速度取决于背后的 provider。

内存表 provider 可以用于：

- 小表和临时中间表。
- 高频预览样例。
- 不需要立即持久化的节点中间结果。
- 后续可重跑、可丢弃的短生命周期数据。

跨进程执行时，普通进程内存不能天然共享。若要减少跨进程复制，应由内存表 provider 选择共享内存、内存映射、Arrow IPC、临时 SQLite/DuckDB 或其他机制，主程序仍只持有 `TableRef`。

## 多表方向

多表节点可以先不引入多端口 UI：

- 主程序仍传递 `input_refs` 列表。
- 节点配置负责说明这些输入表的含义。
- 输入顺序必须稳定，避免节点收到随机顺序的表。
- 合并、对比、映射、写回等复杂语义由节点自身配置解释。

示例：

```text
输入表 1：主表
输入表 2：参照表
节点配置：匹配字段、输出字段、冲突策略
节点输出：结果表 TableRef
```

## 连接语义

连接保留在工作流定义中，但它是内部结构语义：

```text
上游节点 output_refs -> 下游节点 input_refs
```

连接本身不代表复制表，也不代表共享发布。它只说明下游节点依赖哪些上游节点的表结果。

## 保存与物化策略

保存、导出和发布应继续通过节点完成，而不是由主程序在每一步自动落盘。

推荐策略：

```text
中间过程：TableRef / MemoryTableRef / RuntimeTableRef / LazyTableRef
明确保存：SaveToFileNode / SaveToSqlNode / PublishSharedTableNode
```

原因：

- 工作流内部可能多次修改和预览，不应每一步都写入本地 SQL 或文件。
- 中间结果频繁落盘会增加 IO、降低性能，并增加清理成本。
- 保存动作通常具有业务含义，应由用户明确放置保存节点。
- 需要崩溃恢复或断点恢复时，可以单独引入 checkpoint 策略，而不是默认全部保存。

## 常见复制判断

通常可以避免复制：

- 连接传递上游 `output_refs`。
- SQL 源表只映射为外部 `TableRef`。
- provider 支持的筛选、排序、选择列、分页预览。
- 可表达为视图、查询计划或懒执行链路的中间结果。

通常可能发生复制：

- 保存到 Excel、CSV、本地 SQL 或外部数据库。
- 从外部 SQL 表进入无法下推的通用处理节点。
- 多表合并且来源 provider 不同。
- 发布不可变共享表并需要固定快照。
- 为崩溃恢复主动创建 checkpoint。

## UI 方向

- 默认隐藏连接区域，只作为高级入口。
- 用户主要操作节点：新增、删除、复制、上移、下移、预览到选中节点、运行。
- 新增节点时，UI 尽量自动维护线性连接。
- 自动连接失败时，提示用户缺少上游输入，而不是要求用户直接编辑连接 JSON。
- 数据预览只面向表结果，不暴露底层资源来源。

## 后续扩展原则

- 新资源类型优先新增节点或 provider，不扩主程序资源模型。
- 新业务配置优先进入节点 `config_schema`，不写死到 UI。
- 新表存储形态优先扩展 `TableRef`/provider 读取能力。
- 只有当通用配置控件不够表达时，才增强 UI 的通用配置渲染能力。
- 主程序始终保持“编排表数据流”的角色，不升级为文件管理器、数据库管理器或接口管理器。
