# FlowWeaver 后续节点大方向模板

> 文档用途：用于后续讨论、汇总和拆分新增节点方向时套用。本文只写节点建设的大方向和优先级，不展开具体节点功能清单。
> 对照来源：已参考 `00_第一阶段技术接口与验收规范.md`、`01_第一阶段执行方案.md`、`02_数据库共享表与节点故障隔离_讨论方案.md`、`03_架构讨论阶段总结.md` 的已实现、未实现和后续讨论项。
> 适用范围：后续新增业务节点、节点配置体验、节点目录、节点调试和节点生态规划。
> 暂不进入：具体节点实现、具体数据格式、具体业务场景、代码改动。

## 一、优先级总览

| 优先级 | 方向 | 目标 |
| --- | --- | --- |
| P0 | 节点基础设施 | 先让新增节点有统一模板、统一注册、统一数据契约、统一资源配置映射和统一验收方式 |
| P1 | 常用业务节点主线 | 再补真正能组成业务工作流的输入、处理、输出、共享和版本读取能力 |
| P2 | 节点使用体验 | 让节点多起来以后仍然好找、好配、好预览、好调试和好定位问题 |
| P3 | 节点安全与稳定性 | 降低误操作、卡顿、错误难理解、副作用、取消传播和资源泄漏风险 |
| P4 | 高级节点生态 | 等基础稳定后，再考虑血缘、变更摘要、组节点、插件生命周期和分发能力 |

## 二、对照 00 到 03 的补充判断

### 已覆盖

- 已覆盖节点模板、节点配置、运行辅助、验收方式、常用业务节点、节点目录、预览、安全保护、血缘、组节点和插件化这些大方向。
- 已覆盖“先基础、再业务、再体验、再安全、最后生态”的总体顺序。

### 需要补充

| 补充方向 | 应放优先级 | 补充原因 |
| --- | --- | --- |
| 资源配置映射 | P0 | `00` 到 `03` 多次提到资源配置、真实地址映射和权限边界；后续节点变多后，不能每个节点重复填写底层资源信息 |
| 节点数据契约 | P0 | 已实现主线依赖 TableRef、DataRef、NodeResult、InputSnapshot、ReadLease；后续节点必须先按统一输入输出契约接入 |
| 共享数据版本策略 | P1 | 当前只收口了明确版本读取主线，后续要扩展更多读取策略时，需要先作为方向保留 |
| 数据调试与质量检查 | P2 | `03` 提到字段监视器、数据断言、条件断点和二分定位，这些更像调试能力，不宜混在具体业务节点里 |
| 失败策略与重试语义 | P3 | `01`、`02` 中 RETRY、SKIP_DEPENDENTS 等策略后置，说明需要单独作为稳定性方向拆出来 |
| 取消传播与资源回收 | P3 | `03` 明确把取消传播和资源回收列为后续空白，尤其影响组节点、插件节点和长时间运行节点 |
| 组节点运行模型 | P4 | 组节点不能只当 UI 折叠，需要后续决定是否拥有独立输入输出、权限、状态和生命周期 |
| 插件生命周期 | P4 | 插件节点需要明确进程创建、复用、取消、销毁和资源归属，适合在基础节点稳定后推进 |
| 变更摘要与血缘追踪 | P4 | ChangeSet、字段血缘和单元格追踪属于高级可观测能力，适合分层推进 |

## 三、P0：节点基础设施

### 方向说明

P0 的目标不是增加很多节点，而是先把“以后怎么加节点”这条路铺稳。

### 需要实现的能力

- 节点开发模板：新增节点时有固定结构、固定命名、固定注册入口和固定测试方式。
- 节点配置模板：节点参数能按统一格式描述，方便后续生成表单。
- 节点数据契约：节点统一说明自己读取什么、输出什么、输出是否可复用、失败时是否保留中间结果。
- 资源配置映射：外部资源先登记为逻辑配置，节点只引用逻辑名称，避免直接暴露真实路径、连接信息或敏感参数。
- 节点能力标记：节点声明执行方式、资源需求、是否可能长时间运行、是否有副作用、是否支持预览和是否支持取消。
- 节点运行辅助：节点能统一读取输入、写出输出、上报进度、响应取消和返回错误。
- 节点验收模板：每个节点都能按相同清单检查输入、输出、错误、取消和边界行为。

### 完成标准

- 新增一个节点不需要重新设计目录和注册方式。
- 节点配置能被 UI 识别并生成基础编辑入口。
- 节点输入输出能被运行框架、预览界面和共享能力识别。
- 资源配置不散落在每个节点内部，后续可统一授权、审计和迁移。
- 节点失败时能给出稳定、可解释的错误信息。

## 四、P1：常用业务节点主线

### 方向说明

P1 的目标是让工作流从“平台能跑”变成“能处理真实业务数据”，并能稳定复用上游或其他流程产生的数据。

### 需要实现的能力

- 数据输入方向：负责把外部数据转换为 FlowWeaver 可运行的数据输入。
- 数据处理方向：负责清洗、转换、筛选、组合和调整数据。
- 数据输出方向：负责把运行结果交付给用户或后续流程。
- 共享与复用方向：负责把可复用结果发布给其他工作流或后续节点。
- 版本读取方向：负责说明读取最新数据、固定版本数据或跟随上游数据时的规则。
- 数据交接方向：负责把节点之间的数据引用、版本、生命周期和释放方式讲清楚。

### 完成标准

- 一个基础业务流程可以从输入、处理到输出完整跑通。
- 节点之间的数据传递稳定，不需要用户手动处理底层引用。
- 同一份共享数据在一次运行中不会因为版本变化导致前后不一致。
- 常见失败能在节点层给出明确提示。

## 五、P2：节点使用体验

### 方向说明

P2 的目标是解决节点变多后的使用问题，让用户能快速找到、配置、预览、调试和定位数据问题。

### 需要实现的能力

- 节点目录分类：按用途组织节点，避免所有节点堆在一个列表里。
- 节点搜索与筛选：快速找到需要的节点。
- 节点配置表单：减少直接编辑 JSON 的场景。
- 高频节点专用编辑入口：对复杂配置提供更直观的编辑方式。
- 节点预览体验：选中节点后能更方便地查看输入、输出和运行结果。
- 数据调试入口：能围绕字段、关键值、运行结果和失败位置定位问题。
- 数据质量检查：能对关键节点或关键结果做基础检查，帮助用户判断数据是否异常。
- 问题定位辅助：长流程出错时，能缩小排查范围，而不是要求用户从头到尾逐个节点查看。

### 完成标准

- 用户不需要记住节点内部类型名也能找到节点。
- 常用节点可以通过表单完成配置。
- 调试单个节点时，不必每次运行完整工作流。
- 数据结果异常时，用户能更快知道问题大概从哪个节点或哪段流程开始出现。

## 六、P3：节点安全与稳定性

### 方向说明

P3 的目标是让节点在真实使用中更稳，尤其是避免误操作、难以理解的失败、失控重试、取消不完整和资源泄漏。

### 需要实现的能力

- 副作用节点保护：对可能修改外部状态的节点提供提示、限制或确认。
- 大数据预览保护：避免因为表太大、字段太多或刷新太频繁导致 UI 卡顿。
- 错误解释优化：把底层异常转成用户能理解的原因和处理建议。
- 节点运行保护：统一处理取消、超时、失败清理和资源释放。
- 节点兼容性检查：避免旧节点配置在新版本中静默出错。
- 失败策略治理：明确失败后是停止、跳过依赖、继续独立分支还是允许重试。
- 重试语义：只有可重复、无危险副作用或已明确确认的节点，才允许自动重试。
- 取消传播：明确取消单个节点、组节点、插件节点和整个工作流时的传播边界。
- 资源回收：节点成功、失败、取消、超时、崩溃和主程序异常退出后，都要有清理规则。

### 完成标准

- 高风险节点不会在用户无感知的情况下产生副作用。
- 大数据场景不会明显拖垮 UI。
- 节点失败后用户能知道大概哪里错、下一步该检查什么。
- 取消或失败不会留下不可解释的临时数据、锁、进程或运行状态。
- 重试和跳过行为有清楚规则，不会让结果看起来成功但实际不可信。

## 七、P4：高级节点生态

### 方向说明

P4 的目标是建立更强的长期扩展能力，但应在 P0 到 P3 稳定后再进入。

### 需要实现的能力

- 字段级追踪：查看字段经过哪些节点处理。
- 单元级追踪：在必要场景下追踪更细粒度的数据来源。
- 变更摘要：记录节点对数据结构、字段和值域造成了哪些重要变化。
- 组节点：把一组节点封装成可复用的流程单元，并决定它是否拥有独立输入输出、权限、状态和生命周期。
- 节点插件化：让节点可以独立安装、更新和管理，并明确插件进程的创建、复用、取消和销毁规则。
- 节点版本治理：支持节点能力演进、兼容提示和迁移策略。
- 高级审计分层：按需要记录从基础运行事件到字段级变化的不同审计深度。

### 完成标准

- 复杂流程可以被封装和复用。
- 节点生态可以独立扩展，而不必每次都改主程序。
- 用户能理解不同节点版本之间的兼容关系。
- 插件节点和组节点的运行、取消、失败、资源回收都有明确边界。
- 需要追踪问题时，可以从节点级逐步深入到字段级或更细粒度，而不是一开始就承担全部复杂度。

## 八、性能占用分析

### 总体判断

模板里的能力不是都会明显占用性能。真正容易占用性能的，通常有三类：

- 会反复读取或扫描大量数据的能力。
- 会记录大量中间状态、字段变化或单元级明细的能力。
- 会启动额外进程、重复执行节点或长期保持监听的能力。

### 性能影响分级

| 功能方向 | 性能影响 | 主要消耗 | 说明 |
| --- | --- | --- | --- |
| 节点开发模板 | 低 | 启动和加载元数据 | 主要是规范和结构，不直接处理大量数据 |
| 节点配置模板 | 低 | 配置解析 | 配置较多时会影响编辑体验，但通常不是运行瓶颈 |
| 节点数据契约 | 低 | 元数据校验 | 主要检查输入输出结构，成本可控 |
| 资源配置映射 | 低 | 配置读取和权限判断 | 只做逻辑名称到真实资源的映射，不直接处理数据 |
| 节点能力标记 | 低 | 元数据读取 | 用于调度和 UI 判断，性能占用很小 |
| 节点运行辅助 | 低到中 | 心跳、进度、错误上报 | 单个节点成本低；节点很多或上报太频繁时会增加事件压力 |
| 共享数据版本策略 | 中 | 版本查询、租约、快照固定 | 要维护数据版本一致性，会增加元数据读写和生命周期管理 |
| 数据交接方向 | 中 | 引用登记、生命周期维护 | 数据本体不一定复制，但引用、版本、释放状态需要维护 |
| 节点目录分类和搜索 | 低到中 | 索引、筛选、UI刷新 | 节点数量很大时需要缓存或索引 |
| 节点配置表单 | 低到中 | 表单生成、校验 | 高频编辑时会影响 UI 流畅度，但不影响后台运行主链路 |
| 节点预览体验 | 中到高 | 数据读取、抽样、渲染 | 如果直接预览大数据或频繁刷新，容易拖慢 UI |
| 数据调试入口 | 中到高 | 结果查询、字段查找、运行记录读取 | 调试越细，读取历史和中间结果越多 |
| 数据质量检查 | 中到高 | 数据扫描、规则计算 | 如果对完整数据做检查，可能接近一次额外节点运行 |
| 问题定位辅助 | 中到高 | 局部重跑、结果对比 | 长流程二分定位会减少人工排查，但可能增加后台执行次数 |
| 大数据预览保护 | 低到中 | 抽样、分页、限制刷新 | 本身是保护能力，会有少量开销，但能避免更大的卡顿 |
| 失败策略治理 | 中 | 状态判断、下游传播 | 策略本身不重，但复杂分支会增加调度判断 |
| 重试语义 | 高 | 节点重复执行 | 自动重试会直接增加运行时间和资源占用 |
| 取消传播 | 中 | 状态同步、进程通知 | 需要通知节点、组节点、插件节点和工作流状态 |
| 资源回收 | 中 | 临时数据、锁、进程清理 | 回收越彻底，退出阶段或异常恢复阶段越需要额外检查 |
| 字段级追踪 | 高 | 字段映射、历史记录、查询 | 每个节点都要记录字段变化时，存储和查询成本会上升 |
| 单元级追踪 | 高到极高 | 明细记录、索引、存储 | 如果默认全量开启，会快速放大数据量 |
| 变更摘要 | 中到高 | 差异计算、摘要存储 | 比完整血缘轻，但仍要计算节点前后变化 |
| 组节点 | 中到高 | 嵌套调度、状态聚合、资源边界 | 如果组节点拥有独立生命周期，调度和状态管理会更复杂 |
| 节点插件化 | 中到高 | 进程启动、IPC、内存占用 | 插件隔离更安全，但会带来进程和通信成本 |
| 节点版本治理 | 低到中 | 兼容检查、迁移提示 | 主要是元数据判断，迁移时成本会升高 |
| 高级审计分层 | 中到高 | 日志写入、批量落盘、查询 | BASIC 通常可控，FULL 或 DEBUG 会明显增加记录量 |

### 高性能风险项

后续最需要谨慎开启的是：

- 单元级追踪：最容易放大存储量，不适合默认全量开启。
- 字段级追踪：适合先做节点级和字段级摘要，再逐步细化。
- 数据质量检查：适合按关键节点、关键字段或抽样执行，不适合默认全流程全量检查。
- 节点预览体验：必须限制行数、字段数、刷新频率和加载方式。
- 问题定位辅助：局部重跑和二分定位要有运行次数限制。
- 重试语义：必须限制次数，并区分是否有副作用。
- 节点插件化：要控制插件进程数量、复用策略和空闲释放。
- 高级审计分层：默认不应开启最细粒度审计。

### 建议控制方式

- 默认轻量：默认只开基础进度、基础审计、节点级状态和必要错误信息。
- 按需开启：字段追踪、单元级追踪、深度调试、完整审计由用户或调试模式打开。
- 限制范围：预览、质量检查和追踪能力都要支持行数、字段数、节点范围和时间范围限制。
- 抽样优先：大数据场景先显示抽样结果，再按用户需要加载更多。
- 异步处理：审计、日志、调试索引和变更摘要尽量异步写入，避免阻塞节点主流程。
- 缓存复用：预览结果、字段索引、质量检查结果和局部运行结果尽量可复用。
- 分层追踪：先做工作流级和节点级，再做字段级，最后才考虑单元级。
- 明确预算：每次工作流运行可以设置最大预览量、最大重试次数、最大调试记录量和最大插件进程数。

## 九、后台极简运行模式

### 模式定位

后台极简运行模式用于无人值守、定时执行、批量执行或生产环境运行。

它的目标不是帮助用户调试，而是尽可能低成本地把工作流跑完，并在失败时留下足够定位问题的最小记录。

### 必须保留的节点功能

| 功能 | 后台极简状态 | 保留原因 |
| --- | --- | --- |
| 节点定义读取 | 保留 | 节点必须知道自己是谁、使用什么配置、依赖哪些输入 |
| 节点配置快照 | 保留 | 保证本次运行可复现，避免运行中配置变化影响结果 |
| 节点依赖判断 | 保留 | 判断上游是否完成，决定当前节点能不能运行 |
| 输入引用读取 | 保留 | 节点必须能读取上游交付的数据引用 |
| 输出引用发布 | 保留 | 节点必须能把结果交给下游或最终输出 |
| 基础状态流转 | 保留 | 至少记录等待、运行中、成功、失败、取消、超时 |
| 最小心跳 | 保留 | 用来判断节点是否还活着，避免卡死无法处理 |
| 粗粒度进度 | 可保留 | 只记录开始、运行中、完成或失败，不记录细分阶段 |
| 错误摘要 | 保留 | 失败时必须知道哪个节点失败、大概原因是什么 |
| 超时控制 | 保留 | 防止后台节点无限运行 |
| 取消响应 | 保留 | 用户或系统停止工作流时，节点能尽快退出 |
| 失败清理 | 保留 | 失败后清理临时输出、锁和未发布结果 |
| 基础资源回收 | 保留 | 释放临时文件、临时表、进程、连接和租约 |
| 最小权限检查 | 保留 | 节点访问资源前仍要确认是否允许 |
| 最小审计 | 保留 | 只记录运行开始、结束、失败、取消和关键资源访问 |
| 版本固定 | 保留 | 后台运行期间同一输入不能前后读取到不同版本 |
| 原子发布 | 保留 | 节点成功后一次性发布结果，失败时不发布半成品 |

### 默认关闭或降级的节点功能

| 功能 | 后台极简状态 | 处理方式 |
| --- | --- | --- |
| 节点预览 | 默认关闭 | 不主动读取样例数据，不刷新 UI 预览 |
| 字段监视器 | 默认关闭 | 不记录字段 watch 历史 |
| 条件断点 | 默认关闭 | 后台运行不暂停等待用户判断 |
| 数据断言 | 默认关闭 | 除非用户把它声明为正式校验规则 |
| 数据二分定位 | 默认关闭 | 失败后不自动反复重跑排查 |
| 字段级追踪 | 默认关闭 | 只保留节点级输入输出关系 |
| 单元级追踪 | 默认关闭 | 不记录单元格级来源和变化 |
| 完整 ChangeSet | 默认关闭 | 不计算完整差异，只可保留轻量摘要 |
| 完整调试日志 | 默认关闭 | 只保留错误摘要和必要上下文 |
| 详细进度 metrics | 默认关闭 | 不频繁上报行数、耗时、阶段明细等指标 |
| 高频事件推送 | 默认降级 | 合并事件，减少 WebSocket 或 IPC 压力 |
| 自动重试 | 默认关闭或限制一次 | 避免后台重复消耗资源或重复产生副作用 |
| FULL/DEBUG 审计 | 默认关闭 | 只启用 MINIMAL 级别审计 |
| 插件常驻 | 默认关闭 | 后台任务完成后尽快释放插件进程 |
| 大数据全量预览缓存 | 默认关闭 | 不为无人查看的结果生成预览缓存 |

### 后台模式建议档位

| 档位 | 用途 | 默认能力 |
| --- | --- | --- |
| BACKGROUND_MINIMAL | 最省性能后台运行 | 状态、心跳、错误摘要、原子发布、最小审计、资源回收 |
| BACKGROUND_NORMAL | 常规后台运行 | 在 MINIMAL 基础上增加粗粒度进度、轻量运行统计和失败上下文 |
| INTERACTIVE_DEBUG | 用户调试运行 | 开启预览、调试、质量检查、追踪和更详细事件 |

### 当前建议

后续如果要正式落地后台运行，可以先做 `BACKGROUND_MINIMAL`。

它不追求调试舒服，只保证：

- 跑得起来。
- 结果能交付。
- 失败能知道哪个节点出了问题。
- 卡死能被发现并终止。
- 临时资源能回收。
- 不产生明显的后台性能负担。

## 十、运行记录策略分层与界面放置

### 总体结论

这个功能不适合只做工作流级，也不适合只做节点级。推荐采用：

```text
全局默认策略
-> 工作流默认策略
-> 节点覆盖策略
-> 单次运行临时策略
```

其中最重要的是：

- 工作流级负责先把整体设置为低占用模式。
- 节点级负责对少数关键节点、慢节点或易失败节点开启更完整记录。
- 单次运行策略只影响本次运行，不直接改工作流定义。
- 运行历史负责给出建议，不应自动替用户打开高成本记录。

### 策略优先级

最终运行时建议按下面顺序合并：

```text
节点覆盖策略
> 单次运行临时策略
> 工作流默认策略
> 全局默认策略
> 系统内置兜底策略
```

如果某个节点没有单独设置，就继承工作流默认策略。

如果工作流也没有设置，就继承全局默认策略。

### 工作流级设置

工作流级适合设置“这条工作流默认怎么记录”。

建议字段：

```text
run_record_profile: BACKGROUND_MINIMAL / BACKGROUND_NORMAL / INTERACTIVE_DEBUG
audit_level: MINIMAL / BASIC / FULL / DEBUG
progress_mode: NONE / COARSE / DETAILED
event_mode: MINIMAL / NORMAL / VERBOSE
preview_mode: OFF / SAMPLE / FULL
trace_mode: OFF / NODE / FIELD / CELL
metrics_mode: OFF / BASIC / DETAILED
retention_policy: 临时保留 / 保留失败运行 / 全部保留
```

后台运行建议默认：

```text
run_record_profile = BACKGROUND_MINIMAL
audit_level = MINIMAL
progress_mode = COARSE
event_mode = MINIMAL
preview_mode = OFF
trace_mode = OFF
metrics_mode = BASIC
```

### 节点级覆盖设置

节点级适合设置“这个节点要不要比工作流默认记录得更详细”。

适合单独提高记录等级的节点：

- 运行时间最长的节点。
- 历史失败率高的节点。
- 结果影响很大的关键节点。
- 访问外部资源或有副作用的节点。
- 用户正在排查问题的节点。

建议字段：

```text
inherit_workflow_record_profile: true / false
node_record_profile: BACKGROUND_MINIMAL / BACKGROUND_NORMAL / INTERACTIVE_DEBUG
progress_mode_override
event_mode_override
trace_mode_override
metrics_mode_override
audit_level_override
preview_mode_override
retain_intermediate_outputs
```

节点级默认应该是继承工作流，不主动覆盖。

### 基于运行时间的建议

系统可以根据历史运行记录给建议，但不要直接自动开启高成本记录。

建议逻辑：

```text
如果节点平均耗时明显高于其他节点
或节点 P95 耗时超过阈值
或节点多次进入 LONG_RUNNING
或节点失败后排查成本高
则提示用户：建议为该节点开启 BACKGROUND_NORMAL 或更详细记录
```

提示方式应该是：

- 标记“建议增强记录”。
- 展示原因，例如“最近 5 次运行平均耗时最高”。
- 用户手动确认后，写入节点覆盖策略。

不要默认自动改成详细记录，因为详细记录本身也会增加后台占用。

### 单次运行临时设置

单次运行临时设置适合在用户点击运行前选择：

```text
本次按后台极简运行
本次按常规后台运行
本次按交互调试运行
本次只增强慢节点记录
本次只增强失败节点记录
```

这类设置只进入本次 `WorkflowRun` 的配置快照，不直接修改工作流定义。

### 适合放入的现有页面

| 页面 | 是否适合 | 建议承载内容 |
| --- | --- | --- |
| `SettingsPage` | 适合 | 全局默认运行记录策略、后台默认档位、事件保留上限、审计默认等级 |
| `WorkflowPage` / `WorkflowSummaryView` | 最适合 | 当前工作流默认记录策略、运行按钮旁的本次运行档位、工作流级性能提示 |
| `WorkflowSelectedNodeConfigView` | 最适合 | 选中节点的记录覆盖策略，例如是否继承工作流、是否增强记录、是否保留中间结果 |
| `WorkflowNodeListView` | 适合展示 | 在节点列表中显示该节点是否有覆盖策略、是否建议增强记录 |
| `RunMonitorPage` | 最适合分析 | 展示每次运行实际使用的策略、耗时最长节点、失败节点、建议增强记录的节点 |
| `NodeRunListView` | 适合展示 | 标记慢节点、长运行节点、记录等级、进度模式和失败摘要 |
| `RunDetailPanelView` | 适合展示和轻量操作 | 展示本次运行的有效策略，并提供“保存为工作流默认”或“保存为节点覆盖”的入口 |
| `LogsAuditPage` | 只适合查看 | 查看 runtime event 和 audit event，不建议承担策略配置主入口 |
| `DataPage` | 不建议 | 数据页应关注 TableRef、SharedPublication 和数据摘要，不适合放运行记录策略 |

### 是否需要新页面

短期不需要新页面。

当前已有页面可以承载：

```text
SettingsPage：全局默认
WorkflowPage：工作流默认和节点覆盖
RunMonitorPage：运行后分析和建议
LogsAuditPage：日志审计查看
```

如果后续策略变复杂，再新建独立页面：

```text
Run Policy Page / 运行策略页面
```

新页面适合承载：

- 所有工作流的后台策略总览。
- 批量设置多个工作流默认档位。
- 批量查看哪些节点开启了高成本记录。
- 按历史耗时自动生成慢节点候选清单。
- 查看当前后台运行资源预算。

### 推荐第一版界面落点

第一版建议不要新建页面，按最小闭环做：

1. `SettingsPage` 增加全局默认后台档位。
2. `WorkflowSummaryView` 增加当前工作流默认档位。
3. `WorkflowSelectedNodeConfigView` 增加选中节点覆盖策略。
4. `RunMonitorPage` 显示本次运行使用的有效策略和慢节点建议。

这样既能满足“整个工作流先设最少”，也能满足“耗时最长的节点单独增强记录”。

## 十二、当前建议顺序

建议后续按以下顺序推进：

```text
节点基础设施
-> 节点配置模板
-> 资源配置映射
-> 节点数据契约
-> 节点运行辅助
-> 共享数据版本策略
-> 常用业务节点主线
-> 节点目录和配置体验
-> 数据调试和质量检查
-> 节点安全与稳定性
-> 失败策略、取消传播和资源回收
-> 高级节点生态
```

核心原则：

- 先统一新增节点的方法，再大量增加节点。
- 先做能跑通真实流程的节点，再做高级可视化能力。
- 先保证节点可测试、可解释、可维护，再考虑插件化和复杂生态。
