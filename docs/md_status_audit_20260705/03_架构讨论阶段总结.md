# 核心阶段边界调整：取消工作流内置沙盒与信任分级

> 更新日期：2026-06-26
> 状态：正式架构决策
> 优先级：本节覆盖文档中此前关于“节点信任等级、低权限插件进程和平台内置沙盒”的要求。

## 调整结论

当前核心阶段不在工作流平台内部定义或实现：

```text
节点 / 插件信任等级
统一低权限插件进程
通用插件沙盒
外部文件、Web、Office等系统资源的强制隔离规则
```

正式改为：

```text
工作流平台负责数据、数据流、节点调度、运行状态、权限标志和输出记录。
节点或插件负责自身的数据获取、业务处理和外部写回。
具体执行程序自行决定是否使用独立进程、低权限环境、容器或沙盒。
平台不把外部执行安全策略写入核心节点协议。
```

## 核心平台管理范围

核心平台只管理：

```text
工作流定义和连接关系
节点启用、跳过、预览和执行状态
节点输入输出协议
表、字段和数据引用
数据版本、ChangeSet和血缘
工作流进程与Worker生命周期
节点心跳、超时、取消和错误
工作流内部表级、字段级和写入模式权限
审计等级和运行记录
```

核心平台暂不统一管理：

```text
外部程序如何获得系统权限
插件代码是否可信
文件系统、网络、Office等操作系统级隔离
执行程序内部采用何种沙盒
第三方程序的安全模型
```

## 节点和执行程序的边界

节点或插件负责：

```text
获取数据
解析和转换数据
执行业务逻辑
连接外部程序或资源
选择自身运行环境
需要时自行启用进程隔离或沙盒
返回标准NodeResult / DataRef
```

平台只要求节点遵守统一的数据交接和运行协议，不要求节点采用统一的外部资源访问实现。

## 执行标志的定位

节点仍可声明：

```text
enabled
requires_external_access
has_external_side_effects
read_only
supports_preview
requires_confirmation
execution_mode
```

这些标志用于：

```text
计划校验
UI提示
运行路由
执行确认
审计记录
选择进程内执行或独立Worker
```

这些标志不是平台级沙盒规则，也不表示核心平台负责限制操作系统资源。

## 独立Worker继续保留

取消内置沙盒不等于取消进程隔离。

以下任务仍可根据稳定性和生命周期使用独立Worker：

```text
Office COM
浏览器自动化
模型推理
屏幕识别
动作执行
长连接
外部脚本
高内存或易阻塞任务
```

独立Worker解决的是：

```text
卡死隔离
崩溃隔离
资源回收
常驻生命周期
依赖环境分离
```

是否在Worker内部进一步使用低权限、容器或沙盒，由该Worker或外部执行程序自行决定。

## 权限层的收缩

权限层当前重点管理工作流内部的数据操作：

```text
节点是否允许执行
允许读取哪些输入表和字段
允许写入哪些目标表和字段
允许采用哪种写入模式
是否允许创建、追加、覆盖、清空或修改结构
是否允许登记和发布节点输出
权限句柄的有效期、派生和撤销
```

外部访问和外部副作用标志可以继续记录，但在核心阶段主要用于提示、确认和审计，不要求平台对所有外部系统形成统一强制规则。

## 最终边界

```text
权限层：管理工作流内部数据权限和节点执行许可
节点层：获取、处理、筛选和写回具体数据
运行层：调度、进程、心跳、取消和错误处理
数据管理层：登记数据引用、版本、Schema、血缘和生命周期
外部执行层：自行决定进程隔离、系统权限和沙盒策略
```

# 项目方向补充：工作流运行进程与故障域正式分隔

> 更新日期：2026-06-26
> 状态：正式架构决策
> 说明：本节根据故障域隔离原则，对平台进程结构进行定向调整。权限管理部分继续沿用原方案，本节重点调整工作流运行、普通节点、高风险Worker、Supervisor和后台核心服务之间的边界。

## 核心决策

平台采用：

```text
一个职责一个模块
一个故障域一个进程
```

但不采用“每个模块都独立进程”的机械拆分方式。

正式确定：

```text
EngineHost不直接执行用户节点业务代码。
每次活跃的WorkflowRun默认运行在独立WorkflowRunProcess中。
普通节点默认在所属WorkflowRunProcess内部执行。
高风险节点、第三方插件和常驻能力继续由独立Worker进程执行。
Supervisor统一监督WorkflowRunProcess和各类Worker进程。
```

## 更新后的总体进程结构

```text
可选Guardian
│
└─ EngineHost
   ├─ WorkflowScheduler
   ├─ Supervisor
   ├─ TableRegistry
   ├─ RuntimeDataRegistry
   ├─ EventRouter
   ├─ RuntimeStore
   └─ Gateway管理
       │
       ├─ WorkflowRunProcess A
       │  ├─ 普通节点任务
       │  ├─ GroupRun
       │  └─ 高风险Worker客户端
       │
       ├─ WorkflowRunProcess B
       │
       ├─ OfficeWorker
       ├─ BrowserWorker
       ├─ VisionWorker
       ├─ ActionWorker
       ├─ 可选WebGateway
       └─ 可选ExternalGateway

Management UI独立运行
```

## EngineHost职责边界

EngineHost负责：

```text
接收控制命令
保存平台运行状态
调度工作流
创建和监督WorkflowRunProcess
管理资源元数据
管理事件、日志和运行记录
决定取消、终止、重启和资源回收
```

EngineHost不负责：

```text
直接执行普通节点业务逻辑
直接运行第三方插件代码
直接执行Office、浏览器、识别或动作逻辑
在自身进程内承载用户工作流长任务
```

## WorkflowRunProcess定位

一次活跃的工作流运行实例默认对应一个独立的`WorkflowRunProcess`。

该进程负责：

```text
读取本次运行配置快照
解析节点依赖
调度普通节点和组节点
调用高风险Worker
维护工作流局部缓存
上报运行心跳
处理本工作流取消
发布节点结果和最终结果
```

工作流计划本身只是配置和定义，不长期占用进程：

```text
WorkflowDefinition
    ↓ 创建运行实例
WorkflowRun
    ↓ 默认分配
WorkflowRunProcess
```

## 普通节点执行边界

“普通节点默认在所属WorkflowRunProcess内部执行”正式解释为：

```text
普通节点默认在所属WorkflowRunProcess内部执行，
而不是在EngineHost核心服务进程中执行。
```

WorkflowRunProcess内部可以采用：

```text
直接函数调用
普通工作线程
统一线程池
异步任务
SQL执行后端
向量化执行后端
```

普通节点不默认单独创建操作系统进程，以避免高频IPC、序列化和进程启动成本。

## 高风险节点与Worker

以下能力继续使用独立Worker进程：

```text
Office COM
BrowserWorker
VisionWorker
ActionWorker
第三方插件
外部脚本
AI模型
不稳定二进制扩展
网络长连接
可能出现死循环或高内存占用的任务
```

故障隔离层级为：

```text
EngineHost
└─ WorkflowRunProcess
   ├─ 普通节点
   └─ 高风险Worker
```

高风险Worker卡死时，优先终止Worker；若普通节点导致整个WorkflowRunProcess失去响应，则终止当前工作流进程。

## Supervisor监督范围

Supervisor统一监督：

```text
WorkflowRunProcess
PluginWorker
OfficeWorker
BrowserWorker
VisionWorker
ActionWorker
WebGateway
ExternalGateway
其他常驻Worker
```

Supervisor至少维护：

```text
进程ID
所属工作流运行ID
当前状态
最近心跳
当前任务
启动时间
退出原因
重启次数
资源租约
```

典型退出原因：

```text
NORMAL_EXIT
CANCELLED
TIMEOUT
CRASHED
FORCED_TERMINATION
OUT_OF_MEMORY
```

## 两层卡死兜底

### 节点级处理

```text
节点进度长时间不变
→ 检查任务心跳
→ 请求协作取消
→ 高风险Worker无响应时终止Worker
```

### 工作流级处理

```text
WorkflowRunProcess心跳停止
→ Supervisor标记无响应
→ 请求工作流优雅退出
→ 超过宽限时间后终止WorkflowRunProcess
→ 回滚或废弃未发布结果
→ 清理WORKFLOW_SCOPE资源
```

因此，即使普通节点无法安全单独终止，也可以通过工作流进程边界实现最终强制回收。

## 组节点位置

组节点继续作为WorkflowRunProcess内部的逻辑执行单元和资源作用域：

```text
WorkflowRunProcess
└─ GroupRun
   ├─ GroupContext
   ├─ 普通内部节点
   ├─ 高风险Worker任务
   └─ GROUP_SCOPE资源
```

组节点默认不单独创建操作系统进程，避免组嵌套形成复杂进程树。

## 控制面与数据面保持不变

控制命令继续经过EngineHost；大型数据不通过EngineHost反复复制，而是使用：

```text
TableRef
共享内存
Arrow IPC
内存映射文件
临时数据库
持久数据库
```

EngineHost只管理元数据、版本、租约和句柄。

## Guardian定位

Guardian作为可选最外层守护进程，负责监测EngineHost存活、记录异常退出、按策略重启以及清理明显残留子进程。该能力保留为后续稳定运行阶段功能，不作为第一版强制内容。

## WebGateway与ExternalGateway

WebGateway在启用Web管理时建议独立进程；第一版本地桌面UI阶段可暂不实现。

ExternalGateway负责外部程序注册、协议版本、心跳、事件订阅、数据发布和格式转换。第一版可先作为EngineHost内部逻辑模块，后续再根据连接量和安全要求拆分进程。

## 更新后的MVP运行链路

```text
启动EngineHost
→ Management UI连接后台
→ 创建WorkflowRunProcess
→ 普通节点在WorkflowRunProcess内执行
→ 高风险节点调用独立Worker
→ Supervisor获取工作流和Worker心跳
→ 用户取消工作流
→ 正常退出或强制终止WorkflowRunProcess
→ 回收WORKFLOW_SCOPE资源
→ UI继续连接EngineHost并查看结果
```

第一版可以只支持：

```text
单个WorkflowRunProcess串行执行
少量普通节点
一个高风险Worker
基础心跳
基础取消
进程异常退出记录
```

## 与原方案的关系

本次更新不替换以下内容：

```text
节点标准协议
RuntimeContext
NodeResult
组节点设计
字段调试器
ChangeSet
TableRegistry
资源版本和租约
节点化多数据源扩展
高风险Worker隔离
控制面和数据面分离
```

本次只正式明确：

```text
EngineHost不执行节点
WorkflowRun默认独立进程
普通节点在WorkflowRunProcess内执行
Supervisor同时监督工作流进程和Worker
```

# 项目方向补充：数据源扩展、权限与运行数据管理边界

> 更新日期：2026-06-26
> 状态：正式架构决策
> 说明：本节修正此前将多种外部数据源统一收口到核心资源代理和统一数据存储模块的设想。平台核心不持续内置文件、Web、Office、数据库、串口、屏幕等数据源实现；具体资源获取、转换和写回由节点或插件负责。

## 核心结论

正式确定：

```text
权限检查始终在线，但权限系统不代替节点执行资源操作。
节点和插件负责获取、解析、处理和写回具体数据。
平台核心只管理标准化后的输入输出、引用、版本、血缘和生命周期。
多数据源通过新增节点和插件扩展，不通过持续修改平台核心扩展。
平台统一的是运行与数据交接协议，不是所有业务数据源的访问实现。
```

## 取消的核心模块设想

以下模块不再作为平台核心的强制组成部分：

```text
通用ResourceBroker
强制所有结果进入统一DatasetStore
核心层内置全部数据库、文件、Web和Office适配器
由中央模块代替节点执行所有真实资源访问
```

原因是：

- 新数据源会不断增加，核心模块无法持续枚举和维护。
- 不同资源需要不同依赖、生命周期和故障隔离方式。
- 强制统一存储会导致格式转换、重复写入和性能损耗。
- 节点和插件失去独立封装能力，平台核心逐渐变成巨型业务模块。

## 平台核心职责

平台核心负责：

```text
工作流调度
进程与Worker生命周期
权限申请、批准、句柄和撤销
节点输入输出协议
运行数据引用登记
数据版本、来源和血缘
字段关系和ChangeSet
进度、日志、心跳和错误
结果路由与生命周期回收
```

平台核心不负责：

```text
打开具体文件
访问具体网页
控制Office COM
连接所有类型数据库
解析每一种外部格式
替节点实现数据采集和写回
```

## 节点和插件职责

节点或插件完整负责自身业务能力：

```text
获取外部数据
解析和转换数据
执行业务处理
选择合适的输出承载方式
写回外部资源
返回标准NodeResult
```

典型示例：

```text
网页采集节点：登录、访问、等待、提取、整理、输出
Office插件：打开、读取、解析、修改、保存、关闭COM对象
数据库节点：连接、查询、批量写入、事务处理、输出结果
文件节点：枚举、读取、解析、创建、覆盖或重命名
```

用户层保持节点化，底层实现保持插件化；新增数据源只需新增节点或插件，不要求修改EngineHost。

## 权限管理边界

权限管理当前重点判断：

```text
节点是否允许执行
是否允许读取指定输入表和字段
是否允许写入指定目标表和字段
是否允许采用追加、覆盖、清空或结构修改等写入模式
是否允许登记和发布节点输出
是否需要确认、严格模式或更高审计等级
权限句柄的范围、有效期、派生和撤销
```

权限管理不直接：

```text
读取文件
访问网页
连接数据库
调用Office
保存全部节点数据
定义插件可信等级
统一限制外部程序的系统权限
```

外部访问和外部副作用仍可由节点声明，用于提示、确认、运行路由和审计；具体隔离策略由节点或外部执行程序自行实现。

## 节点执行标志

节点至少声明：

```text
enabled                   是否启用
requires_external_access  是否访问外部资源
has_external_side_effects 是否产生外部副作用
read_only                 是否只读
supports_preview          是否支持无副作用预览
requires_confirmation     是否需要执行前确认
```

这些标志用于计划校验、UI提示、运行路由、执行确认和审计记录。

这些标志只描述节点行为，不构成平台级操作系统安全规则。节点或外部执行程序可以根据自身需要选择普通进程、独立Worker或自定义隔离环境。

## 标准数据交接

平台不统一所有数据源实现，但统一节点之间的数据交接方式。

节点输出使用：

```text
NodeResult
└─ DataOutput / DataRef
```

允许的输出承载形式包括：

```text
inline             小型字段或结构化对象
memory_table       工作流进程内表对象
runtime_table      工作流运行数据库中的表
external_table_ref 外部数据库表或查询结果引用
arrow_ipc          跨进程大型表
shared_memory      高频或低延迟共享数据
file_ref           文件结果引用
stream_ref         实时事件或数据流
```

节点自行选择合适承载形式，平台负责验证输出描述、登记引用、路由给下游和管理生命周期。

## RuntimeDataRegistry定位

平台保留轻量`RuntimeDataRegistry`，但它不是统一数据仓库。

它只登记：

```text
data_id
数据类型
存储方式
句柄或位置引用
Schema
创建节点和工作流
版本
生命周期
当前使用者
有效状态
清理责任方
```

它不强制复制或保存实际数据，也不要求所有数据转换为同一种格式。

## 资源配置映射

可保留轻量`ResourceProfileRegistry`，用于减少节点重复填写路径和连接参数：

```text
生产物料数据库 → 数据库类型、连接配置、默认表
PDM系统        → 基础地址、登录会话配置
标准文件目录   → 实际目录路径
确认函模板     → 模板文件位置
```

`ResourceProfileRegistry`只提供配置映射，不执行资源访问。

节点从配置目录取得所需配置后，自行完成资源操作。敏感连接信息应按权限范围提供，避免直接暴露给无权节点。

## 数据透明规则

数据透明不要求平台亲自获取数据，而要求节点输出后留下统一记录：

```text
数据由哪个节点产生
来源类型和资源配置
获取或处理时间
关键配置摘要
Schema和行数
输出承载方式
数据版本
下游使用关系
字段变化和ChangeSet
```

节点结果可以按保留等级管理：

```text
SUMMARY_ONLY   仅摘要和血缘
DIFF_ONLY      摘要加差分
INSPECTABLE    保留可查看结果一段时间
PERSISTENT     明确持久化
```

## 数据源扩展规则

多数据源通过节点和插件扩展：

```text
SQLite节点
DuckDB节点
MySQL插件
PostgreSQL插件
Web采集节点
Office插件
串口节点
屏幕识别插件
自定义企业系统插件
```

平台可提供可选SDK和工具函数，但不要求所有节点使用统一资源代理。

## 与进程分隔方案的结合

```text
EngineHost
└─ WorkflowRunProcess
   ├─ 普通节点直接执行自身数据操作
   ├─ 高风险节点调用独立Worker
   ├─ 节点返回NodeResult / DataRef
   └─ RuntimeDataRegistry登记输出引用
```

Supervisor负责处理节点或Worker卡住、退出、超时后的终止和引用清理。

## 最终边界

```text
权限层：决定是否允许执行和允许哪些能力
节点层：真正获取、处理和写回数据
运行层：调度、状态、进程和错误管理
数据管理层：登记节点交回的数据引用、版本和血缘
外部执行层：自行决定系统权限、进程隔离和沙盒策略
```

# 项目方向升级：从桌面数据工作流工具到本地工作流运行平台

> 升级日期：2026-06-26
> 本节性质：顶层架构定义
> 优先级说明：当本节与文档早期关于“单桌面程序、单工作流、单数据库”的假设发生冲突时，以本节为准。早期节点、权限、配置、组节点、字段调试和运行协议仍然作为平台内部核心设计保留。

## A.1 项目新定位

项目由原来的：

```text
模块化数据工作流桌面工具
```

升级为：

```text
面向数据处理、自动化和智能体接入的本地模块化工作流运行平台
```

平台主要承担：

- 工作流设计、保存、预览和执行。
- 多个工作流实例的后台运行和统一调度。
- 普通节点、插件节点、识别Worker、动作Worker等进程管理。
- 跨工作流共享数据表、实时状态表和事件流。
- 通过节点和插件接入SQLite、DuckDB、文件、Web、Office及后续外部数据源。
- 屏幕识别、UI智能体、自动点击、Office处理和文件监控等项目接入。
- 权限检查、审计分级、运行监控和字段调试。

项目仍以实用性为首要目标，不以一次性实现完整分布式平台为目标。

## A.2 本次升级不推翻的内容

以下设计继续保留，并成为平台内部的执行核心：

```text
NodeDefinition
IOSchema
ConfigSchema
NodeResult
RuntimeContext
ExecutionController
PermissionManager
组节点
字段调试器
ChangeSet
稳定table_id / field_id / row_id
节点和插件生命周期管理
```

本次升级主要增加：

```text
EngineHost后台服务
Management UI管理客户端
WorkflowScheduler多工作流调度
Supervisor进程监督
TableRegistry共享表注册中心
EventRouter事件路由
RuntimeDataRegistry运行数据引用管理
跨工作流资源生命周期
控制面与数据面通信
```

## A.3 新总体架构

```text
┌─────────────────────────────────────────┐
│ Management UI / Workflow Editor         │
│ 管理窗口、编辑器、日志、调试和数据查看   │
└───────────────────┬─────────────────────┘
                    │ 控制面通信
                    ▼
┌─────────────────────────────────────────┐
│ EngineHost 后台运行服务                  │
│ WorkflowScheduler / Supervisor          │
│ PermissionManager / AuditManager        │
│ EventRouter / TableRegistry             │
│ RuntimeDataRegistry / RuntimeStore      │
└───────────────────┬─────────────────────┘
                    │ 创建并监督
                    ▼
┌─────────────────────────────────────────┐
│ WorkflowRunProcess                      │
│ 普通节点 / GroupRun / Worker客户端       │
└──────────────┬───────────────┬──────────┘
               │               │
               ▼               ▼
       节点自行访问数据源     高风险Worker
               │               │
               └───────┬───────┘
                       ▼
          标准NodeResult / DataRef / 事件流
```

### 管理窗口

管理窗口只是客户端，主要负责：

- 编辑和配置工作流。
- 启动、暂停、恢复和停止工作流。
- 查看节点、组节点、插件和Worker状态。
- 查看共享表版本、运行数据引用、资源配置和审计状态。
- 使用字段搜索、监视器、断点和单值追踪。

管理窗口关闭后，后台工作流可以继续运行；重新打开后可重新连接`EngineHost`。

### EngineHost

`EngineHost`是后台主服务，负责：

- 保存平台运行状态。
- 接收管理客户端和外部程序的控制请求。
- 创建、监督和回收WorkflowRunProcess。
- 管理节点输出的数据引用、共享表版本和运行数据生命周期。
- 监督WorkflowRunProcess、插件和Worker进程。
- 汇总日志、进度、心跳和错误。

`EngineHost`不直接打开具体文件、连接所有数据库、访问网页或调用Office；这些行为由相应节点或插件完成。

## A.4 控制面与数据面

平台通信分为两类。

### 控制面

传递体积较小的命令和状态：

```text
启动工作流
停止工作流
暂停与恢复
查询运行状态
修改工作流参数
设置审计等级
订阅日志和事件
查询共享表句柄
```

第一版可使用本地命名管道或封装后的本地RPC；如果后续需要C++、C#、Java、Web等多语言接入，可升级为gRPC。

### 数据面

传递或访问体积较大的数据：

```text
大型表格
图像帧
识别特征
模型输入输出
节点快照
中转副表
```

控制消息中尽量只传递：

```text
resource_id
version
storage_type
handle
schema_id
访问模式
```

真实数据由共享内存、内存映射文件、Arrow、临时数据库或持久数据库承载，避免通过JSON或普通RPC重复复制大对象。

## A.5 多工作流运行模型

平台允许同时存在多个：

```text
WorkflowDefinition   工作流定义
WorkflowInstance     工作流实例
WorkflowRun          一次运行记录
```

同一个工作流定义可以创建多个运行实例，但每次运行必须拥有独立的：

```text
workflow_run_id
配置快照
输入资源版本
节点运行记录
权限上下文
资源租约
审计等级
```

`WorkflowScheduler`负责：

- 工作流排队和并发限制。
- 启动、暂停、恢复和停止。
- 工作流依赖和事件触发。
- 运行优先级。
- 资源不足时的等待和降级。

## A.6 共享表与跨工作流数据资源

原来的“中转副表”升级为标准运行数据引用体系。节点和插件负责产生实际数据，平台只登记其引用、版本和生命周期。

至少区分：

```text
组内临时表
工作流内部临时表
跨工作流共享表
持久化中转表
实时状态表
事件数据流
图像帧缓冲区
```

表类型数据由`TableRegistry`登记，其他节点输出由轻量`RuntimeDataRegistry`登记：

```text
table_id / resource_id
version
schema
storage_type
storage_handle
created_by
updated_by
readers
write_owner
lifecycle
status
```

共享表原则上采用：

```text
不可变版本
写时复制
原子切换current版本
读取租约
无活跃引用后释放旧版本
```

避免多个工作流直接同时修改同一份可变内存表。

## A.7 共享数据生命周期

共享资源需要独立于节点生命周期进行管理。

建议状态：

```text
CREATED
PUBLISHED
ACTIVE
SUPERSEDED
RELEASING
RELEASED
ORPHANED
```

读取方获得`TableLease / ResourceLease`：

```text
resource_id
version
workflow_run_id
access_mode
acquired_at
expires_at
```

释放资源需要同时满足：

- 没有活跃读取租约。
- 没有进行中的写任务。
- 不再是当前版本，或已经超过保留时间。
- 不被运行快照、调试记录或回放功能引用。

后台服务启动时，需要扫描并清理异常退出遗留的孤儿资源。

## A.8 Worker与插件统一管理

插件不再只理解为“节点的外部实现”，而是统一纳入Worker体系。

Worker类型可以包括：

```text
PluginWorker
ModelWorker
ScreenCaptureWorker
RecognitionWorker
ActionWorker
FileWatchWorker
SerialWorker
OfficeWorker
```

`Supervisor`负责：

- 创建、启动和停止Worker。
- 心跳和健康检查。
- 进程崩溃检测。
- 正常取消和强制终止。
- 重启策略。
- CPU、内存和任务状态监控。
- 清理残留共享内存、子进程和临时文件。

不同Worker可以使用不同生命周期策略：

```text
一次性进程      低频、任务独立、易污染环境的插件或外部任务
常驻专用进程    模型、Office、串口、屏幕识别等高初始化成本任务
进程池          大量相似、独立的CPU计算任务
```

## A.9 UI智能体与实时数据接入

UI智能体应区分快速路径和工作流路径。

### 快速路径

```text
屏幕捕获
→ 特征识别
→ 快速规则判断
→ 动作执行
```

由常驻Worker完成，强调低延迟、有界队列和只保留最新帧。

### 工作流路径

```text
识别事件
→ 数据整理
→ 特征筛选
→ 历史分析
→ 规则更新
→ 数据归档
```

工作流主要处理结构化结果，不默认逐帧接收和保存完整屏幕图像。

建议数据资源：

```text
FrameStream    最近图像帧环形缓冲
FeatureEvent  识别结果和特征事件
StateTable    当前场景和目标状态
```

自动点击、键盘和系统操作必须通过独立`ActionManager / ActionWorker`执行，并接受权限、频率、目标窗口和紧急停止控制。

## A.10 多数据源扩展与节点职责

平台核心不再通过“统一数据源注册中心 + 全量适配器”实现所有数据源访问。

正式采用：

```text
具体数据源访问由节点或插件实现
平台只规定权限申请、节点输出和运行数据引用协议
```

可扩展节点和插件示例：

```text
SQLite读取/写入节点
DuckDB分析节点
MySQL或PostgreSQL插件
文件读取与写入节点
Web采集节点
Office读取与写回插件
串口、屏幕识别和外部程序插件
```

平台可以提供可选SDK、连接工具和公共数据结构，但不要求新增数据源时修改EngineHost。

节点可从`ResourceProfileRegistry`读取已登记的逻辑配置，例如：

```text
resource_profile://production_sqlite
resource_profile://pdm_system
resource_profile://standard_files
```

该注册表只负责配置映射，不执行数据库、文件、Web或Office操作。

节点运行后返回统一的：

```text
NodeResult
DataOutput
DataRef
```

`RuntimeDataRegistry`登记输出引用、Schema、来源、版本和生命周期，不强制把所有结果复制到统一数据库。

多数据库工作流第一阶段不承诺全局分布式事务。涉及多个数据库或外部系统时，由相关节点按照自身协议执行：

```text
预览或校验
分阶段写入
明确提交点
失败记录和补偿动作
```

## A.11 权限检查与审计分级

权限检查与详细审计必须分开。

### 权限检查

始终开启，不允许为了性能完全关闭。

权限层当前判断：

```text
节点或插件是否允许进入本次工作流运行
允许读取哪些输入表、字段和数据引用
允许写入哪些目标表、字段和运行时数据
允许采用哪种写入模式
是否允许创建、追加、覆盖、清空或修改结构
是否允许登记和发布节点输出
是否需要执行前确认
权限句柄是否仍然有效
```

文件、数据库、Web、Office、串口和屏幕等具体访问由对应节点或插件完成。外部访问、外部副作用和执行模式可以作为节点标志被记录和提示，但当前核心权限层不尝试为所有外部系统建立统一强制规则。

是否需要系统级隔离，由对应节点、Worker或外部执行程序自行决定。

### 审计等级

```text
MINIMAL  记录工作流、节点、资源配置、允许/拒绝、主要错误
BASIC    增加节点输入输出、影响行数、版本、插件调用和ChangeSet摘要
FULL     增加字段变化、行级差分、关键参数和完整权限追踪
DEBUG    增加规则命中、调用链、IPC耗时和异常诊断
```

审计记录采用异步队列和批量落盘，不能让节点长期同步等待日志写入。

“关闭审计”实际表示降为`MINIMAL`，不表示取消权限检查和最低运行记录。

## A.12 生命周期层级升级

后续生命周期管理不再只覆盖节点和插件，而是包括：

```text
服务生命周期
管理客户端连接生命周期
工作流实例生命周期
组节点生命周期
节点任务生命周期
Worker / 插件进程生命周期
共享数据资源生命周期
数据库连接与事务生命周期
事件订阅生命周期
```

取消和关闭需要按层级传播：

```text
平台关闭
→ 停止接收新任务
→ 工作流进入DRAINING / STOPPING
→ 组和节点收到取消
→ Worker正常退出
→ 超时后强制终止
→ 提交或回滚事务
→ 释放租约和共享资源
→ 关闭数据库连接
```

## A.13 后台管理窗口方向

后台管理窗口建议至少包含：

```text
工作流页    状态、当前节点、运行时间、启动/暂停/停止
进程页      PID、CPU、内存、心跳、任务、重启次数
共享数据页  表版本、存储方式、占用、读写者、生命周期
运行数据页  数据引用、存储方式、版本、生命周期和错误
审计页      审计等级、日志队列、降级状态
智能体页    捕获、识别、动作Worker和紧急停止
```

UI不直接持有工作流执行状态，而是从`EngineHost`读取。

## A.14 升级后的第一版MVP

第一版仍然要控制范围，建议只验证以下完整链路：

```text
EngineHost独立运行
→ Management UI连接后台
→ UI启动一个工作流
→ 一个外部Worker发布共享表
→ 两个工作流读取同一表版本
→ 其中一个工作流生成新版本
→ 管理窗口查看进程、表版本和运行状态
→ 停止后正确释放资源
```

第一版建议包含：

- `EngineHost`后台服务。
- 最小管理窗口。
- `WorkflowScheduler`。
- `Supervisor`。
- 现有节点标准协议。
- 普通节点默认在所属WorkflowRunProcess内部执行。
- 一个插件或Worker子进程。
- `TableRegistry`。
- 一个返回内存表或运行表引用的示例节点。
- 一个自行连接SQLite并返回标准NodeResult的示例节点。
- 本地控制通信。
- 基础节点执行权限、外部访问标志和副作用确认。
- `MINIMAL / BASIC`两档审计。
- 基础状态、心跳和资源回收。

第一版暂不包含：

- 远程机器连接。
- 完整分布式运行。
- 全部数据库适配器。
- 完整零拷贝体系。
- 多数据库全局事务。
- 完整UI智能体。
- 全量字段和单元格血缘。
- 复杂自动恢复和集群调度。

## A.15 对早期假设的正式升级

```text
旧：主UI是主程序
新：EngineHost是后台主服务，UI是管理客户端

旧：工作流之间相互独立
新：工作流可以通过共享表、资源版本和事件交换数据

旧：中转副表只属于一个工作流
新：数据资源可以拥有组级、工作流级、跨工作流和持久化生命周期

旧：核心层通过统一数据源适配器连接具体存储
新：节点和插件自行接入具体数据源，平台统一NodeResult、DataRef、版本和生命周期

旧：插件只是节点的外部实现
新：插件、识别器、模型和动作执行器统一作为受Supervisor管理的Worker

旧：关闭权限审计可提升性能
新：权限强制检查始终存在，只允许降低详细审计等级
```

## A.16 后续文档使用方式

现有后续章节继续保留，按以下关系理解：

- 原“节点协议、配置、权限”章节：平台节点内核。
- 原“组节点”章节：工作流内部复用和局部生命周期。
- 原“字段调试器”章节：平台调试和数据追踪能力。
- 原“插件进程生命周期”章节：Supervisor和Worker设计基础。
- 原“中转副表”章节：TableRegistry与共享资源设计基础。

后续讨论应优先围绕：

1. `EngineHost`与管理UI通信协议。
2. 工作流实例、Worker和资源的完整生命周期。
3. NodeResult、DataRef、共享表版本、租约和清理规则。
4. 节点/插件的数据源扩展协议与多工作流并发。
5. 组节点与平台级运行调度的边界。
6. 第一版MVP的具体目录和接口。

## 1. 当前阶段判断

现有数据工作流项目已经完成了大量功能开发和结构重构，具备较丰富的节点、插件、表映射、权限管理、条件跳转和Office处理能力。

但随着项目进入后期，继续在原结构上进行硬兼容，已经产生明显成本：

- 历史代码和旧UI形成较强耦合。
- 新接口需要兼容大量旧节点和特殊逻辑。
- 每次重构都要同时考虑旧模板、旧数据结构和旧调用方式。
- 为保证兼容，AI分析、代码核查和修改消耗大量算力。
- 新功能经常需要先处理历史包袱，再处理功能本身。
- 架构验证容易被旧项目中的具体问题干扰。
- 很难判断新增抽象究竟是合理设计，还是为了兼容旧代码形成的补丁。

因此，当前更合适的方向不是继续无限扩展旧项目，而是：

> 以现有项目中已经验证过的需求和经验为基础，新建一个扩展性、模块化和可替换性更好的工作流项目。

旧项目可以继续作为：

1. 实际工作的稳定工具。
2. 需求验证和业务逻辑参考。
3. 新项目功能行为的对照样本。
4. 部分成熟算法和节点逻辑的迁移来源。

新项目则用于：

1. 验证新的节点协议。
2. 建立UI与执行逻辑解耦结构。
3. 建立统一权限、节点输出和运行数据引用机制。
4. 建立标准运行状态、数据交接与诊断体系。
5. 训练系统拆解、接口设计和架构能力。

---

## 2. 新项目的基本原则

### 2.1 不以复制旧项目为目标

新项目不应一开始就追求覆盖旧项目全部功能。

更合理的目标是：

> 先建立一条结构正确、职责清晰、可以持续扩展的最小工作流执行链路。

第一阶段只需要验证：

- 节点能够注册。
- UI能够读取节点配置描述。
- 用户配置能够形成标准配置对象。
- 节点能够接收输入并执行。
- 权限能够集中审核。
- 执行状态能够被主程序观察。
- 结果能够传给下一个节点。
- 相同节点可以被不同UI调用。

### 2.2 优先建立协议，再增加节点数量

旧项目的经验说明，单个节点的小逻辑通常可以解决，真正困难的是后期统一和兼容。

因此，新项目优先级应当是：

1. 节点协议。
2. 输入输出协议。
3. 配置协议。
4. 权限协议。
5. 运行状态协议。
6. 数据交接与引用协议。
7. UI渲染协议。
8. 节点和插件功能。

### 2.3 采用渐进式通用化

不要求所有节点完全使用统一表单，也不要求所有插件运行方式完全相同。

建议采用“90/10”结构：

- 约80%～90%的普通节点使用标准配置Schema和标准执行接口。
- 少量复杂节点允许使用自定义UI。
- 自定义UI最终仍需输出标准配置对象。
- 高风险插件允许使用独立子进程。
- 所有节点最终都通过统一结果和状态协议与主程序交互。

## 3. 总体架构方向（节点内核视角）

> 说明：本节保留为节点内核的分层结构。平台级总体架构以文档开头“项目方向升级”章节为准。


建议把新项目拆分为以下层次：

```text
┌────────────────────────────────────┐
│               UI层                 │
│ Tkinter / Qt / Web / 其他界面      │
└─────────────────┬──────────────────┘
                  │ 读取配置Schema
                  │ 生成标准Config
                  ▼
┌────────────────────────────────────┐
│          节点定义与注册层          │
│ Definition / IO / Config / Policy  │
└─────────────────┬──────────────────┘
                  ▼
┌────────────────────────────────────┐
│          工作流编排与执行层        │
│ DAG / 顺序执行 / 跳转 / 调度       │
└─────────────────┬──────────────────┘
                  ▼
┌────────────────────────────────────┐
│          ExecutionController       │
│ 状态机 / 进度 / 心跳 / 取消 / 超时 │
└──────────────┬──────────────┬──────┘
               │              │
               ▼              ▼
┌───────────────────┐  ┌──────────────────┐
│ PermissionManager │  │  RuntimeContext  │
│ 权限审核与授权    │  │ 日志/进度/心跳等 │
└─────────┬─────────┘  └─────────┬────────┘
          └────────────┬─────────┘
                       ▼
┌────────────────────────────────────┐
│          节点与插件执行层          │
│ 获取 / 解析 / 处理 / 写回具体数据  │
│ execute(inputs, config, context)   │
└─────────────────┬──────────────────┘
                  ▼
┌────────────────────────────────────┐
│       标准数据交接与引用登记层     │
│ NodeResult / DataRef / 版本 / 血缘 │
│ 路由 / 生命周期 / ChangeSet        │
└────────────────────────────────────┘
```

核心思想：

> UI只负责展示和收集配置；节点和插件负责业务逻辑及具体资源操作；主程序负责调度；中央模块负责权限批准、状态、引用和生命周期管理。

## 4. 权限管控设计

### 4.1 基本结论

权限不应完全由每个节点自己控制，也不能完全由中央模块推测。

推荐方式是：

> 节点声明权限需求，中央权限模块审核并生成运行授权。

可以概括为：

> 权限申请在节点，权限决策在中央，具体数据操作仍由节点或插件执行。

### 4.2 节点负责的内容

节点应负责：

- 声明自身可能使用的能力。
- 根据当前配置解析本次运行实际需要的权限。
- 指明要读取或写入的表。
- 指明要操作的字段。
- 指明是否会新建、追加、覆盖或清空数据。
- 指明是否会修改表结构。
- 指明可能产生的副作用。
- 声明是否访问外部资源、是否只读、是否支持预览和是否需要确认。
- 需要时选择已登记的资源配置，但由节点自己完成实际访问。
- 执行后返回标准`NodeResult / DataRef`和必要的来源摘要。

示例：

```python
capabilities = {
    "table.read",
    "field.write",
    "table.append",
    "table.create",
    "table.clear",
}
```

节点还需要提供动态权限解析入口：

```python
def resolve_permissions(inputs, config):
    ...
```

因为同一个节点在不同设置下，需要的权限可能不同。

例如：

- 预览模式：只需要读取权限。
- 覆盖原表：需要字段写入权限。
- 输出新表：需要新建表权限。
- 追加模式：需要追加权限。
- 清空后写入：需要清空和写入权限。

### 4.3 中央模块负责的内容

中央权限模块负责：

- 汇总节点的权限请求。
- 对照工作流、项目或运行策略。
- 判断节点是否允许执行，以及允许访问哪些工作流内部数据。
- 生成本次运行的权限句柄。
- 管理句柄有效期、派生、续期和撤销。
- 记录允许、拒绝和实际输出摘要。
- 检查节点声明、运行配置和标准输出是否一致。
- 记录节点使用的执行模式、输入输出权限和外部访问标志。

中央权限模块不负责：

- 代替节点读取文件、网页、数据库或Office。
- 内置所有数据源适配器。
- 强制保存全部节点实际数据。
- 解析具体业务数据格式。

### 4.4 节点执行与结果登记

节点和插件在获得批准后，自行执行具体资源操作：

```text
权限批准
→ 节点或插件执行读取、处理或写回
→ 返回标准NodeResult
→ RuntimeDataRegistry登记DataRef、Schema、版本和生命周期
```

节点应声明：

```text
enabled
requires_external_access
has_external_side_effects
read_only
supports_preview
requires_confirmation
```

以上声明用于计划校验、UI提示、运行控制、执行确认和审计记录。

平台核心不依据这些声明建立可信等级或统一沙盒；节点、插件或外部执行程序自行决定是否需要额外隔离。

平台不要求节点通过通用`ResourceBroker`访问所有资源，也不要求把全部输出写入统一`DatasetStore`。平台统一的是节点结果协议、数据引用、运行记录和生命周期。

### 4.5 写入模式与权限映射

| 写入行为 | 建议权限 |
|---|---|
| 读取表 | `table.read` |
| 读取字段 | `field.read` |
| 空缺补齐 | `field.write_empty` |
| 覆盖字段 | `field.write` |
| 追加数据 | `table.append` |
| 清空后写入 | `table.clear` + `table.write` |
| 新建表 | `table.create` |
| 新增字段 | `schema.add_field` |
| 删除字段 | `schema.delete_field` |
| 字段改名 | `schema.rename_field` |

“空缺补齐”建议单独作为受限写入权限，避免节点以补齐名义覆盖已有数据。

文件、Web、Office、串口、屏幕和外部程序等节点可以定义自身能力，例如：

```text
file.read / file.write / file.delete
web.request / web.download / web.upload
office.read / office.write
device.serial.read / device.serial.write
action.mouse / action.keyboard / process.launch
```

这些能力由节点声明并由中央权限层批准，但具体操作仍由节点或插件执行。

## 5. 节点标准协议

一个节点建议拆分为四部分。

### 5.1 NodeDefinition

描述节点身份、能力和执行策略。

```python
{
    "type": "batch_replace",
    "version": 1,
    "name": "批量替换",
    "category": "字段处理",
    "capabilities": [],
    "execution_policy": {}
}
```

建议至少包含：

- 节点类型。
- 节点版本。
- 节点名称。
- 节点分类。
- 功能说明。
- 权限能力。
- 执行隔离策略。
- 是否支持取消。
- 是否支持进度。
- 是否需要自定义UI。

### 5.2 IOSchema

描述节点输入和输出。

输入可以包括：

- 数据表。
- 字段集合。
- 文件路径。
- 单值参数。
- 记录集合。
- 上游节点结果。
- 外部资源引用。

输出可以包括：

- 主数据结果。
- 异常数据。
- 未匹配数据。
- 统计结果。
- 文件路径。
- 日志摘要。
- 多分支输出。

业务输出、状态输出和诊断输出应分开，避免下游节点误把日志当作数据。

### 5.3 ConfigSchema

描述节点设置和UI规则。

建议字段定义至少支持：

- 名称。
- 类型。
- 标签。
- 默认值。
- 是否必填。
- 可选项。
- 校验规则。
- 帮助文本。
- 启用条件。
- 显示条件。
- 必填条件。
- 只读条件。
- 动态选项来源。

示例：

```python
{
    "name": "new_table_name",
    "type": "text",
    "label": "新表名称",
    "enabled_if": {
        "field": "output_mode",
        "equals": "new_table"
    },
    "required_if": {
        "field": "output_mode",
        "equals": "new_table"
    }
}
```

### 5.4 RuntimeContract

统一运行入口：

```python
def validate(inputs, config, context):
    ...

def resolve_permissions(inputs, config):
    ...

def execute(inputs, config, context):
    ...
```

基础执行形式：

```python
result = execute(inputs, config, context)
```

其中：

- `inputs`：标准化输入数据。
- `config`：与UI无关的配置对象。
- `context`：运行时公共能力。
- `result`：标准节点结果。

## 6. 节点运行状态与“卡住”检测

### 6.1 最终结果与实时状态必须分离

节点完成后可以返回：

```python
NodeResult(
    outputs={},
    affected_rows=125,
    skipped_rows=8,
    warnings=[],
    metrics={}
)
```

但运行中的状态不能等到执行结束才返回。

节点需要通过`RuntimeContext`实时上报：

```python
context.report_progress(...)
context.heartbeat()
context.log(...)
context.check_cancelled()
```

### 6.2 主程序不能直接询问节点是否卡住

真正卡住的节点通常无法响应查询。

因此，应由独立的`ExecutionController`从外部观察：

- 节点启动时间。
- 当前运行阶段。
- 最近心跳时间。
- 当前进度。
- 最近进度更新时间。
- 最近日志。
- 是否请求取消。
- 是否超过最大运行时间。

### 6.3 建议状态机

```text
PENDING
VALIDATING
WAITING_PERMISSION
RUNNING
CANCEL_REQUESTED
SUCCEEDED
FAILED
TIMED_OUT
SUSPECTED_HUNG
CANCELLED
```

### 6.4 心跳与进度应分开判断

| 心跳 | 进度 | 判断 |
|---|---|---|
| 正常 | 持续变化 | 正常运行 |
| 正常 | 长时间不变 | 可能是耗时操作 |
| 超时 | 不变化 | 疑似卡住 |
| 超过最大时限 | 任意 | 运行超时 |

仅仅进度不变不能直接判定卡死，因为Office COM、文件解析和大型事务可能长时间停留在同一步。

### 6.5 执行隔离策略

普通节点可采用进程内执行：

```python
"isolation": "in_process"
```

高风险插件建议采用子进程：

```python
"isolation": "subprocess"
```

适合子进程隔离的节点：

- Office COM操作。
- 外部脚本。
- 第三方插件。
- 网络调用。
- 可能出现死循环的代码。
- 外部模型调用。
- 不稳定的二进制扩展。

线程模式只能协作式取消，无法可靠强制结束卡死代码；子进程模式则可以在必要时终止进程并清理资源。

## 7. UI设置联动设计

### 7.1 不使用固定True/False表示全部状态

设置是否可用通常依赖其他配置项，因此不能只保存静态：

```python
enabled = True
```

应使用条件规则：

```python
"enabled_if": {
    "field": "use_regex",
    "equals": True
}
```

### 7.2 第一阶段建议支持的规则

#### `enabled_if`

条件不成立时控件变灰，但可以保留原值。

#### `visible_if`

条件不成立时隐藏控件。

#### `required_if`

满足条件时必须填写。

#### `readonly_if`

允许显示但禁止修改。

#### `options_source`

根据其他配置动态加载候选项，例如：

- 选择表后加载字段。
- 选择节点后加载输出端口。
- 选择数据库后加载表名。

### 7.3 禁用后的值处理

建议支持：

```text
preserve       保留原值，执行时忽略
clear          自动清空
reset_default  恢复默认值
```

默认采用：

> UI禁用时保留原值，但在生成有效配置和运行校验时忽略。

这样用户重新启用选项时，不需要重复输入。

### 7.4 UI联动不能替代后端校验

配置可能来自：

- 旧模板。
- JSON文件。
- API调用。
- 插件。
- 手工编辑。
- 复制的节点配置。

因此，节点执行前必须再次验证配置合法性。

UI规则的职责是改善操作体验，运行前校验的职责是保证安全与正确。

## 8. RuntimeContext建议能力

`RuntimeContext`是节点与主程序之间的运行桥梁，建议提供：

```text
context.inputs             已解析的上游输入和DataRef
context.permissions        本次运行批准的权限句柄
context.resource_profiles  允许读取的资源配置视图
context.register_output    登记NodeResult中的输出引用
context.report_progress    上报进度
context.heartbeat          上报心跳
context.log                写入日志
context.check_cancelled    检查取消请求
context.run_id             工作流运行ID
context.node_run_id        节点运行ID
context.temp               临时资源管理
context.events             运行事件上报
```

`RuntimeContext`不提供一个强制接管所有文件、Web、Office和数据库操作的通用`context.data`代理。节点或插件自行执行具体资源操作，再通过标准输出协议登记结果。

节点不直接依赖主窗口、Treeview、按钮、弹窗或具体UI对象。

## 9. 节点结果建议结构

建议统一使用`NodeResult`：

```python
NodeResult(
    outputs={},
    status="success",
    affected_rows=0,
    skipped_rows=0,
    warnings=[],
    errors=[],
    metrics={},
    diagnostics={}
)
```

其中：

### 数据输出

- 主结果。
- 分支结果。
- 文件结果。
- 异常数据。
- 未匹配数据。

### 指标

- 输入行数。
- 输出行数。
- 修改行数。
- 跳过行数。
- 执行耗时。
- 读取量。
- 写入量。

### 诊断

- 实际操作的表。
- 实际修改的字段。
- 权限使用记录。
- 关键步骤耗时。
- 异常堆栈。
- 插件版本。
- 节点配置摘要。

## 10. 旧项目与新项目的关系

### 10.1 旧项目不立即废弃

旧项目已有大量成熟能力，应继续承担实际工作。

不建议立即停止维护，也不建议一次性迁移全部节点。

### 10.2 不直接复制全部旧代码

新项目迁移功能时，应区分：

#### 可以直接迁移

- 纯算法函数。
- 正则处理逻辑。
- 数据转换逻辑。
- 与UI无关的工具函数。
- 已经稳定的字段处理规则。

#### 需要改造后迁移

- 直接操作SQLite连接的节点。
- 依赖全局变量的节点。
- 执行逻辑与UI混合的节点。
- 内部直接弹窗的插件。
- 使用主程序对象回调的节点。

#### 暂时不迁移

- 高度依赖旧UI状态的节点。
- 尚未稳定的实验功能。
- 为单次场景写的特殊补丁。
- 使用率很低且结构复杂的节点。

### 10.3 新项目优先验证行为，不追求数量

可以选取两个代表性节点：

1. 一个简单节点：例如删除列、填充值或列移动。
2. 一个复杂节点：例如多表高级筛选、Office映射或插件节点。

简单节点用于验证通用流程。

复杂节点用于验证接口是否具有足够的扩展能力。

## 11. 新项目第一阶段建议范围（原节点内核MVP）

> 说明：本节作为节点内核最小闭环保留；平台级第一版MVP以“A.14 升级后的第一版MVP”为准。


第一阶段只实现最小闭环：

```text
创建工作流
→ 注册节点
→ 添加节点
→ 配置节点
→ 校验配置
→ 解析权限
→ 执行节点
→ 上报状态
→ 返回结果
→ 传给下一节点
```

# 后续追加内容：节点可视化关系、表差异与数据追踪设计

> 追加日期：2026-06-25
> 状态：后续方案讨论内容
> 说明：本节是在前述架构方案基础上追加，重点讨论节点关系树、表格重构、字段血缘、单元格追踪、运行预览和内存中转副表。

## 16. 本轮新增目标

节点关系可视化不应只显示“节点A连接节点B”，还应帮助用户回答：

1. 哪个节点改变了表结构。
2. 哪个字段由哪个节点生成。
3. 哪个节点读取、更新或删除了某个字段。
4. 某个结果单元格经过了哪些节点处理。
5. 问题数据第一次在哪个节点出现。
6. 当前关系图是根据配置推导，还是根据真实运行结果生成。
7. 工作流设置变化后，旧关系是否仍然有效。

因此，建议将关系可视化拆分为三种粒度：

```text
工作流节点关系
字段血缘关系
单元格运行追踪
```

三者共用基础标识，但不要求一次显示全部细节。

## 17. 表格重构节点的关系显示

筛选、去重、整行删除、整行追加、列新增、列删除、列移动等操作，会改变表的整体结构。这类节点可以统一归类为：

```text
表格重构节点
```

### 17.1 关系图显示方式

当节点执行的是整体结构变化时：

- 上游表关系统一收束到该节点。
- 不强制为所有字段分别绘制大量连接线。
- 节点上显示“表格重构”标识。
- 节点详情中显示前后表结构和数据统计差异。

### 17.2 节点详情建议显示

```text
输入行数
输出行数
新增行数
删除行数
保留行数
新增字段
删除字段
字段重命名
字段顺序变化
数据类型变化
发生值变化的字段
```

### 17.3 不同重构行为的建议记录方式

| 行为 | 建议记录 |
|---|---|
| 筛选 | 保留行数、移除行数、筛选条件 |
| 去重 | 重复组数量、删除行数、保留策略 |
| 追加行 | 新增行数、来源表或来源节点 |
| 删除行 | 删除行数、删除条件 |
| 新增列 | 新字段ID、字段名、来源或默认值 |
| 删除列 | 被删除字段ID、字段名 |
| 列移动 | 字段顺序变化 |
| 字段改名 | 字段ID不变，显示名称变化 |
| 表合并 | 多个来源表与连接条件 |
| 表拆分 | 输出分支与分支条件 |

表格重构节点只在总览图中使用收束显示，但其内部仍应保存字段和行级变化记录，供进一步追踪。

## 18. 字段血缘关系

当节点没有重构整行数据，只新增、读取、更新或删除某些字段时，可以建立字段级关系。

例如：

```text
节点A生成“板号”
节点B、C仅新增或删除其他列
节点E更新“板号”
```

关系图可直接显示：

```text
节点A.板号 ─────────────→ 节点E.板号
```

中间没有影响“板号”的节点不必强制出现在该字段的关系线上。

### 18.1 字段关系建议分类

```text
CREATE      创建字段
READ        读取字段
UPDATE      更新字段
RENAME      字段改名
COPY        复制字段
DERIVE      根据其他字段生成
MERGE       多字段合并
SPLIT       单字段拆分
DELETE      删除字段
PASS        原样传递
```

### 18.2 字段必须使用稳定标识

字段显示名称会变化，因此不能只依靠字段名建立关系。

建议字段结构：

```python
FieldRef(
    field_id="稳定字段ID",
    display_name="板号",
    source_node_id="node_a",
    source_port="output_1",
    data_type="string"
)
```

字段改名时，`field_id`保持不变，仅更新`display_name`。

### 18.3 关系线显示内容

字段关系线可以显示：

```text
板号
板号 → 新板号
程序编码 + 版本号 → 完整程序标识
源字段拆分 → 型号 / 版本 / 后缀
```

关系总览默认只显示字段名称；选中关系线后，再展开转换规则和运行统计。

## 19. 单元格级数据追踪

用户在某个节点的处理结果中选择一个单元格后，系统应能显示：

1. 当前单元格的值。
2. 当前节点处理前的值。
3. 当前节点处理后的值。
4. 该值来自哪个上游节点。
5. 上游节点对应行和字段中的值。
6. 哪些节点真正修改过该值。
7. 哪些节点只是原样传递。

示意：

```text
节点A：生成板号 HYBP2529GK
   ↓
节点B：原样传递
   ↓
节点C：原样传递
   ↓
节点E：更新为 HYBP2529GK-26
```

选中节点E中的结果单元格后：

- 节点A和节点E高亮。
- 节点B、C可以淡化或折叠。
- 节点A上方显示当时结果。
- 节点E上方显示修改前后值。
- 关系线显示字段名称和变化类型。

### 19.1 单元格追踪的两个必要标识

必须同时存在：

```text
稳定行ID
稳定字段ID
```

建议单元格引用：

```python
CellRef(
    row_id="稳定行ID",
    field_id="稳定字段ID"
)
```

如果仅使用Excel式行号，筛选、排序、插入或删除行后，行号变化会导致关系错误。

### 19.2 行ID来源

优先级建议：

1. 数据本身的主键。
2. 用户指定的组合业务键。
3. 导入时生成的稳定UUID。
4. 临时运行中的内部行ID。

行ID可以不展示给用户，但必须在运行链路内部保留。

### 19.3 单元格来源关系

一个输出单元格可能对应：

- 一个输入单元格：直接复制或修改。
- 多个输入单元格：合并、运算、拼接。
- 一个输入单元格拆成多个输出。
- 多行聚合成一行。
- 外部固定值或用户输入。
- 无直接来源的新值。

因此，单元格血缘建议允许多个来源：

```python
CellLineage(
    output_cell=...,
    source_cells=[...],
    operation="merge / derive / update / aggregate",
    node_id="..."
)
```

## 20. 单元格追踪的性能控制

完整保存每个节点、每个单元格的全部历史，在几十万行、几十个节点下可能产生很大的内存和磁盘占用。因此不建议默认保存完整全量单元格快照。

### 20.1 默认模式：稀疏变更记录

只记录：

- 新增单元格。
- 被修改单元格。
- 被删除单元格。
- 字段来源发生变化的单元格。
- 聚合、拆分和合并关系。
- 行新增、删除或合并信息。

原样传递的数据不逐格复制历史，只记录该字段或表段从上游原样继承。

### 20.2 调试模式：指定范围详细追踪

用户可以针对指定节点、指定字段、指定行ID或指定数据范围开启详细追踪。

### 20.3 按需回放

如果运行时保留了必要输入快照、节点配置和版本，可以在用户选中某个单元格时重新回放相关上游节点。

推荐组合：

```text
常规运行保存稀疏血缘
重点节点保存快照
详细单元格关系按需回放
```

## 21. SQL表前后差异对比

在数十万行以内，使用SQL或基于SQL思想的差异计算通常可行，但性能和准确性取决于是否存在稳定行键和合理索引。

### 21.1 不同差异的计算成本

#### 字段结构差异

新增列、删除列、字段改名、类型变化和顺序变化通常只读取表结构元数据，成本很低。

#### 新增行和删除行

需要基于稳定行ID或业务主键进行连接：

```text
新表有、旧表无 → 新增行
旧表有、新表无 → 删除行
```

主键有索引时，数十万行通常可以较快完成。

#### 同行同字段的单元格差异

先按稳定行ID连接前后表，再比较同一字段。宽表、字段很多时成本会提高，但通过批量SQL、索引和分块处理仍可控制。

### 21.2 最大前提：不能依赖当前行号

筛选、排序、插入或删除后，旧表第100行不一定对应新表第100行。必须依赖主键、业务组合键或导入时生成的稳定行ID。

### 21.3 推荐的两阶段对比

第一阶段先计算行级哈希：

```text
row_id + 参与比较字段 → row_hash
```

若前后哈希一致，则跳过逐字段比较；若哈希不同，再对该行逐字段比较。

```text
按row_id连接
→ 判断新增/删除
→ 比较row_hash
→ 只对哈希变化行逐字段比较
```

### 21.4 字段改名的识别

纯SQL无法可靠判断“旧字段A改名为新字段B”，还是“删除A并新增B”。字段改名最好由节点运行事件直接记录。

### 21.5 推荐采用“运行事件 + 差异验证”

节点主动记录实际操作，差异引擎验证前后结果。例如：

```text
删除行120条
新增字段“板号”
更新字段“程序编码”82个单元格
```

这样比完全依赖前后表反推更准确，也能发现节点声明和实际结果不一致。

## 22. 预览运行与关系图生成

节点关系分为两种来源。

### 22.1 配置关系图

不运行工作流，仅根据配置生成：

- 节点连接。
- 输入输出端口。
- 字段引用。
- 预计读写字段。
- 预计表结构变化。
- 预计权限需求。

这部分属于“预计关系”。

### 22.2 运行关系图

执行预览或正式运行后生成：

- 实际读取字段。
- 实际输出字段。
- 实际修改行数。
- 实际单元格变化。
- 实际分支路径。
- 实际中转副表读写。
- 实际异常位置。

这部分属于“确认关系”。

### 22.3 设置变化后的状态

当工作流设置发生变化时：

- 保留旧运行关系作为历史记录。
- 当前关系图标记为“已过期”。
- 显示配置变化摘要。
- 提示重新预览。
- 新预览完成后生成新的运行关系版本。

建议状态：

```text
未预览
预计关系
已确认关系
配置已变化，关系已过期
预览失败
```

### 22.4 在关系图中执行预览

关系图界面可以提供：

```text
预览整个工作流
预览到当前节点
从当前节点继续预览
仅预览选中分支
仅追踪选中字段
```

## 23. 内存中转副表

内存中转副表本质上是工作流运行时的数据资源，不应被当作普通节点内部变量。建议定义为：

```text
RuntimeTable / TemporaryTable
```

### 23.1 中转副表需要的基础信息

```text
副表ID
显示名称
创建节点
最后更新节点
字段结构
当前行数
生命周期
读节点列表
写节点列表
写入模式
是否允许多节点共享
是否需要持久化
```

### 23.2 生命周期

建议支持：

```text
NODE_SCOPE       当前节点执行期间
GROUP_SCOPE      当前节点组运行期间
WORKFLOW_SCOPE   当前工作流运行期间
SESSION_SCOPE    当前软件会话期间
PERSISTENT       持久化保存
```

初期可以只实现`WORKFLOW_SCOPE`与`PERSISTENT`。

### 23.3 中转副表关系显示

关系图中将中转副表显示为独立资源节点：

```text
节点A ─写入→ 中转副表X ─读取→ 节点D
节点B ─追加→ 中转副表X
节点C ─更新→ 中转副表X
```

### 23.4 中转副表版本

每次写入形成版本：

```text
副表版本V1
副表版本V2
副表版本V3
```

节点读取时记录读取的是哪个版本，方便排查副表被覆盖、追加或后续修改导致的结果变化。

## 25. 性能建议

### 25.1 优先保证稳定标识

```text
稳定行ID
稳定字段ID
稳定表ID
节点运行ID
快照版本ID
```

没有这些标识，后续关系追踪会不断依赖模糊匹配。

### 25.2 不默认保存全量快照

默认保存：

- 表结构。
- 行数。
- 行级哈希。
- 变更集合。
- 少量预览数据。
- 必要运行事件。

仅在用户明确开启、关键节点、写入前后、调试模式或高风险操作时保存完整快照。

### 25.3 优先使用批量处理

避免逐单元格执行SQL查询，使用：

- 批量插入。
- 批量更新。
- 临时表。
- 索引连接。
- 行级哈希。
- 分块处理。
- 事务。
- 只比较发生变化的字段。

### 25.4 预览范围控制

支持：

```text
全量预览
前N行预览
抽样预览
指定条件预览
指定字段预览
指定节点链路预览
```

结构关系和字段血缘可通过抽样快速验证；最终行数、新增行和删除行统计则需要全量运行。

## 26. 本轮方案结论

1. 节点关系图分为节点总览、字段血缘和单元格追踪。
2. 筛选、去重、增删行列等节点在总览中归类为“表格重构节点”，关系线统一收束，但内部仍保存详细变更记录。
3. 字段关系依赖稳定字段ID；单元格追踪同时依赖稳定行ID和稳定字段ID。
4. 完整单元格血缘成本较高，默认采用稀疏变更记录，详细追踪按字段、节点或行范围开启。
5. 数十万行以内的SQL差异对比通常可行，但前提是存在稳定行键和索引；推荐使用“按键连接 + 行哈希过滤 + 变化行逐字段比较”。
6. 节点应主动记录操作事件，差异引擎负责验证，不应完全依赖前后表结果反推所有操作。
7. 关系图分为配置推导的预计关系和预览运行后的确认关系；设置变化后，旧确认关系标记为过期。
8. 内存中转副表应作为独立运行时资源进入关系图，记录创建、读取、写入、版本和生命周期。

## 27. 后续建议继续讨论

1. 稳定行ID在导入、筛选、去重、合并和追加时如何继承。
2. 多行聚合为一行时，行血缘如何保存。
3. 表合并时同名字段和字段ID如何处理。
4. 节点主动上报ChangeSet的标准接口。
5. 表快照使用SQLite临时表、Arrow、Parquet还是其他结构。
6. 关系图第一版只做节点级，还是同时实现字段级。
7. 单元格追踪的默认保存范围。
8. 中转副表多节点同时写入时的版本和锁策略。
9. 预览运行和正式运行是否共享缓存。
10. 关系历史和工作流计划版本如何关联。

# 后续追加内容：工作流字段调试器与简化排错方案

> 追加日期：2026-06-25
> 状态：后续方案讨论内容
> 说明：本节是在“节点关系树、表差异与数据追踪设计”基础上追加，重点提出一种比完整关系树更简便、更贴近实际排错过程的方案。

## 28. 核心判断

工作流字段排错与代码调试非常相似。

代码调试关注：

```text
变量在哪里定义
变量在哪里被读取
变量在哪里被修改
变量什么时候第一次出现异常
最终错误值由哪一段逻辑产生
```

工作流调试关注：

```text
字段在哪里生成
字段被哪些节点读取
字段在哪里更新、改名或删除
字段在哪个节点第一次出现异常
最终错误结果由哪个节点产生
```

因此，第一阶段不必优先实现完整的数据关系树，可以先实现一个更轻量、更实用的：

```text
工作流字段调试器
```

推荐最终方向：

```text
全局字段查找
+
字段监视器
+
条件断点
+
单值追踪
```

这四项可以覆盖大部分“最终结果不对，但不知道哪一步出错”的问题。

## 29. 全局字段查找

工作流提供类似代码编辑器“查找引用”的字段搜索入口。

用户输入：

```text
板号
```

系统扫描整个工作流，显示：

```text
节点1：创建“板号”
节点2：读取“板号”
节点3：原样传递“板号”
节点5：更新“板号”
节点7：将“板号”改名为“PCB板号”
节点9：删除“PCB板号”
节点4：将“板号”写入中转副表A
```

字段行为建议统一分类：

```text
CREATE      创建
READ        读取
UPDATE      更新
RENAME      改名
COPY        复制
DERIVE      派生
MERGE       合并
SPLIT       拆分
DELETE      删除
PASS        原样传递
```

默认搜索结果优先显示真正改变字段的节点：

```text
CREATE
UPDATE
RENAME
DERIVE
DELETE
```

读取和原样传递节点通过开关显示，避免结果过多。

## 30. 静态查找与运行查找

### 30.1 静态查找

不执行工作流，根据节点配置和字段绑定关系进行查询。

可以回答：

```text
哪些节点预计读取该字段
哪些节点预计更新该字段
哪些节点预计创建该字段
哪些节点将该字段映射到其他表或中转副表
```

静态查找可以在打开计划后立即使用，但只能表示预计行为。

### 30.2 运行查找

执行预览或正式运行后，根据实际运行事件查询。

例如：

```text
节点1实际生成1000个板号
节点2实际读取1000个板号
节点5实际修改82个板号
节点7本次未执行
节点9本次删除0个字段
```

搜索结果应区分：

```text
预计使用
实际使用
```

运行查找比静态查找更适合最终排错。

## 31. 字段处理时间线

搜索字段后，优先使用时间线展示，而不是直接打开大型关系图。

示例：

```text
字段：板号

节点1  创建
输出行数：1000
非空：998
唯一值：820
样例：HYBP2529GK

节点5  更新
修改数量：82
修改前：HYBP2529GK
修改后：HYBP2529GK-26

节点7  改名
板号 → PCB板号
```

中间没有修改该字段的节点可以折叠。

字段时间线主要回答：

```text
字段第一次在哪里产生
字段最后在哪里修改
字段什么时候改名
字段什么时候被删除
字段从哪一步开始异常
```

后续完整关系树也可以使用同一批字段事件生成。

## 32. 创建节点与最后修改节点

对多数错误，最先需要知道的是：

```text
字段最初由谁创建
当前结果最后由谁修改
```

因此，每个字段运行结果应保存：

```text
created_by
last_modified_by
last_renamed_by
deleted_by
```

用户可以在结果表字段标题或单元格上执行：

```text
跳转到字段创建节点
跳转到最后修改节点
查看完整处理历史
加入字段监视
追踪当前值
```

“跳转到最后修改节点”应作为第一版高优先级功能。

## 33. 单值追踪

当只有某一行或某个单元格结果错误时，字段级搜索仍然不够。

用户选中：

```text
行：产品编码1.04.01.00001AJA
字段：板号
当前值：HYBP2529GK-26
```

系统按照稳定行ID和稳定字段ID向上追踪：

```text
节点1
处理前：空
处理后：HYBP2529GK
操作：正则提取

节点5
处理前：HYBP2529GK
处理后：HYBP2529GK-26
操作：追加版本后缀

节点8
值未变化
操作：原样传递
```

默认只显示真正改变过当前值的节点：

```text
节点1 → 节点5
```

需要查看完整经过路径时，再展开PASS节点。

这相当于代码调试中的变量值历史。

## 34. 字段监视器

字段监视器类似IDE中的Watch窗口。

用户可以把关键字段加入监视：

```text
板号
产品编码
程序编码
版本号
匹配结果
```

每个节点预览结束后，显示字段摘要：

| 节点 | 字段是否存在 | 非空数 | 空值数 | 唯一值数 | 修改数 |
|---|---|---:|---:|---:|---:|
| 节点1 | 是 | 998 | 2 | 820 | 998 |
| 节点2 | 是 | 998 | 2 | 820 | 0 |
| 节点3 | 是 | 860 | 0 | 720 | 0 |
| 节点5 | 是 | 860 | 0 | 720 | 82 |

通过监视摘要，可以快速发现：

```text
字段在哪个节点消失
非空数量在哪个节点下降
唯一值数量在哪个节点异常变化
修改数量在哪个节点突然增加
```

不必先查看完整输出表。

## 35. 字段摘要指标

字段监视不应只显示少量样例，建议生成轻量统计摘要：

```text
字段是否存在
数据类型
总行数
非空数量
空值数量
唯一值数量
重复数量
最短长度
最长长度
数字最小值
数字最大值
字段哈希
样例值
本节点修改数量
```

不同字段类型可使用不同摘要。

### 文本字段

```text
最短长度
最长长度
空值数量
唯一值数量
正则不匹配数量
```

### 数值字段

```text
最小值
最大值
平均值
空值数量
异常值数量
```

### 日期字段

```text
最早日期
最晚日期
非法日期数量
空值数量
```

## 36. 条件断点

用户可以对字段设置调试条件。

例如：

```text
板号出现空值时暂停
板号发生变化时暂停
板号字段被删除时暂停
板号唯一值数量下降时暂停
板号不符合指定正则时暂停
板号重复数量超过10时暂停
程序编码长度超过指定范围时暂停
```

预览运行时，系统在第一次满足条件的节点暂停：

```text
已在节点5暂停

原因：
字段“板号”出现3个不符合规则的值

异常样例：
HYBP-26
2529GK
空值
```

条件断点可以帮助用户定位“第一次出现问题”的节点，而不是从最终结果反向逐个排查。

## 37. 数据断言

除了临时断点，还可以为关键节点或工作流设置固定数据检查规则。

例如：

```text
节点3执行后，“板号”字段必须存在
节点3执行后，“板号”不允许为空
“产品编码”必须唯一
输出行数不得少于900
“板号”必须符合指定正则
“匹配结果”成功率不得低于95%
```

断言失败时：

```text
节点3数据校验失败

规则：
板号空值数量必须等于0

实际：
空值数量为12
```

断言适合作为调试和质量检查能力，不一定需要设计成大量独立节点。

## 38. 数据二分定位

长工作流可以采用类似代码中的二分排错。

假设工作流有20个节点，最终字段结果错误：

```text
先检查节点10
节点10正常 → 错误在11～20
检查节点15
节点15异常 → 错误在11～15
继续缩小范围
```

系统可以提供：

```text
查找字段第一次异常的位置
```

用户指定异常条件，例如：

```text
首次出现空值
首次不符合正则
首次唯一值数量下降
首次行数异常减少
首次值与目标结果不同
```

系统结合节点缓存和预览能力，逐步缩小错误范围。

这一能力类似`git bisect`，适合较长工作流。

## 39. 中转副表字段查找

全局字段搜索必须覆盖主表和中转副表。

搜索结果需要区分资源：

```text
主表.板号
中转副表A.板号
标准文件表.板号
```

字段查找结果可以显示：

```text
节点3将主表.板号写入中转副表A.板号
节点5读取中转副表A.板号
节点7覆盖中转副表A.板号
```

内部索引应采用：

```text
table_id + field_id
```

而不是只使用字段显示名称。

## 40. 默认只保存变化数据

第一版不建议保存每个节点的完整单元格历史。

默认只记录：

```text
发生变化的字段
发生变化的行
修改前值
修改后值
执行节点
字段摘要
表结构变化
```

原样传递不逐单元格保存，只记录继承关系：

```text
节点B的板号从节点A原样继承
```

用户追踪具体值时，再沿继承链向上查找。

这样能显著降低几十万行数据下的存储和内存成本。

## 41. 节点最小运行事件

节点运行后建议返回轻量字段影响记录：

```python
FieldEffect(
    table_id="main_table",
    field_id="field_board_no",
    action="UPDATE",
    changed_rows=82,
    sample_before=["HYBP2529GK"],
    sample_after=["HYBP2529GK-26"]
)
```

表结构节点返回：

```python
TableEffect(
    added_rows=0,
    removed_rows=120,
    added_fields=["field_board_no"],
    removed_fields=[]
)
```

字段调试器可以根据这些事件生成：

```text
字段搜索结果
字段时间线
节点跳转
字段监视摘要
条件断点
后续关系图
```

因此，关系图不必作为第一阶段最复杂的底层功能，它可以是运行事件的一种可视化表现。

## 42. 第一版建议范围

第一版字段调试器建议只实现：

1. 全局字段搜索。
2. 字段创建节点定位。
3. 字段最后修改节点定位。
4. 字段处理时间线。
5. 字段监视摘要。
6. 选择一行和字段进行单值追踪。
7. 中转副表字段搜索。
8. 简单条件断点。
9. 默认保存稀疏变化记录。

第一版暂不实现：

```text
所有单元格的完整历史
自动生成全部字段关系图
复杂多行聚合血缘
跨多次运行的完整数据回放
大规模全量调用栈
```

## 43. 推荐最终方向

比完整数据血缘系统更简便、更实用的方案是：

```text
全局字段查找
+
字段监视器
+
条件断点
+
单值追踪
```

典型使用流程：

```text
最终结果发现“板号”错误
→ 搜索“板号”
→ 跳转到最后修改节点
→ 查看处理前后值
→ 选择错误行进行单值追踪
→ 找到第一次错误变化
```

底层优先保留：

```text
稳定字段ID
稳定行ID
字段操作事件
变化单元格记录
字段统计摘要
```

在这些基础数据具备后，后续既可以继续完善字段调试器，也可以生成节点关系树、字段血缘图和单元格处理链。

关系树应当成为调试数据的图形化展示结果，而不是第一阶段必须一次完成的复杂核心。

# 后续追加内容：组节点与节点/插件进程生命周期现状梳理

> 追加日期：2026-06-26
> 状态：后续方案讨论内容
> 说明：本节对当前文档中已经形成的组节点、普通节点执行、插件子进程、运行状态、取消机制和临时资源生命周期进行统一整理。当前内容主要属于基础框架定义，尚未细化为最终运行协议。

## 44. 当前组节点的基础逻辑

目前组节点相关内容主要通过节点协议、资源作用域、运行记录和字段调试体系间接体现，尚未形成完整独立规范。

### 44.1 组节点是内部工作流片段的封装

当前默认思路：

```text
组节点
├─ 内部节点A
├─ 内部节点B
├─ 内部节点C
└─ 内部连接关系
```

组内节点仍然遵循普通节点协议：

```python
def validate(inputs, config, context):
    ...

def resolve_permissions(inputs, config):
    ...

def execute(inputs, config, context):
    ...
```

因此，组节点目前不是与普通节点完全不同的运行对象，而是对一段内部工作流结构的封装。

### 44.2 组节点可拥有组级资源作用域

当前资源生命周期定义中已经包含：

```text
NODE_SCOPE
GROUP_SCOPE
WORKFLOW_SCOPE
SESSION_SCOPE
PERSISTENT
```

其中：

```text
GROUP_SCOPE
```

表示资源在组节点开始运行时创建，在整个组运行完成、失败或取消后释放。

适合组内共享的资源包括：

```text
组内临时表
组内缓存
组内临时文件
组内共享变量
组内插件结果
组内预览快照
组内日志缓存
```

但目前尚未定义具体由哪个组件创建、登记、继承和释放这些资源。

### 44.3 组内节点仍需保留独立运行记录

组节点在外部可以折叠显示，但内部节点仍应保留：

```text
独立节点运行ID
运行状态
输入输出
进度
心跳
日志
字段变化
表变化
错误信息
执行耗时
权限使用
```

例如：

```text
组节点：编码提取组
状态：失败
进度：2/4

内部：
✓ 节点A 读取字段
✓ 节点B 正则提取
✗ 节点C 匹配失败
○ 节点D 未执行
```

这样既可以在普通使用时保持界面简洁，也可以在排错时展开组内执行过程。

### 44.4 当前尚未定义的组节点接口

后续仍需明确：

```text
组节点如何声明对外输入
组节点如何声明对外输出
内部字段如何映射到外部字段
哪些内部设置允许公开
组内字段名称冲突如何处理
组内节点失败后是否继续
组节点整体进度如何汇总
组内插件是否共用进程
组节点是否允许嵌套
组取消时如何停止内部节点
组内临时资源如何释放
组内部分结果是否允许保留
```

后续可以考虑把组节点正式定义为一级对象：

```text
GroupDefinition
GroupInstance
GroupRun
GroupContext
```

## 45. 普通节点当前执行方式

普通节点当前建议默认采用：

```python
"isolation": "in_process"
```

含义是：

```text
节点运行在主程序操作系统进程内
```

但不代表一定运行在UI主线程。

具体执行方式可以是：

```text
工作线程
线程池
异步任务
主进程中的统一执行队列
```

适合进程内执行的节点：

```text
字段处理
数值计算
筛选
去重
普通SQLite操作
可控的Python数据转换
低风险纯逻辑节点
```

主要优点：

```text
调用成本低
数据共享方便
无需频繁序列化大表
适合大量短任务
```

主要限制：

```text
节点崩溃可能影响主进程
线程卡死后难以安全强制终止
第三方库异常可能污染主程序状态
```

## 46. 当前节点运行状态机

当前文档已经提出以下节点状态：

```text
PENDING
VALIDATING
WAITING_PERMISSION
RUNNING
CANCEL_REQUESTED
SUCCEEDED
FAILED
TIMED_OUT
SUSPECTED_HUNG
CANCELLED
```

基本流程：

```text
PENDING
→ VALIDATING
→ WAITING_PERMISSION
→ RUNNING
→ SUCCEEDED / FAILED / TIMED_OUT / CANCELLED
```

异常观察状态：

```text
SUSPECTED_HUNG
```

表示节点可能卡住，但尚未完全确认。

取消过程：

```text
RUNNING
→ CANCEL_REQUESTED
→ CANCELLED
```

或取消失败后进入：

```text
TIMED_OUT
FAILED
SUSPECTED_HUNG
```

最终状态应由 `ExecutionController` 统一确认，不应由节点自行直接修改全局运行状态。

## 47. 当前 RuntimeContext 职责

> 本节已按“节点负责具体资源操作，平台负责权限和标准数据交接”的正式决策更新。

每次节点运行时，节点接收统一的 `RuntimeContext`。

当前建议能力包括：

```text
context.inputs
context.permissions
context.resource_profiles
context.register_output
context.report_progress
context.heartbeat
context.log
context.check_cancelled
context.run_id
context.node_run_id
context.temp
context.events
```

### 47.1 输入与数据引用

`context.inputs` 提供已经解析的上游输入、普通值和 `DataRef`。节点根据引用类型打开或读取数据，不要求平台把全部输入转换为同一种数据结构。

### 47.2 权限信息

`context.permissions` 提供本次运行经过中央权限模块批准的权限句柄。权限层决定能否执行，但不代替节点访问具体数据源。

### 47.3 资源配置

`context.resource_profiles` 提供当前节点获准读取的资源配置视图，例如数据库连接配置、目录配置或Web系统配置。节点或插件使用这些配置自行执行实际访问。

### 47.4 输出登记

`context.register_output` 用于登记节点返回的 `DataOutput / DataRef`，包括Schema、版本、承载方式、生命周期和清理责任方。

### 47.5 进度与心跳

`context.report_progress` 和 `context.heartbeat` 用于主程序观察节点是否正常运行。

### 47.6 日志

`context.log` 将节点运行日志统一写入执行记录。

### 47.7 取消检查

`context.check_cancelled` 由节点在循环、分块和关键步骤之间主动调用。

### 47.8 运行标识

`context.run_id` 和 `context.node_run_id` 分别标识当前工作流运行实例和当前节点运行实例。

### 47.9 临时资源

`context.temp` 负责登记临时表、临时文件、缓存和快照，但不强制指定实际存储技术。

### 47.10 运行事件

`context.events` 用于上报节点状态、数据发布、警告、错误和外部事件。

节点不再依赖一个接管所有数据库、文件、Web和Office操作的通用 `context.data`。数据库事务、Office会话、网页会话等具体资源生命周期由相应节点或插件管理，并向平台上报结果和清理状态。

## 48. 当前节点卡住检测

主程序不能直接向节点询问：

```text
你是否卡住
```

因为真正卡住的节点可能已经无法响应。

因此，由 `ExecutionController` 在外部观察：

```text
节点开始时间
最近心跳时间
最近进度更新时间
当前进度值
当前执行步骤
最近日志
最大允许运行时间
```

建议判断方式：

| 心跳 | 进度 | 判断 |
|---|---|---|
| 正常 | 持续变化 | 正常运行 |
| 正常 | 长时间不变 | 可能是耗时操作 |
| 超时 | 不变化 | 疑似卡住 |
| 超过运行上限 | 任意 | 运行超时 |

核心原则：

```text
进度不变化不一定卡死
心跳长时间停止是更强的异常信号
```

Office COM、文件解析、大型事务和第三方库调用，可能长时间停留在一个进度阶段，因此需要节点上报当前阶段信息。

## 49. 当前普通节点取消机制

进程内节点采用协作式取消：

```python
context.check_cancelled()
```

节点应在适当位置主动检查：

```python
for chunk in chunks:
    context.check_cancelled()
    process(chunk)
```

适合检查取消的位置：

```text
每个数据分块开始前
批量数据库提交前
长循环内部
外部步骤切换之间
写入文件前
生成最终结果前
```

协作式取消的限制：

```text
第三方库阻塞时无法检查
COM调用阻塞时无法检查
C扩展卡死时无法检查
无限循环且未调用检查时无法取消
普通线程无法安全强制杀死
```

因此：

```text
线程内节点只能请求取消，不能保证安全强杀
```

## 50. 当前插件子进程隔离逻辑

高风险插件建议采用：

```python
"isolation": "subprocess"
```

适用场景：

```text
Office COM
外部Python脚本
第三方插件
网络调用
AI模型调用
不稳定二进制库
可能死循环的逻辑
容易导致主进程崩溃的代码
```

子进程隔离的主要价值：

```text
插件崩溃不直接拖垮主程序
插件卡死后可以强制终止
插件可以使用独立环境
插件资源可随进程退出清理
插件依赖冲突可以隔离
```

当前基础执行流程：

```text
主程序创建插件子进程
→ 传入标准输入和配置
→ 插件初始化运行环境
→ 插件上报心跳、进度和日志
→ 插件执行
→ 返回标准NodeResult
→ 主程序回收子进程
```

异常处理流程：

```text
检测心跳超时
→ 标记SUSPECTED_HUNG
→ 发送正常取消请求
→ 等待插件自行退出
→ 超过宽限时间后强制终止
→ 清理临时资源
→ 回滚事务
→ 标记FAILED / CANCELLED / TIMED_OUT
```

目前尚未确定：

```text
正常取消宽限时间
强制终止条件
是否自动重试
是否自动重启插件进程
是否复用已有进程
```

## 51. 普通节点与插件节点的统一协议

无论节点运行在主进程还是子进程，对工作流调度器都应保持统一接口。

统一内容：

```text
标准输入
标准配置
权限声明
运行状态
进度
心跳
日志
标准结果
运行事件
错误信息
取消请求
```

理想调用形式：

```python
result = runner.execute(
    node_definition,
    inputs,
    config,
    runtime_context
)
```

执行器根据隔离策略选择：

```text
InProcessRunner
SubprocessRunner
```

工作流调度层不直接关心节点内部是普通函数、线程任务还是插件进程。

## 52. 当前临时资源生命周期

当前定义了五类资源作用域。

| 作用域 | 释放时间 |
|---|---|
| `NODE_SCOPE` | 当前节点完成、失败或取消后 |
| `GROUP_SCOPE` | 当前组节点完成、失败或取消后 |
| `WORKFLOW_SCOPE` | 当前工作流运行结束后 |
| `SESSION_SCOPE` | 软件会话结束后 |
| `PERSISTENT` | 用户明确删除、覆盖或清理时 |

适用资源：

```text
内存中转副表
SQLite临时表
临时文件
插件输出缓存
预览结果
表快照
运行日志缓存
插件工作目录
中间序列化数据
```

此前文档建议第一版优先实现：

```text
WORKFLOW_SCOPE
PERSISTENT
```

但进入组节点细化后，建议将：

```text
GROUP_SCOPE
```

也纳入第一版基础设计。

## 53. 组节点当前尚未解决的生命周期问题

### 53.1 组运行上下文

### 53.2 组内执行顺序

需要确定：


### 53.3 组失败策略

需要确定：

```text
任一节点失败是否立即终止组
```

```text

```

## 54. 普通节点当前尚未解决的生命周期问题

需要继续细化：



## 56. 当前运行结构概括

```text
WorkflowRun
│
├─ NodeRun
│  ├─ InProcessRunner
│  └─ RuntimeContext
│
├─ PluginNodeRun
│  ├─ SubprocessRunner
│  ├─ PluginProcess
│  └─ RuntimeContext代理
│
└─ GroupRun
   ├─ GroupContext
   ├─ Internal NodeRun
   ├─ Internal PluginNodeRun
   └─ GROUP_SCOPE Resources
```

其中：

- `WorkflowRun`：负责整个工作流运行生命周期。
- `NodeRun`：负责一个普通节点的一次运行实例。
- `PluginNodeRun`：负责一个插件节点的一次运行实例，以及对应子进程关系。
- `GroupRun`：负责组节点整体状态、内部节点调度和组级资源。
