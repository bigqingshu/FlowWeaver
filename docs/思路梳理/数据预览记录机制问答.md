# 数据预览记录机制问答

更新时间：2026-07-08

## 这份文档用来说明什么？

回答：

这份文档用于快速梳理 FlowWeaver 当前“数据预览页面”的真实机制。

重点说明：

- 数据预览的数据从哪里来。
- 后端记录什么。
- 前端记录什么。
- `TableRef` 是不是唯一的。
- UI 是否自己保存预览数据。
- `preview_to_node` 和普通运行有什么关系。

## 数据预览页面显示的数据，是后端返回的吗？

回答：

是的。

数据预览页面最终显示的 rows 来自后端接口：

```text
GET /api/v1/data/{table_ref_id}/rows
```

前端不会凭空生成这些表格数据。

前端只是根据当前选择的 `TableRef`，向后端请求对应 rows，然后把返回结果显示到表格里。

## 后端是不是只运行节点，不负责保存数据预览？

回答：

不是。

后端不保存一个单独叫“数据预览记录”的对象，但后端会保存数据预览所依赖的运行产物。

这些产物包括：

```text
WorkflowRun
NodeRun
NodeTaskResult
TableRef
RuntimeEvent
实际表数据
```

数据预览页面看到的内容，本质上就是从这些后端运行产物里读取出来的。

## 后端记录的是什么？

回答：

后端主要记录运行态和表引用。

典型结构是：

```text
WorkflowRun
  NodeRun
    NodeTaskResult
    TableRef
      provider rows
```

其中：

- `WorkflowRun` 记录一次工作流运行。
- `NodeRun` 记录某个节点在这次运行中的执行状态。
- `NodeTaskResult` 记录节点任务执行结果。
- `TableRef` 记录节点输出表的引用信息。
- 实际 rows 保存在对应 provider 中，例如 runtime sqlite 表。

## TableRef 是不是唯一一个？

回答：

不是。

一个 workflow run 里可以有很多个 `TableRef`。

一个节点运行也可能产生 0 个、1 个或多个 `TableRef`。

例如：

```text
WorkflowRun: run-001

NodeRun: source
  -> TableRef: source_output

NodeRun: filter
  -> TableRef: filter_output
  -> TableRef: filter_status

NodeRun: loop_judge
  -> TableRef: loop_control_status
```

所以不能理解成：

```text
一个工作流运行只有唯一一个 TableRef
```

更准确的理解是：

```text
每个节点运行可以产生自己的输出 TableRef。
```

## UI 为什么一次只看到一张表？

回答：

因为 UI 当前表格区域一次只加载一个 `LoadedDataPreviewTableRef`。

后端可以有很多个 `TableRef`，但 UI 当前工作台只会选择其中一个来显示。

可以理解为：

```text
后端记录很多张表
UI 当前只打开其中一张表
```

## UI 是怎么拿到可预览表列表的？

回答：

UI 先根据当前选中的 workflow run 请求表引用列表：

```text
GET /api/v1/runs/{workflow_run_id}/table-refs
```

后端返回这个 run 下所有 `TableRef`。

然后 UI 把这些 `TableRef` 按：

```text
workflow_run_id + node_run_id
```

分组成数据预览状态。

这就是数据预览左侧列表和右侧表选择的来源。

## UI 是怎么加载表格 rows 的？

回答：

当用户选中某个 `TableRef` 并点击加载时，UI 调用：

```text
GET /api/v1/data/{table_ref_id}/rows
```

后端会：

1. 根据 `table_ref_id` 查 `TableRef`。
2. 检查这个 `TableRef` 是否可读。
3. 找到对应 provider。
4. 按 offset / limit 分页读取 rows。
5. 返回 columns、rows、row_count、has_more。

UI 再把返回的数据渲染成表格。

## UI 会不会把后端返回的数据长期保存起来？

回答：

当前不会。

UI 只保存当前页面需要显示的临时状态，例如：

```text
当前选中的 run
当前选中的 node
当前选中的 TableRef
当前已经加载的 TableRef
当前页 rows
当前 offset / limit
搜索文本
本地编辑草稿
```

这些是前端界面状态，不是后端运行记录。

如果 UI 关闭再打开，只要后端运行记录和表数据还存在，UI 可以重新请求后端接口恢复显示。

## 数据预览是否靠 UI 自己记住每次节点返回的数据？

回答：

不是。

正确机制是：

```text
节点运行结果由后端记录
表数据由后端 provider 保存
UI 只按需读取后端记录并显示
```

错误理解是：

```text
节点每次运行后把数据返回给 UI
UI 自己保存这些数据
之后用户查看的是 UI 保存的数据
```

当前不是这种机制。

## 后端负责报错吗？

回答：

负责。

后端会负责节点运行错误、工作流运行错误、表读取错误等。

例如：

```text
NodeRun FAILED
WorkflowRun FAILED
NodeTaskResult FAILED
TableRef not found
TableRef not readable
provider unsupported
data read rejected
```

UI 只是把后端返回的错误信息展示出来。

UI 自己也会处理一些界面级错误，例如：

- 没有选择 run。
- 没有选择 TableRef。
- 请求已经过期。
- 当前加载被新的选择覆盖。

但真正的运行错误和数据读取错误来源在后端。

## 预览选中节点是不是 UI 自己模拟执行？

回答：

不是。

预览选中节点会创建一次后端 workflow run。

这个 run 的模式是：

```text
run_mode = preview_to_node
target_node_instance_id = 目标节点
```

后端只执行目标节点所需的上游闭包。

这次 preview run 仍然会产生正常的：

```text
WorkflowRun
NodeRun
TableRef
RuntimeEvent
```

UI 再读取这次 run 的 `TableRef` 和 rows 显示数据预览。

## 数据预览机制是否只存在于工作流界面操作？

回答：

从当前 UI 使用方式看，数据预览主要由工作流界面操作驱动。

典型操作是：

```text
选择 workflow run
-> 刷新 TableRefs
-> 选择节点或表
-> 加载 rows
```

或者：

```text
点击预览选中节点
-> 后端创建 preview_to_node run
-> run 完成后刷新 TableRefs
-> UI 加载目标节点输出 rows
```

但后端能力本身不是只能给工作流界面使用。

只要某个 workflow run 产生了 `TableRef`，就可以通过同一套 API 读取。

## 后台工作流完善后，数据预览机制要不要重做？

回答：

一般不需要重做。

后台工作流完善后，仍然建议复用当前这套后端运行产物：

```text
WorkflowRun
NodeRun
TableRef
RuntimeEvent
provider rows
```

后台 run 只要正常产生 `TableRef`，UI 就可以通过：

```text
GET /api/v1/runs/{workflow_run_id}/table-refs
GET /api/v1/data/{table_ref_id}/rows
```

读取并显示。

后续可能需要补的是入口和筛选能力，例如：

- 后台运行列表。
- 最近输出表。
- 按 workflow / run / node 查询输出表。
- 按 run_mode 区分 full run 和 preview_to_node run。
- 后台运行完成后通知 UI 刷新。

但不建议另起一套“数据预览记录表”。

## 当前一句话总结

回答：

当前数据预览机制是：

```text
后端负责运行、记录状态、保存 TableRef 和真实表数据；
前端负责选择 run / node / TableRef，并按需读取 rows 展示。
```

它不是：

```text
UI 自己保存节点返回的数据作为长期预览记录。
```

真正可追溯的数据预览来源是：

```text
WorkflowRun + NodeRun + TableRef + provider rows
```
