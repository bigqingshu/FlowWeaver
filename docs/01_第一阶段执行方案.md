# 模块化数据工作流：第一阶段执行方案

> 2026-07-05 状态更新：默认主程序权限/审计模块已移除并后置。本文中 PermissionManager、权限句柄、AuditEvent、权限检查和审计表相关内容仅保留为早期执行记录，不再作为当前实现目标。当前默认主程序不再负责权限申请、授权、撤销或安全审计事件。

> 文档状态：第一阶段实施稿
> 创建日期：2026-06-27
> 目标：完成模块化数据工作流平台的最小可运行核心，验证工作流进程隔离、节点执行隔离、数据引用传递、共享表发布和故障回收。
> 范围说明：本阶段只实现工作流核心，不统一实现文件、Web、Office、浏览器、沙盒及所有外部数据库能力。

---

# 一、第一阶段目标

第一阶段不是完成全部功能，而是证明以下主链路可以稳定运行：

```text
Management UI
→ EngineHost
→ WorkflowRunProcess
→ NodeExecutorProcess
→ NodeResult
→ RuntimeDataRegistry
→ 下游节点
→ SharedPublication
→ 其他工作流读取
```

第一阶段必须验证：

1. UI可以创建、保存和运行一个工作流；
2. EngineHost可以创建独立的WorkflowRunProcess；
3. WorkflowRunProcess只负责DAG调度和状态控制；
4. 普通业务节点在NodeExecutorProcess中执行；
5. 节点之间通过TableRef或DataRef传递数据；
6. 节点输出先进入STAGING，成功后原子发布；
7. 单个节点卡死时可以终止对应执行器；
8. 同一工作流中的无关分支可以继续运行；
9. 工作流可以发布一个或多个共享表版本；
10. 另一个工作流可以读取固定版本；
11. 工作流结束后可以回收临时数据、租约和子进程。

# 四、核心职责边界

## 4.1 EngineHost

EngineHost是后台核心服务，但不执行普通业务节点代码。

负责：

```text
启动和关闭平台
接收Management UI请求
创建和回收WorkflowRunProcess
维护平台级状态
管理Supervisor
管理RuntimeDataRegistry
管理EventRouter
管理RuntimeStore
汇总日志、进度和错误
```

不负责：

```text
直接执行筛选、替换、正则等节点
直接打开Office或浏览器
直接连接所有数据库
直接保存所有业务大表
直接处理复杂插件逻辑
```

核心原则：

> EngineHost只做控制、登记和监督，尽量避免加载高风险依赖。

---

## 4.2 WorkflowScheduler

负责平台级工作流调度。

第一阶段支持：

```text
手动启动
工作流运行状态检查
限制同时运行数量
等待依赖资源
工作流结束回收
```

后续可扩展：

```text
定时触发
事件触发
优先级
队列调度
资源配额
```

---

## 4.3 WorkflowRunProcess

每次工作流运行对应一个独立进程。

负责：

```text
加载本次工作流定义和配置快照
构建DAG依赖图
维护节点状态
寻找可执行节点
向NodeExecutor提交任务
接收NodeResult
执行失败策略
处理取消
维护InputSnapshot
请求共享表发布
向EngineHost上报心跳
```

不负责：

```text
长时间直接运行普通业务节点
直接处理第三方库
直接持有所有外部资源连接
```

正式定位：

> WorkflowRunProcess是工作流控制进程。

---

## 4.4 NodeExecutorPool

普通业务节点默认交给NodeExecutorProcess执行。

适合：

```text
筛选
替换
正则
去重
字段计算
表合并
列处理
普通数据库查询
普通文件解析
```

第一阶段可以先实现：

```text
每个WorkflowRunProcess启动1～2个NodeExecutorProcess
```

后续再扩展为动态进程池。

NodeExecutor必须支持：

```text
接收节点任务
加载节点实现
读取输入DataRef
执行节点
上报任务心跳
上报进度
返回NodeResult
响应CancelToken
异常退出后可重建
```

---

## 4.5 DedicatedWorker

用于具有特殊生命周期或高阻塞风险的节点。

第一阶段只需预留接口，可选实现一个示例Worker。

适用场景：

```text
Office COM
浏览器
模型
屏幕识别
串口
长连接
数据库固定快照
外部程序
```

是否使用沙盒、低权限或容器，由Worker自身实现，不进入工作流核心协议。

---

## 4.6 PermissionManager

第一阶段只管理工作流内部数据权限。

支持：

```text
表读取
字段读取
表写入
字段写入
追加
覆盖
清空
结构修改
输出发布
共享表读取
数据库逻辑映射使用
```

原则：

```text
权限检查始终开启
审计等级可以切换
```

审计等级：

```text
MINIMAL
STANDARD
FULL
DEBUG
```

第一阶段默认：

```text
STANDARD
```

---

## 4.7 RuntimeDataRegistry

RuntimeDataRegistry是数据引用登记中心，不是统一数据仓库。

记录：

```text
data_id
TableRef / DataRef
数据类型
Schema
存储方式
版本
产生节点
所属工作流
生命周期
当前使用者
读取租约
血缘关系
状态
```

它不强制数据存储在同一个位置。

---

## 4.8 RuntimeStore

保存控制元数据：

```text
工作流定义
节点配置
运行记录
节点状态
错误记录
权限配置
共享发布记录
数据引用元数据
```

第一阶段可使用SQLite。

不保存：

```text
所有大型业务表
所有图片
所有实时数据
所有外部文件内容
```

---

## 4.9 EventRouter

负责轻量事件分发。

事件包括：

```text
WORKFLOW_STARTED
WORKFLOW_FINISHED
WORKFLOW_FAILED
NODE_STARTED
NODE_PROGRESS
NODE_FINISHED
NODE_FAILED
NODE_TIMEOUT
EXECUTOR_CRASHED
DATA_PUBLISHED
SHARED_PUBLICATION_AVAILABLE
RESOURCE_RELEASED
```

事件只传元数据和引用，不传整张大表。

---

## 4.10 ResourceProfileRegistry

负责内部名称与真实配置映射。

例如：

```text
db_profile_001
→ D:\db\SQL\work.db
```

工作流定义只保存：

```text
db_profile_001
```

第一阶段只需要支持：

```text
运行数据库配置
SQLite路径映射
内部显示名称
```

# 七、工作流内部表模型

## 7.1 当前表

定义：

> 当前分支默认参与计算的主TableRef。

当前表是一种逻辑角色，不限制底层存储。

例如：

```text
role = CURRENT
storage_kind = RUNTIME_SQL
```

## 7.2 辅助表

定义：

> 工作流处理过程中产生的非主表数据。

典型用途：

```text
匹配表
异常表
去重结果
统计表
映射表
第二输出表
```

## 7.3 运行SQL表

定义：

> 位于运行中转数据库中的可变工作表。

特点：

```text
可修改
可删除
可重建
适合大表处理
不需要再次物化
```

## 7.4 共享表版本

定义：

> 从某个工作流阶段显式发布的不可变表版本。

特点：

```text
发布后不可原地修改
其他工作流通过固定版本读取
可建立读取租约
可按保留策略释放
```

# 八、运行数据库设计

第一阶段采用运行数据库作为工作流标准中转存储。

建议每个WorkflowRun拥有独立运行数据库：

```text
runtime/{workflow_run_id}.db
```

或者：

```text
runtime.db
└─ namespace = workflow_run_id
```

第一阶段优先推荐独立数据库文件，故障隔离和清理更简单。

运行数据库用于：

```text
当前表
辅助表
节点临时表
STAGING输出
共享发布版本
```

运行数据库默认属于：

```text
ownership_mode = MANAGED_RUNTIME
```

当前工作流默认拥有：

```text
读取
写入
建表
删表
改结构
发布
```

# 九、节点执行主流程

```text
1. WorkflowRunProcess检查DAG依赖
2. 找到READY节点
3. 创建NodeTask
4. 解析输入TableRef
5. 申请内部数据权限
6. 分配NodeExecutor
7. 执行器加载节点实现
8. 执行节点
9. 产生STAGING输出
10. 校验NodeResult
11. 校验Schema
12. 原子发布输出
13. RuntimeDataRegistry登记
14. 更新下游依赖
15. 释放节点级临时资源
```

# 十一、节点心跳与进度

每个执行中的节点任务必须上报：

```text
executor_heartbeat
task_heartbeat
progress
current_stage
last_progress_at
cpu_time
memory_usage
```

判断规则：

## 正常运行

```text
心跳正常
进度变化
```

## 长时间运行

```text
心跳正常
进度长时间不变
```

状态：

```text
LONG_RUNNING
```

## 疑似卡死

```text
任务心跳停止
执行器仍存活
```

状态：

```text
SUSPECTED_HUNG
```

## 执行器崩溃

```text
进程退出
```

状态：

```text
CRASHED
```

# 十二、节点取消和强制终止

```text
节点超时或用户取消
→ 停止继续调度该节点
→ 发送CancelToken
→ 等待宽限时间
→ 仍未退出
→ 终止NodeExecutorProcess
→ 废弃STAGING输出
→ 回收临时资源
→ 创建新NodeExecutorProcess
```

如果WorkflowRunProcess失去响应：

```text
EngineHost
→ 请求优雅退出
→ 等待宽限时间
→ 终止WorkflowRunProcess
→ 清理WORKFLOW_SCOPE资源
```

故障域：

```text
EngineHost
└─ WorkflowRunProcess
   └─ NodeExecutorProcess
```

# 十三、DAG独立分支执行

示例：

```text
       ┌→ B → D
A →────┤
       └→ C → E
```

如果B延迟：

```text
B：RUNNING
D：WAITING_DEPENDENCY
C：SUCCEEDED
E：RUNNING
```

规则：

> 只阻塞真正依赖慢节点的分支，其他独立分支继续执行。

第一阶段必须避免简单的“节点列表串行执行”。

---

# 十四、节点输出原子发布

节点执行中的输出状态：

```text
STAGING
```

节点成功并完成校验后：

```text
STAGING
→ PUBLISHED
```

节点失败或被终止时：

```text
删除STAGING结果
不更新TableRef
不触发共享发布
```

数据库节点使用事务：

```text
BEGIN
→ 写入STAGING表
→ 校验
→ COMMIT
```

失败：

```text
ROLLBACK
```

---

# 十五、节点失败策略

第一阶段支持：

```text
FAIL_FAST
CONTINUE_INDEPENDENT
SKIP_DEPENDENTS
RETRY
```

预留：

```text
USE_LAST_SUCCESS
```

## FAIL_FAST

任意节点失败，停止整个工作流。

## CONTINUE_INDEPENDENT

失败分支停止，无关分支继续。

## SKIP_DEPENDENTS

跳过所有依赖失败节点的下游节点。

## RETRY

需要配置：

```text
最大次数
重试间隔
是否更换执行器
```

具有外部副作用的节点默认不自动重试。

# 十九、InputSnapshot

InputSnapshot固定一次工作流实际读取的数据版本。

```text
InputSnapshot
├─ input_snapshot_id
├─ workflow_run_id
├─ created_at
└─ inputs
   ├─ source_name
   ├─ publication_id
   ├─ publication_version
   └─ selected_members
```

目的：

```text
避免运行过程中读取到不同版本
保证结果可复现
为后续血缘追踪提供基础
```

# 三十一、第一阶段核心原则

```text
EngineHost管平台
WorkflowRunProcess管工作流
NodeExecutor管节点
RuntimeDataRegistry管数据关系
运行数据库管中转表
SharedPublication管跨工作流发布
PermissionManager管内部数据权限
Supervisor管进程与故障回收
```

最终目标：

> 先构建一个轻量、稳定、可扩展的数据工作流核心，再逐步增加数据库类型、外部节点、Office、Web、模型和其他执行能力。
