# 模块化数据工作流：数据库、共享表与节点故障隔离讨论方案

> 文档状态：独立讨论稿
> 创建日期：2026-06-27
> 用途：逐项讨论数据库核心节点、多工作流表通信、共享版本一致性、生命周期管理，以及单节点延迟与卡死隔离。
> 说明：本方案暂不并入主架构MD，后续按章节逐项确认后再决定是否合并。

---

# 一、数据库节点作为核心节点是否可行

完全可行，而且合理。

这里的“核心数据库节点”并不是让 `EngineHost` 负责所有数据库，而是：

```text
核心工作流节点包
└─ 数据库节点
   ├─ 读取
   ├─ 查询
   ├─ 写入
   ├─ 事务
   └─ 外部引用 / 物化输出
```

数据库相关工作仍然由节点完成：

```text
解析数据库配置
建立连接
执行SQL
读取Schema
生成表引用
物化查询结果
执行写入
管理事务
```

主程序只管理：

```text
节点配置
节点执行
内部数据权限
TableRef登记
运行状态
版本和生命周期
```

所以它仍符合当前原则：

> 节点负责能力实现，平台负责调度和数据关系管理。

# 二、真实地址与内部名称映射

当前主方案已经有：

```text
ResourceProfileRegistry
```

它适合承担这项工作，但要明确它只是一个配置映射表。

例如：

```text
内部名称：生产物料数据库
profile_id：db_profile_001
类型：SQLite
真实地址：D:\db\SQL\work.db
```

工作流计划只保存：

```text
db_profile_001
```

而不是保存：

```text
D:\db\SQL\work.db
```

数据库节点配置可以是：

```text
数据库配置：生产物料数据库
逻辑表名：物料汇总
操作：查询
输出模式：外部引用
```

运行时流程：

```text
WorkflowRunProcess
→ 调用数据库节点
→ 数据库节点根据profile_id解析实际连接
→ 执行数据库操作
→ 返回TableRef
```

其他普通节点看到的是：

```text
生产物料数据库
物料汇总
TableRef
```

不需要看到真实路径。

建议边界：

```text
工作流JSON：
只保存profile_id和逻辑表名

ResourceProfileRegistry：
保存逻辑名称与真实连接配置

数据库节点：
运行时读取真实配置

其他节点：
只看到TableRef
```

这与当前主方案兼容，不需要增加通用 `ResourceBroker`。

# 三、是否真的能做到“只有数据库节点看见真实地址”

可以，但需要规定：

> `RuntimeContext` 只向当前数据库节点注入已经解析的数据库配置，不能把全部资源配置直接暴露给所有节点。

例如：

```python
context.resource_profiles.resolve_for_node(
    node_run_id,
    profile_id
)
```

只有获得许可的数据库节点可以解析：

```text
profile_id → 真实连接参数
```

其他节点即使拿到 `profile_id`，也不能通过普通接口解析真实地址。

这仍然属于应用层边界，不是操作系统沙盒，但对当前本地、自用和核心阶段项目已经足够。

# 四、当前权限管理是否支持

当前主方案的权限重点是：

```text
节点是否允许执行
允许读取哪些输入表和字段
允许写入哪些目标表和字段
允许采用哪些写入模式
是否允许登记和发布输出
权限句柄有效期与撤销
```

它已经能覆盖工作流内部数据，但数据库节点还需要增加一种逻辑资源范围。

建议增加：

```text
database_profile.read
database_table.read
database_table.append
database_table.update
database_table.clear
database_table.create
database_schema.modify
```

权限声明不使用真实路径，而使用：

```text
profile_id
logical_table_id
field_scope
write_mode
```

例如：

```text
节点：database_node_03
数据库配置：db_profile_001
逻辑表：material_summary

允许：
- 读取字段：编码、型号、板号
- 更新字段：板号

禁止：
- 删除表
- 修改结构
```

由于数据库节点是内置核心节点，可以由它在真正执行SQL前检查权限句柄。

结构为：

```text
PermissionManager批准
→ 数据库节点获得权限句柄
→ 数据库节点解析资源配置
→ 数据库节点执行前验证操作范围
```

这不需要恢复通用资源代理。

# 五、默认外部引用、物化为次选项

这个方向可行，但需要定义两种输出。

## 1. 外部数据库引用

数据库节点不复制数据，只返回：

```text
ExternalTableRef
```

例如：

```text
ref_kind：EXTERNAL_DATABASE_TABLE
provider：sqlite
profile_id：db_profile_001
logical_table：material_summary
query_id：query_018
schema_fingerprint：schema_62
consistency_mode：LIVE
```

优点：

- 几乎没有复制成本；
- 大表性能好；
- 可以使用数据库索引；
- 可以把筛选、JOIN和聚合推给数据库；
- 数据不需要重复保存。

## 2. 工作流标准表

数据库节点执行查询后，将结果转成：

```text
MaterializedTableRef
```

优点：

- 与外部数据库状态脱离；
- 可以查看稳定快照；
- 跨工作流共享简单；
- 调试和复现更可靠；
- 普通节点可直接处理。

缺点：

- 产生复制；
- 占用额外存储；
- 大表物化成本高。

---

# 六、外部引用模式的实际限制

外部引用不是只返回一个表名就结束了。

下游节点必须能够读取它。

例如：

```text
数据库节点
→ ExternalTableRef
→ 筛选节点
```

筛选节点需要一种方式获得数据。

## 方案一：数据库原生节点链

```text
数据库读取节点
→ 数据库筛选节点
→ 数据库连接节点
→ 数据库聚合节点
```

这些节点继续操作外部引用，并将操作转换成SQL。

这是性能最好的方式，但节点种类会增加。

## 方案二：按需物化

普通筛选节点收到外部引用后，发现自己不支持数据库原生操作，就请求：

```text
ExternalTableRef
→ 按需读取
→ MaterializedTableRef
```

这相当于延迟物化。

推荐结合：

```text
数据库原生节点支持引用直通
普通表格节点在必要时触发物化
```

因此“默认外部引用、物化次选”是可行的，但协议需要告诉下游：

```text
该节点是否支持这种TableRef
是否需要物化
是否支持查询下推
```

# 七、TableRef协议需要补充什么

当前主方案已经有 `TableRef / DataRef` 概念，不需要推翻，只需要细化。

建议加入：

```text
TableRef
├─ table_ref_id
├─ ref_kind
│  ├─ EXTERNAL_DATABASE_TABLE
│  ├─ EXTERNAL_DATABASE_QUERY
│  ├─ MATERIALIZED_TABLE
│  ├─ SHARED_TABLE_MEMBER
│  └─ RUNTIME_TABLE
├─ provider_id
├─ resource_profile_id
├─ logical_table_id
├─ opaque_handle
├─ schema
├─ schema_fingerprint
├─ consistency_mode
├─ version
├─ lifecycle
├─ capabilities
└─ created_by
```

其中 `capabilities` 可以声明：

```text
READ
FILTER_PUSHDOWN
PROJECT_PUSHDOWN
JOIN_PUSHDOWN
AGGREGATE_PUSHDOWN
MATERIALIZE
APPEND
UPDATE
```

这样普通节点可以判断：

```text
我可以直接处理
还是需要先物化
```

# 八、外部引用和不可变版本之间的矛盾

假设：

```text
A工作流在10:00发布数据库表引用V1
```

但是10:05数据库中的表被修改。

那么B和C虽然都读取“V1引用”，实际读到的数据可能不同。

所以必须定义一致性等级。

## `LIVE_REFERENCE`

```text
引用描述不变
但数据库内容可能变化
```

性能最好，但不能严格复现。

## `PINNED_SNAPSHOT`

```text
引用绑定数据库事务快照或数据库快照标识
```

数据保持一致，但要求数据库支持快照，并且可能需要保留连接或事务。

## `MATERIALIZED_SNAPSHOT`

```text
查询结果已经复制成工作流标准表
```

一致性最强，生命周期最容易管理。

建议：

```text
工作流内部普通使用：
默认LIVE_REFERENCE

跨工作流共享：
优先PINNED_SNAPSHOT
无法固定快照时使用MATERIALIZED_SNAPSHOT
```

也可以允许共享 `LIVE_REFERENCE`，但UI必须明确标记：

```text
该共享版本只固定引用，不保证内容快照一致
```

# 九、工作流之间共享的不应只是单张TableRef

如果A工作流有多个表，并希望统一作为V1发布，单独登记：

```text
表1@V1
表2@V1
表3@V1
```

会产生问题：

```text
表1发布成功
表2还没发布
B工作流已经开始读取
```

B可能获得不完整的一组数据。

因此应增加：

> 共享发布包，或者共享表集合版本。

建议名称：

```text
SharedPublication
```

# 十、SharedPublication结构

例如A工作流发布：

```text
A工作流共享数据@V1
├─ 订单表 → table_orders@17
├─ 明细表 → table_items@32
└─ 配置表 → table_config@5
```

这里：

- `V1` 是这次共享发布的整体版本；
- 各成员表可以保留自己的内部版本；
- 三张表以原子方式一起发布。

协议示意：

```text
SharedPublication
├─ publication_id
├─ share_name
├─ publication_version
├─ producer_workflow_id
├─ producer_run_id
├─ created_at
├─ status
├─ consistency_level
├─ members
│  ├─ export_name
│  ├─ table_ref_id
│  └─ exact_table_version
├─ input_snapshot_id
├─ retention_policy
└─ lineage
```

例如：

```text
publication_id：pub_A_001
share_name：A工作流订单数据
publication_version：V1

members：
- orders → table_01@17
- items → table_02@32
```

这样B工作流读取时获得的是一组一致的表，而不是分别查询最新版本。

# 十一、发布共享表节点

哪些表加入 `TableRef`、哪些阶段的数据允许共享，应该由节点显式处理。

可以增加核心节点：

```text
发布共享表节点
```

输入：

```text
当前工作流中的一个或多个TableRef
```

配置：

```text
共享名称
选择发布的表
每张表的导出名称
发布模式
一致性等级
保留策略
是否触发下游工作流
```

输出：

```text
SharedPublicationRef
```

处理流程：

```text
选择表
→ 检查所有表有效
→ 固定每张表的精确版本
→ 创建STAGING发布包
→ 全部登记完成
→ 原子切换为PUBLISHED
→ EventRouter发送事件
```

发布失败时不会出现半套数据。

---

# 十二、订阅 / 读取共享表节点

第二个核心节点：

```text
读取共享表节点
```

配置：

```text
共享名称
选择读取哪些成员表
版本策略
等待策略
超时时间
一致性要求
```

版本策略：

```text
LATEST
EXACT_VERSION
NEXT_VERSION
NEWER_THAN_CURRENT
SAME_AS_UPSTREAM
FOLLOW_LINEAGE
```

输出：

```text
一个或多个TableRef
InputSnapshot
```

# 十三、表通信的存储方式是否可以删除

可以从用户层配置和核心方案重点中删除，但不能从底层协议中完全删除。

原因是读取方仍然必须知道：

```text
这张表怎样打开
```

例如它可能在：

```text
SQLite
外部MySQL
Arrow文件
内存映射
共享内存
```

所以 `TableRef` 内部仍需要：

```text
storage_type
provider_id
opaque_handle
```

但这些应当是节点产生和系统登记的内部元数据，不需要让“发布共享表节点”决定。

正确边界：

```text
生产数据的节点：
决定实际存储方式

发布共享表节点：
只选择哪些TableRef进入发布包

RuntimeDataRegistry：
登记引用、版本和生命周期

读取节点：
按TableRef协议打开数据
```

因此原来的“表通信存储方式”可以从主要设计章节降级为底层实现说明。

# 十四、B与C怎样同步读取A的同一版本

示例：

```text
A发布V1，包含表1、表2
B读取A@V1
C需要与B使用相同的A版本
```

不能依靠C自己再查询“最新版本”，因为此时A可能已经发布V2。

建议引入：

```text
InputSnapshot
```

B开始执行时生成：

```text
input_snapshot_B_01
└─ A工作流共享数据@V1
   ├─ 表1@17
   └─ 表2@32
```

B运行期间，无论A是否发布V2，B始终读取V1。

B发布自己的结果时，可以附带：

```text
source_input_snapshot = input_snapshot_B_01
```

C有两种读取方式。

## 方式一：显式使用B的输入快照

C配置：

```text
读取B使用的上游数据版本
```

于是C自动获取：

```text
A@V1
```

## 方式二：跟随B的血缘

C读取B发布的结果时选择：

```text
FOLLOW_LINEAGE
```

系统从B的发布元数据中解析：

```text
B结果来源于A@V1
```

然后C获得同一版本。

这样不会把B的内部租约直接交给C，而是传递明确的输入版本关系。

# 十五、共享版本生命周期

A@V1的生命周期不能简单规定为“B读取完成就删除”。

需要同时考虑：

```text
B是否仍在读取
C是否仍在读取
B发布的结果是否要求保留A@V1
调试记录是否引用A@V1
是否设置了保留时间
A@V1是否仍然是当前版本
```

建议状态：

```text
STAGING
PUBLISHED
ACTIVE
RETIRED
RELEASABLE
RELEASED
ORPHANED
```

生命周期示例：

```text
A创建V1
→ STAGING

A完成全部表登记
→ PUBLISHED

B获取读取租约
→ ACTIVE

A发布V2
→ V1变成RETIRED

C通过B的InputSnapshot读取V1
→ V1仍保持ACTIVE / RETIRED但不可释放

B、C都完成
→ 释放直接租约

没有调试引用和依赖固定
→ RELEASABLE

满足保留策略
→ RELEASED
```

---

# 十六、读取租约与依赖固定

建议区分两种引用。

## `ReadLease`

表示某个工作流当前正在读取。

```text
lease_id
publication_id
publication_version
selected_members
consumer_workflow_run_id
acquired_at
expires_at
```

工作流结束或读取节点结束后自动释放。

## `DependencyPin`

表示一个下游结果仍然依赖这个上游版本。

例如：

```text
B的结果@V1
依赖
A的输入@V1
```

只要B的结果需要完整回放，就可以继续固定A@V1。

不过不建议默认永久固定所有上游数据，否则存储会不断膨胀。

可配置：

```text
只保留血缘元数据
保留上游完整数据
保留到下游完成
保留指定时间
```

---

# 十七、生命周期完成的判断

在确定的A→B→C链路中：

> C读取完成后，A工作流的V1才算完成一个生命周期。

这个理解成立。

但平台最好不要硬编码“必须等C”，而是统一计算：

```text
没有活跃ReadLease
没有DependencyPin
已经不再是current版本
满足保留策略
没有调试和回放引用
```

满足这些条件后，A@V1才进入 `RELEASABLE`。

这样以后即使再加入D工作流，也不需要修改A的逻辑。

# 十八、四种共享通信模式

## 1. 快照发布

```text
工作流A完成一个阶段
→ 原子发布一组表
→ B读取固定版本
```

这是第一版核心模式。

## 2. 事件触发

```text
A发布新版本
→ EventRouter发送PUBLICATION_AVAILABLE
→ 自动启动B
```

适合工作流串联。

## 3. 追加批次

适合：

```text
文件变化记录
网页新增数据
日志
订单增量
```

表现为：

```text
Batch V1
Batch V2
Batch V3
```

每个批次不可变，消费者保存自己的读取游标。

## 4. 实时流

适合：

```text
屏幕特征
串口
实时状态
识别事件
```

使用：

```text
StreamRef
sequence_id
checkpoint
```

它与普通表版本不同，需要有界队列、丢弃策略和背压机制，建议后续实现。

# 十九、单节点延迟和卡死：当前MD的不足

当前主方案是：

```text
EngineHost
└─ WorkflowRunProcess
   ├─ 普通节点
   └─ 独立Worker
```

它能保证：

```text
一个工作流卡死
不影响EngineHost和其他工作流
```

但不能完全保证：

```text
一个普通节点卡死
不影响当前工作流的其他独立分支
```

因为普通节点目前仍可能直接运行在 `WorkflowRunProcess` 中。

---

# 二十、建议把WorkflowRunProcess变成控制进程

调整为：

```text
WorkflowRunProcess
├─ DAG Scheduler
├─ NodeTaskManager
├─ 状态机
├─ 心跳和取消控制
├─ NodeExecutorPool客户端
└─ DedicatedWorker客户端
```

它不直接运行普通业务节点，而是将节点任务提交给：

```text
NodeExecutorProcess
```

整体结构：

```text
EngineHost
└─ WorkflowRunProcess
   ├─ DAG Scheduler
   ├─ NodeExecutor 1
   ├─ NodeExecutor 2
   └─ DedicatedWorker
```

如果 `NodeExecutor 1` 中的节点卡死：

```text
WorkflowRunProcess仍然正常
NodeExecutor 2仍然正常
其他独立分支继续执行
```

# 二十一、节点执行模式

不再按可信程度分类，而是按运行性质分类。

## `IN_PROCESS`

只用于极短、无阻塞的框架内部节点，例如：

```text
参数转换
逻辑路由
锚点
简单状态判断
```

## `PROCESS_POOL`

作为普通业务节点默认模式：

```text
筛选
替换
正则
去重
数值计算
普通文件解析
普通数据库读取
```

一个执行器卡住后，可以单独终止并补充新执行器。

## `DEDICATED_WORKER`

适合：

```text
数据库固定快照连接
Office
浏览器
模型
串口
屏幕识别
长连接
有状态任务
```

特别是外部数据库的 `PINNED_SNAPSHOT`，可能需要保持数据库连接，应使用专用Worker，而不是普通进程池。

---

# 二十二、DAG调度防止慢节点阻塞无关分支

例如：

```text
       ┌→ B → D
A →────┤
       └→ C → E
```

B很慢时：

```text
B：RUNNING
D：WAITING_DEPENDENCY
C：SUCCEEDED
E：RUNNING
```

只让依赖B的D等待，C和E继续。

这要求工作流调度依据依赖图，而不是简单逐节点串行执行。

---

# 二十三、节点卡死判断

需要同时监控：

```text
执行器进程心跳
节点任务心跳
进度
最后进度时间
当前阶段
CPU时间
内存变化
```

状态区分：

```text
心跳正常、进度变化：
RUNNING

心跳正常、长时间无进度：
LONG_RUNNING

任务心跳停止：
SUSPECTED_HUNG

超过最大时间：
TIMED_OUT

进程退出：
CRASHED
```

数据库长查询、文件等待和网络等待不能只因为进度不变就判定卡死。

---

# 二十四、取消和强制终止流程

```text
节点超时或用户取消
→ 停止继续调度该节点任务
→ 发送CancelToken
→ 等待宽限时间
→ 仍未退出
→ 终止NodeExecutor进程
→ 废弃未完成输出
→ 释放临时资源和租约
→ 创建新的NodeExecutor
```

若节点运行在专用Worker中，则只终止专用Worker。

如果整个 `WorkflowRunProcess` 都失去响应，EngineHost再执行工作流级终止。

这形成三层故障域：

```text
EngineHost
└─ WorkflowRunProcess
   └─ NodeExecutor / DedicatedWorker
```

---

# 二十五、节点结果必须原子发布

节点执行过程中产生的数据标记为：

```text
STAGING
```

只有节点完整成功、Schema校验完成后才切换为：

```text
PUBLISHED
```

节点卡死或被杀死时：

```text
删除STAGING结果
不更新下游TableRef
不触发共享发布事件
```

这样下游不会读取半张表。

数据库写入节点则应使用事务：

```text
BEGIN
→ 执行写入
→ 成功COMMIT
→ 失败ROLLBACK
```

---

# 二十六、节点失败策略

建议支持：

```text
FAIL_FAST
CONTINUE_INDEPENDENT
SKIP_DEPENDENTS
RETRY
USE_LAST_SUCCESS
```

其中数据库写入、Office写回、自动点击等有副作用节点，不能默认自动重试。

节点需要声明：

```text
retry_safe
idempotency_key
has_external_side_effects
```

只有满足幂等条件时才自动重试。

---

# 二十七、多工作流写同一数据库

外部数据库的并发和事务仍由数据库节点负责。

工作流调度层可以增加一个轻量锁声明：

```text
resource_lock_key
```

例如：

```text
database:db_profile_001:table:material_summary
```

读取节点申请共享锁：

```text
READ_SHARED
```

写入节点申请排他锁：

```text
WRITE_EXCLUSIVE
```

这不需要恢复 `ResourceBroker`，只是调度器避免两个工作流明显同时覆盖同一逻辑表。

数据库自身仍负责最终事务和并发控制。

# 后续补充讨论：运行中转数据库与工作流表模型

> 本节用于明确运行中转数据库、持久业务数据库、当前表、辅助表、运行SQL表、外部映射表和共享表版本之间的边界。当前阶段优先讨论运行中转数据库，不展开持久业务数据库的完整安全策略。

## 一、数据库分类应从“安全程度”进一步转为“数据用途”

新的划分可以理解为：

> **运行中转数据库**：为工作流计算、表交换和性能服务。
> **持久业务数据库**：保存有业务价值的数据，需要更严格的写入、审计和一致性控制。

当前阶段只讨论第一类是合理的。这样可以先把工作流内部数据模型做稳，不必过早把业务数据库安全策略混进来。

不过需要先理顺一个概念：

> “当前表、内存副表”描述的是表在工作流中的**角色**；
> “内存、SQL、外部数据库”描述的是表的**存储后端**。

这两组概念不能完全并列，否则以后会出现“当前表到底是内存表还是SQL表”的混乱。

---

## 二、建议将表拆成四个维度

工作流中的一张表，可以同时拥有四类属性。

### 1. 表的工作流角色

```text
CURRENT
AUXILIARY
SHARED
```

- `CURRENT`：当前节点链主要处理的表。
- `AUXILIARY`：工作流产生的辅助表、中转表、匹配表。
- `SHARED`：已经发布给其他工作流的表版本。

### 2. 表的存储后端

```text
MEMORY
RUNTIME_SQL
EXTERNAL_DATABASE
FILE_BACKED
```

- `MEMORY`：内存中的表。
- `RUNTIME_SQL`：运行中转数据库里的表。
- `EXTERNAL_DATABASE`：映射到其他外部数据库的表。
- `FILE_BACKED`：Arrow、Parquet或其他文件形式。

### 3. 生命周期

```text
NODE_SCOPE
GROUP_SCOPE
WORKFLOW_SCOPE
SHARED_SCOPE
PERSISTENT
```

### 4. 可变状态

```text
WORKING_MUTABLE
PUBLISHED_IMMUTABLE
```

所以一张表可以是：

```text
角色：CURRENT
存储：RUNTIME_SQL
范围：WORKFLOW_SCOPE
状态：WORKING_MUTABLE
```

也可以是：

```text
角色：SHARED
存储：RUNTIME_SQL
范围：SHARED_SCOPE
状态：PUBLISHED_IMMUTABLE
```

这种划分会比把表直接分成“当前表、内存副表、SQL表”更稳定。

---

## 三、重新定义“当前表”

当前表不应该是一种固定的存储类型。

更准确的定义是：

> **当前表是当前执行分支默认使用的主表引用。**

例如：

```text
current_table → table_ref_001
```

而 `table_ref_001` 可以实际指向：

- 内存表；
- 临时SQLite表；
- DuckDB表；
- 外部数据库映射表；
- 已物化的数据表。

切换当前表，本质上只是修改工作流中的默认表引用：

```text
current_table：
table_ref_001
→ table_ref_008
```

不一定需要复制数据。

这意味着数据库映射表也可以直接成为当前表。

---

## 四、重新定义“内存副表”

内存副表可以定义为：

> **工作流执行期间产生的、非当前主表的临时辅助数据。**

典型用途：

- 匹配对照表；
- 筛选结果；
- 中间统计；
- 字段映射表；
- 临时缓存；
- 某个节点的第二输出表。

例如：

```text
当前表：订单表
副表1：客户匹配表
副表2：异常记录表
副表3：去重结果表
```

“副表”也不一定永远保存在内存中。

小表可以使用：

```text
MEMORY
```

较大的副表可以自动使用：

```text
RUNTIME_SQL
```

因此更准确的名称其实是：

```text
辅助表
```

而不是严格意义上的“内存副表”。

---

## 五、运行中转数据库的定位

当前提出的第一类数据库，可以正式定义为：

> **Runtime Database：由工作流运行阶段使用的中转数据库，为计算、缓存、表交换和大数据处理提供SQL存储。**

它不是正式业务数据仓库。

主要作用：

```text
保存工作流临时表
承担大表处理
减少内存复制
支持SQL筛选、JOIN和聚合
支持跨节点表引用
支持跨工作流共享
```

它的数据通常满足：

- 可以重新生成；
- 可以修改；
- 可以删除；
- 不要求长期保存；
- 不作为业务唯一数据源；
- 生命周期由工作流或共享表机制控制。

---

## 六、建议将“临时数据库映射节点”改名

“临时数据库映射节点”容易让人误解为它映射的任何数据库都是临时的。

更适合的名称是：

```text
运行数据库映射节点
```

或者：

```text
工作流数据库挂载节点
```

这个节点只负责：

```text
选择数据库配置
解析真实地址
选择允许映射的表
为表设置工作流内部名称
将表注册到当前工作流
输出DatabaseMountRef和TableRef
```

它不负责：

```text
读取具体数据
执行筛选
执行写入
修改字段
决定业务逻辑
```

这些仍由后续节点完成。

---

## 七、运行数据库映射节点的输出

建议映射节点输出两种对象。

### 1. `DatabaseMountRef`

代表本次工作流挂载的数据库环境。

```text
DatabaseMountRef
├─ mount_id
├─ profile_id
├─ provider_type
├─ namespace
├─ ownership_mode
├─ access_mode
├─ lifecycle
└─ mapped_tables
```

### 2. 多个 `TableRef`

例如：

```text
运行数据库映射节点
├─ 订单表 → runtime_db.orders
├─ 明细表 → runtime_db.items
└─ 匹配表 → runtime_db.mapping
```

后续节点只看到：

```text
runtime_db.orders
runtime_db.items
runtime_db.mapping
```

真实数据库路径只有映射节点或数据库执行组件能够解析。

---

## 八、必须区分两种“临时数据库”

这里存在一个重要风险。

### 1. 平台管理的运行数据库

例如：

```text
runtime/workflow_001.db
```

由平台或节点创建，专门用于工作流运行。

这种数据库可以允许：

- 任意建表；
- 任意删表；
- 清空；
- 覆盖；
- 工作流结束后删除。

建议标记：

```text
ownership_mode = MANAGED_RUNTIME
```

### 2. 映射的现有外部数据库

例如：

```text
D:\db\SQL\正式数据.db
```

即使在工作流中称为“临时映射”，节点对它的修改仍然会真实保存。

这种数据库应标记：

```text
ownership_mode = MAPPED_EXTERNAL
```

不能因为被临时挂载，就默认允许任意删除和覆盖。

所以“可以任意更改”的规则，只适合：

```text
MANAGED_RUNTIME
```

不应自动适用于所有映射数据库。

---

## 九、工作流内数据库读写怎么执行

新的设想是：

> 映射节点只负责映射，读取和写入由其他工作流节点完成。

这是合理的。

结构可以是：

```text
运行数据库映射节点
        ↓
   DatabaseMountRef
        ↓
数据库读取节点 / 查询节点 / 写入节点
        ↓
       TableRef
```

或者：

```text
运行数据库映射节点
→ 输出已映射的TableRef
→ 筛选、替换、去重等节点直接处理
```

第二种方式体验更好，但要求表处理节点支持不同存储后端。

---

## 十、为了避免核心模块过重，需要一个很薄的表后端协议

这里不需要恢复之前取消的 `ResourceBroker`。

只需要定义一个很薄的：

```text
TableProvider协议
```

它由对应数据库节点包实现，而不是由主程序实现所有数据库。

最低能力可以包括：

```text
get_schema()
read()
apply_changes()
create_table()
drop_table()
begin_transaction()
commit()
rollback()
create_snapshot()
release()
```

`TableRef` 中记录：

```text
provider_id
opaque_handle
capabilities
```

后续节点只判断：

```text
该TableRef支持什么操作
```

例如：

```text
READ
UPDATE
APPEND
FILTER_PUSHDOWN
JOIN_PUSHDOWN
SNAPSHOT
```

平台负责路由，数据库节点包负责真正执行。

这仍然符合：

> 数据库能力由节点扩展，核心只管理协议和引用。

---

## 十一、临时数据库默认不物化是合理的

运行数据库不需要再物化数据。

因为运行数据库本身已经是工作流标准中转存储。

例如：

```text
SQLite运行表
→ 筛选
→ JOIN
→ 更新
→ 共享发布
```

没有必要再做：

```text
SQLite运行表
→ 复制成内存表
→ 再处理
```

这样会增加：

- 数据复制；
- 内存占用；
- 格式转换；
- 跨进程传输。

所以运行数据库表可以直接使用：

```text
RuntimeSqlTableRef
```

而不是 `MaterializedTableRef`。

这里可以重新理解“物化”：

- 从正式外部数据库复制到工作流表，属于物化；
- 数据本来就在运行数据库中，就已经是工作流可直接使用的数据，不需要再次物化。

---

## 十二、持久业务数据库以后单独设计

第二类数据库，更适合命名为：

```text
持久业务数据库
```

或者：

```text
受控持久数据库
```

而不只是“安全数据库”。

它的核心区别是：

```text
数据具有业务价值
不能随意丢失
修改需要可追踪
可能需要备份和恢复
```

以后需要考虑：

- 写入权限；
- 字段级权限；
- 事务；
- 覆盖确认；
- 版本冲突；
- 审计；
- 备份；
- 回滚；
- 外部引用与物化快照。

这些暂时不应混入运行数据库方案。

---

## 十三、工作流内部表的建议分类

结合当前想法，可以初步定义为以下几类。

| 名称 | 含义 | 默认可变 | 默认生命周期 |
|---|---|---:|---|
| 当前表 | 当前分支主要处理的表引用 | 是 | 工作流 |
| 辅助表 | 节点处理产生的中间表 | 是 | 节点/组/工作流 |
| 运行SQL表 | 中转数据库中的工作表 | 是 | 工作流 |
| 外部映射表 | 指向其他数据库的表 | 取决于配置 | 外部资源 |
| 共享表版本 | 已发布给其他工作流的表 | 否 | 共享 |
| 持久业务表 | 有长期业务价值的表 | 受控 | 持久 |

其中：

> 当前表和辅助表是逻辑角色；运行SQL表和外部映射表是存储方式。

---

## 十四、任意更改与不可变共享并不冲突

工作流内的表可以任意更改，也可以作为工作流之间的表，但必须规定一个发布边界。

### 工作流内部

表处于：

```text
WORKING_MUTABLE
```

允许：

- 修改；
- 追加；
- 删除；
- 清空；
- 改结构；
- 切换当前表。

### 发布给其他工作流时

发布节点固定当前版本：

```text
WORKING_MUTABLE
→ 创建共享版本V1
→ PUBLISHED_IMMUTABLE
```

之后工作流内部仍可继续修改工作副本：

```text
工作表继续变化
→ 发布V2
```

其他工作流读取的是：

```text
V1或V2
```

而不是正在修改中的工作表。

因此可以形成：

```text
运行表：持续可变
共享版本：发布后不可变
```

---

## 十五、运行SQL表发布时不一定要复制全部数据

发布不可变版本不等于一定完整复制数据库。

可以采用：

### 1. 版本表

```text
orders_working
→ orders_pub_v1
```

### 2. 数据库快照

数据库支持时，固定一个快照或事务版本。

### 3. 写时复制

发布后原数据块只读，后续修改产生新数据块。

### 4. 表重命名与新工作表

```text
working_table
→ 重命名为published_v1
→ 创建新的working_table
```

对于第一版，SQLite运行数据库可以采用简单方式：

```text
发布时复制为版本表
```

实现简单，后续再优化。

---

## 十六、多个工作流共享这些表

无论表原来是：

- 当前表；
- 辅助表；
- 运行SQL表；
- 外部映射表；

都可以通过“发布共享表节点”选择性注册。

例如：

```text
A工作流当前拥有：
├─ 当前表：orders
├─ 辅助表：errors
├─ 运行SQL表：mapping
└─ 临时统计表：summary
```

发布节点选择：

```text
发布：
- orders
- summary

不发布：
- errors
- mapping
```

形成：

```text
SharedPublication A@V1
├─ orders
└─ summary
```

共享范围由发布节点决定，而不是数据库映射节点决定。

## 十七、权限管理怎么适配

对于平台管理的运行数据库，可以采用简化权限。

例如映射节点创建：

```text
mount_id = runtime_db_01
ownership_mode = MANAGED_RUNTIME
```

默认授予当前工作流：

```text
读取
写入
建表
删表
修改结构
发布表
```

不需要用户逐项审核。

权限管理仍记录：

```text
哪个工作流
哪个节点
对哪张表
执行了什么操作
```

审计可以保持轻量。

对于外部映射数据库，则不能默认全部通过，需要根据之后的持久数据库方案处理。

所以可以形成：

```text
MANAGED_RUNTIME：
默认允许工作流内部操作

MAPPED_EXTERNAL：
按数据库映射配置决定

PERSISTENT_BUSINESS：
后续使用受控写入规则
```

## 十九、这套方案带来的性能优势

运行数据库可以直接承担：

- 大表筛选；
- JOIN；
- 去重；
- 排序；
- 聚合；
- 批量更新；
- 工作流中间表保存。

数据流可以变成：

```text
运行SQL表
→ SQL处理节点
→ 新运行SQL表
→ 共享发布
```

避免：

```text
SQL
→ Python对象
→ 跨进程复制
→ Python处理
→ 再写SQL
```

只要节点支持SQL引用和查询下推，性能会很好。

## 二十、需要重点避免的问题

### 1. 把现有业务数据库误标为临时数据库

数据库的真实性质由它是否可以安全删除和重建决定，不由节点名称决定。

### 2. 当前表直接等于某种存储

当前表应该是逻辑指针，而不是固定内存表。

### 3. 发布共享表时仍允许原地修改

发布后必须固定版本；后续修改应进入新的工作版本。

### 4. 一个运行数据库被多个工作流直接同时写

共享发布版本应只读。需要共同加工时，应由各工作流创建自己的工作副本或新版本。

### 5. 临时数据库无限增长

需要配置：

- 工作流结束清理；
- 共享租约结束后清理；
- 最大保留时间；
- 异常退出后的孤儿表回收。

## 二十一、建议当前阶段正式确定的定义

### 当前表

> 当前节点链默认处理的主 `TableRef`，它是一种逻辑角色，不限制底层存储方式。

### 辅助表

> 工作流处理过程中产生的非主表数据，可以使用内存或运行数据库存储。

### 运行SQL表

> 保存在运行中转数据库中的可变工作表，用于大数据处理、中间结果和跨节点传递。

### 外部映射表

> 通过数据库配置映射到外部数据库的表，其修改是否允许由映射配置和数据库类别决定。

### 共享表版本

> 从某个工作流阶段显式发布的不可变表版本，可由其他工作流通过 `RuntimeDataRegistry` 读取。

### 持久业务表

> 具有长期业务价值、需要受控写入和审计的数据表，后续单独设计。

# 本轮补充收口

当前方向可以成立，并且比之前“默认外部数据库引用、必要时物化”的说法进一步清晰。

```text
运行中转数据库：
本身就是工作流标准存储
不需要再次物化
允许工作流内部自由处理
```

```text
持久业务数据库：
作为正式外部数据源
需要单独的映射、写入和安全策略
物化与否由该类节点决定
```

同时应把表模型明确为：

> **当前表、辅助表是工作流角色；内存、运行SQL和外部数据库是存储方式；共享版本是发布状态；持久业务表是数据价值和生命周期类别。**

这样后续讨论共享表、节点处理和权限时，不会把“表的用途”和“表存在哪里”混在一起。
