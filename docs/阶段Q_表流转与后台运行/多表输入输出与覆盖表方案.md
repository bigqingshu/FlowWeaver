# 多表输入输出与覆盖表方案

> 文档状态：前置方案整理
> 当前用途：收口多表输入、多表输出、插件节点表槽位、覆盖已有表和删除表节点的规划
> 核心目标：让复杂节点和插件节点可以清晰接收多张表、输出多张表，同时保持主程序低耦合

## 1. 一句话结论

多表能力建议按“槽位绑定”实现：

```text
输入槽位：节点声明自己需要哪些表
输出槽位：节点声明自己会输出哪些表
用户配置：给每个槽位选择当前表、内存表、运行内 SQL 表或外部 SQL 引用表
节点执行：节点或插件按槽位名自行理解每张表的业务含义
```

这样不用把多张表硬揉成一张表，也不用让主程序理解插件内部业务。

本方案的总原则是：

```text
节点管理表操作。
主程序只提供表映射、表状态登记和基础读写能力。
```

也就是说，主程序不决定“怎么处理表”。主程序只告诉节点：有哪些表、这些表在哪里、能不能读写、怎么拿到读写接口。节点自己决定读哪几张表、怎么处理、写到哪里，以及使用新建、覆盖、追加、按主键更新还是合并。

## 2. 核心规则

| 模块 | 规则 | 说明 |
| --- | --- | --- |
| 多表输入 | 节点定义声明输入槽位数量和名称 | 例如 `main_table`、`rules_table`、`mapping_table` |
| 输入表来源 | 当前表、内存表、运行内 SQL 表、外部 SQL 引用表 | 仍使用稳定选择器，不保存运行期 `table_ref_id` |
| 插件节点输入 | 插件自行区分多个输入源 | 主程序只映射表引用和槽位名，插件负责业务解释 |
| 多表输出 | 节点定义声明输出槽位 | 普通表处理节点至少保留当前表输出，可额外输出辅助表 |
| 输出到当前表 | 表名不可修改 | 当前表仍是主链路概念 |
| 输出到新建表 | 表名可填写 | 节点可新建内存表或运行内 SQL 表 |
| 输出到已有表 | 表名不可修改 | 主程序提供目标表映射，具体写入语义由节点决定 |
| 覆盖已有表 | 允许作为节点能力 | 节点使用底层事务 / 原子替换能力完成覆盖 |
| 删除表节点 | 可以规划为节点能力 | 用于删除或释放内存表、运行内 SQL 表状态 |
| 真实 SQL 写入 | 不进通用规则 | 继续由专门 SQL 写入 / 修改节点负责 |

### 2.1 收束后的配置形态

节点方案可以收束为“节点声明表槽位，UI 按声明渲染表选择区”。

节点定义声明：

```text
我需要哪些输入表。
我会产生或修改哪些输出表。
每个槽位的名称、显示名、说明、是否必填、允许的表类型。
```

UI 根据节点声明显示：

```text
输入表
main_table / 主数据表        [下拉选择表]
rules_table / 规则表         [下拉选择表]
mapping_table / 映射表       [下拉选择表]

输出目标
result_table / 结果表        [当前表 / 新建内存表 / 新建运行SQL表 / 已有内存表 / 已有运行SQL表]
error_rows / 异常行          [不输出 / 新建内存表 / 新建运行SQL表 / 已有表]
```

这样 UI 不需要理解节点业务，只需要按节点 schema 渲染“槽位名称 + 槽位说明 + 表下拉菜单 + 输出目标选择”。

主程序提供的是表目录和可写基座：

| 类型 | UI 建议显示 | 用途 | 边界 |
| --- | --- | --- | --- |
| 当前表 | `当前表: 来源节点 / 当前状态` | 默认主链路输入和输出 | 不手动命名 |
| 内存表 | `内存表: 表名` | 轻量中转、新建或已有目标 | 不承诺长期回看 |
| 运行内 SQL 表 | `运行SQL: 表名` | 稳定中转、预览、回看 | FlowWeaver runtime sqlite，不是真实外部数据库 |
| 外部 SQL 引用表 | `SQL: 数据库名 / 表名` | SQL 节点映射进工作流的外部表引用 | 普通节点只读或按引用处理；真实写入交给 SQL 节点 |
| 本地文件 / 外部文件 | 不进入通用表目录 | 真实文件读取、写入、覆盖、删除、重命名 | 必须由指定文件类节点或插件节点显式处理 |

如果外部 SQL 当前只有连接别名，没有真实数据库名，UI 可以显示：

```text
SQL: 连接名 / 表名
```

这套模型里，“内存表”和“运行内 SQL 表”是普通节点可使用的工作流内基座；“外部 SQL 引用表”是 SQL 节点引入的外部引用，真实外部数据库的新建、覆盖、追加、更新和删除继续由专门 SQL 节点负责。

### 2.2 内部表与本地文件边界

当前表、内存表、运行内 SQL 表属于 FlowWeaver 内部数据流转。

```text
当前表
内存表
运行内 SQL 表
-> 都是工作流内部表状态
-> 由节点通过 TableRef / provider 自行读取、增加、修改、覆盖或另存
```

它们不等于用户本地文件，也不代表主程序会自动操作文件系统。

真正操作本地文件或外部资源时，必须使用明确的指定节点：

| 操作对象 | 处理方式 |
| --- | --- |
| 本地文件读取 | 文件读取 / Excel 读取 / CSV 读取 / Word 读取等指定节点 |
| 本地文件写入 | 文件写入 / Excel 导出 / CSV 导出 / Word 写入等指定节点 |
| 本地文件覆盖 | 指定写入类节点自己配置路径、覆盖策略和失败处理 |
| 本地文件删除 / 重命名 | 指定文件管理节点处理，不由普通表输出触发 |
| 外部数据库真实写入 | SQL 写入 / SQL 修改节点处理 |
| 插件访问外部资源 | 插件节点在自己的业务配置中显式声明和执行 |

主程序的表目录只提供内部表和外部引用的选择，不把“输出到表”自动解释成“写到本地文件”。如果用户需要把内部表落成 xlsx、csv、json、docx 或其他真实文件，应通过对应导出 / 写入节点完成。

## 3. 多表输入方案

### 3.1 输入槽位

节点定义声明自己支持哪些输入槽位，而不是让用户随意塞入一组无名表。

示例：

| 槽位 | 是否必填 | 说明 |
| --- | --- | --- |
| `main_table` | 是 | 主数据表 |
| `rules_table` | 否 | 规则表 |
| `mapping_table` | 否 | 映射表 |

节点运行时拿到的是：

```text
main_table -> TableRef
rules_table -> TableRef
mapping_table -> TableRef
```

插件节点可以根据槽位名自行处理业务逻辑。

### 3.2 输入来源

每个输入槽位都可以选择：

| 来源 | 说明 |
| --- | --- |
| 当前表 | 上游主链路当前表 |
| 内存表 | 上游节点生成的临时内存表 |
| 运行内 SQL 表 | runtime sqlite 表，适合稳定预览和回看 |
| 外部 SQL 引用表 | SQL 节点映射进工作流的外部数据库表引用 |

输入源继续使用稳定选择器：

```text
source_node_instance_id
+ logical_table_id
+ storage_kind
+ output_role
+ optional output_alias
```

工作流定义不保存运行期 `table_ref_id`。

## 4. 多表输出方案

### 4.1 输出槽位

节点定义声明自己会输出哪些表。

示例：

| 槽位 | 默认角色 | 说明 |
| --- | --- | --- |
| `current_output` | `CURRENT` | 主链路输出 |
| `matched_rows` | `AUXILIARY` | 匹配成功的辅助表 |
| `error_rows` | `AUXILIARY` | 异常行辅助表 |

普通表处理节点默认继续输出 `CURRENT`。特殊节点可以按节点定义声明不同输出角色。

### 4.2 输出目标

输出目标建议分为“目标绑定”，不等于主程序替节点执行表操作：

| 输出模式 | 是否可改表名 | 行为 |
| --- | --- | --- |
| 当前表 | 否 | 绑定为主链路当前表 |
| 新建内存表 | 是 | 节点可写入新的 `MEMORY / AUXILIARY` 表 |
| 新建运行内 SQL 表 | 是 | 节点可写入新的 `RUNTIME_SQL / AUXILIARY` 表 |
| 已有内存表 | 否 | 节点可对已有内存表执行覆盖、追加、更新等节点内语义 |
| 已有运行内 SQL 表 | 否 | 节点可对已有运行内 SQL 表执行覆盖、追加、更新等节点内语义 |

真实 SQL 数据库写入不进入这里，继续由专门 SQL 节点处理。

输出目标选择只解决“写到哪里”。至于写入方式是新建、覆盖、追加、按主键更新还是合并，由节点配置和节点实现决定。

## 5. 表名规则

| 场景 | 表名规则 |
| --- | --- |
| 当前表 | 不可命名，由系统管理 |
| 新建内存表 | 用户可填写表名 |
| 新建运行内 SQL 表 | 用户可填写表名 |
| 选择已有内存表 | 表名锁定为已有表名，不可修改 |
| 选择已有运行内 SQL 表 | 表名锁定为已有表名，不可修改 |

新建表名用于生成新的逻辑表。已有表名代表用户选择了一个明确目标，UI 不再允许修改名称，避免“看起来选了旧表，实际写到新名字”的误解。

## 6. 表操作语义归节点

### 6.1 主程序不管理表操作语义

选择已有表作为输出目标时，主程序只负责把目标表映射给节点，不负责决定这个表应该被覆盖、追加、按主键更新还是合并。

表操作语义全部归节点：

| 操作 | 归属 |
| --- | --- |
| 新建表 | 节点能力 |
| 覆盖表 | 节点能力 |
| 追加表 | 节点能力 |
| 按主键更新 | 节点能力 |
| 合并表 | 节点能力 |
| 冲突策略 | 节点能力 |
| 字段对齐 | 节点能力 |
| 业务校验 | 节点能力 |

主程序只提供：

| 能力 | 说明 |
| --- | --- |
| 表映射 | 把用户选择的输入槽位、输出目标解析成 TableRef 或可写目标 |
| 表状态登记 | 记录哪个节点产出了或更新了哪些表 |
| 基础读写接口 | 提供读取、写入、事务或原子替换等底层能力 |
| 错误承接 | 表不可读、目标不可写、选择器匹配失败时给出明确错误 |

### 6.2 覆盖是节点内行为

覆盖已有表仍然允许，但它是节点选择的一种写入方式，不是主程序统一替所有节点执行的逻辑。

覆盖语义建议是：

```text
输出到已有表
-> 使用已有表名
-> 节点决定采用覆盖
-> 清空目标表当前所有数据
-> 写入本节点输出数据
-> 数据预览显示最新状态
```

底层需要保证“清空旧数据”和“写入新数据”尽量是一个原子动作：

| 表类型 | 建议 |
| --- | --- |
| 运行内 SQL 表 | 节点调用事务化写入能力，清空和写入同一事务完成 |
| 内存表 | 节点先产出新结果，成功后再原子替换目标引用或状态 |
| 外部 SQL 引用表 | 不进入通用覆盖，继续由专门 SQL 节点负责 |

### 6.3 同名输出规则

如果输出目标是已有表，或新建表名与当前可写表状态中的表名相同，UI 应明确让用户选择“新建失败提示”还是“按已有表交给节点处理”。不要由主程序偷偷改名，也不要自动把覆盖改成追加。

第一版主程序不内置这些写入语义：

| 写入方式 | 行为 |
| --- | --- |
| 新建 | 节点创建新表并写入 |
| 覆盖 | 节点先清空目标表，再写入本次节点输出 |
| 追加 | 节点自行定义追加规则 |
| 按主键更新 | 节点自行定义主键、匹配列和冲突策略 |
| 合并 | 节点自行定义字段对齐、冲突策略和结果表 |

第一版可以优先实现具备新建和覆盖能力的节点；追加、按主键更新、合并后续按节点业务单独实现。关键是这些能力不进入主程序通用表策略。

### 6.4 覆盖后的预览语义

允许覆盖后，数据预览需要接受一个简单原则：

```text
表名代表当前状态，不代表历史版本。
```

用户覆盖 `stage_orders` 后，数据预览里的 `stage_orders` 显示最新覆盖结果。旧内容不强制保留。用户需要保留旧内容时，应自行另存为新表名。

这里不否定现有“节点处理前状态 / 节点处理后状态”的预览能力。节点运行链路里的前后状态仍然可以查看；这里限制的是“同一个命名表被覆盖后，数据预览层不自动把旧内容升级成历史版本管理”。

## 7. 删除表节点

可以规划一个专门的删除表节点，用于清理运行内中转表。

建议范围：

| 目标 | 第一版建议 |
| --- | --- |
| 内存表 | 可删除或释放 |
| 运行内 SQL 表 | 可删除或标记不可用 |
| 当前表 | 不建议普通删除；当前表属于主链路 |
| 外部 SQL 引用表 | 不删除真实外部数据，只移除工作流内引用或交给 SQL 节点处理 |
| 真实 SQL 表 | 不进入通用删除表节点，由专门 SQL 节点负责 |

删除表节点也应走明确配置，不应由配置字控制。

## 8. 数据预览需要的修改

数据预览需要从“只看节点输出表”升级为“查看运行内表状态”。

| 功能 | 修改方向 |
| --- | --- |
| 表状态列表 | 按当前表、内存表、运行内 SQL 表、外部 SQL 引用表分组 |
| 多输入展示 | 在节点运行详情中显示每个输入槽位绑定了哪张表 |
| 多输出展示 | 一个节点下展示多个输出槽位 |
| 覆盖状态 | 显示“由哪个节点覆盖 / 覆盖时间 / 当前有效内容” |
| 删除状态 | 删除表后显示为已删除，或从可选列表隐藏 |
| 表来源 | 每张表显示来源节点、输出槽位、storage kind、logical name |
| 表选择 | 节点配置下拉从数据预览状态层读取可用表 |
| 刷新逻辑 | 节点运行后刷新相关表状态，覆盖表直接显示最新内容 |

覆盖已有表后，数据预览不需要保留旧表内容。它只需要清楚显示当前状态来自哪次节点输出。

## 9. 职责边界

主程序只管“表映射与表状态”，不管“表怎么被操作”。

| 主程序职责 | 说明 |
| --- | --- |
| 读取槽位声明 | 从节点定义读取输入槽位和输出槽位 |
| 输入解析 | 将稳定选择器解析为运行期 TableRef |
| 输出目标映射 | 将用户选择的新表名或已有表解析成节点可用的输出目标 |
| 表目录提供 | 给 UI 提供当前可选的当前表、内存表、运行内 SQL 表和外部 SQL 引用表 |
| 可写基座提供 | 给节点提供内存表和运行内 SQL 表的创建 / 写入入口 |
| 表访问能力 | 提供读取、写入、事务、原子替换等基础接口 |
| 表状态登记 | 记录表当前状态、来源节点、输出槽位和最近更新时间 |
| 错误提示 | 缺少必填槽位、表不可读、选择器匹配失败、目标不可写时明确报错 |

节点或插件负责：

| 节点 / 插件职责 | 说明 |
| --- | --- |
| 业务含义 | 自行理解各输入槽位代表什么 |
| 数据处理 | 根据槽位表执行处理逻辑 |
| 表操作语义 | 自行决定新建、覆盖、追加、按主键更新或合并 |
| 输出内容 | 给每个输出槽位产出对应表结果 |
| 业务校验 | 校验字段、主键、映射关系、规则含义、冲突策略等业务条件 |
| 写入调用 | 调用底层表接口完成事务化写入或原子替换 |

## 10. 性能与耦合边界

### 10.1 性能硬边界

| 边界 | 要求 | 原因 |
| --- | --- | --- |
| 不把多表输入合并成一张大表 | 主程序只解析槽位到 TableRef，不自动 join、merge、concat | 避免大表复制和业务含义混乱 |
| 不传整表 rows 给插件 | 传 `slot_name -> TableRef`，插件按需读取 | 避免节点启动时一次性加载多张大表 |
| 读取按需分页或批量 | 大表读取必须走 provider 的分页 / batch reader | 避免内存峰值随表数量叠加 |
| 选择器不全局扫历史表 | 第一版只在已完成上游节点输出中解析，建议按 `source_node_instance_id` 建索引 | 避免每个节点运行都扫描整个 run 的所有 TableRef |
| 槽位数量要有限制 | 节点定义声明槽位数量，UI 和后端可设置合理上限 | 防止插件节点一次绑定过多表造成不可控 IO |
| 节点覆盖运行内 SQL 表要事务化 | 节点调用底层事务能力，清空目标表和写入新数据必须在同一事务内完成 | 避免失败后留下半空或半写状态 |
| 节点覆盖内存表要原子替换 | 节点先生成新结果，成功后再替换目标内存表引用或状态 | 避免处理失败时把原表提前清空 |
| 数据预览只刷新受影响表 | 节点运行后刷新该节点输入 / 输出 / 覆盖相关表状态，不强制全量重载 | 避免多表工作流中 UI 刷新成本放大 |
| 预览 rows 继续分页读取 | 下拉列表展示元数据，真实 rows 只在用户选择表后加载 | 避免打开工作流时把所有表数据读入 UI |

### 10.2 耦合硬边界

| 边界 | 要求 | 原因 |
| --- | --- | --- |
| 主程序不理解插件业务 | 主程序只知道槽位声明、TableRef、读写能力和错误状态 | 保持插件可扩展，不把业务逻辑塞进调度层 |
| 主程序不维护表操作语义 | 新建、覆盖、追加、按主键更新、合并都由节点决定 | 避免主程序变成表操作策略中心 |
| UI 由节点槽位声明驱动 | UI 只按输入 / 输出槽位渲染表选择区 | 避免为每个节点写一套表选择逻辑 |
| 槽位名来自节点定义 | UI 不允许用户随意创建未声明槽位 | 保证后端、插件和配置 schema 能稳定对齐 |
| 输出角色由节点声明 | 主程序只登记节点输出的 TableRef 和表状态 | 避免每类节点都改主程序分支 |
| 覆盖只作为节点能力 | 内存表和运行内 SQL 表可由节点覆盖；真实外部 SQL 仍由 SQL 节点处理 | 避免通用表策略变成数据库写入框架 |
| 删除表节点不删除真实 SQL 表 | 删除节点只清理内存表、运行内 SQL 表或工作流内引用 | 避免误把外部数据库副作用纳入主程序 |
| 内部表不等于本地文件 | 当前表、内存表、运行内 SQL 表只服务工作流内部流转 | 避免普通节点输出意外写入、覆盖或删除用户文件 |
| 本地文件操作必须指定节点 | 文件读写、覆盖、删除、重命名都由文件类节点或插件节点显式执行 | 外部副作用必须可见、可配置、可测试 |
| 配置字不控制多表行为 | 配置字不控制多表输入、覆盖、删除、快照、新建表 | 配置字只做日志、事件、进度、metrics、payload 的减法 |
| 数据预览不保存业务历史版本 | 覆盖后的表展示当前状态；历史保留由用户另存新表或运行记录策略决定 | 避免预览层变成版本管理系统 |

### 10.3 覆盖失败处理

覆盖已有表时，第一版应把失败处理说清楚：

| 场景 | 建议处理 |
| --- | --- |
| 节点处理失败 | 不进入覆盖步骤，原目标表保持不变 |
| 运行内 SQL 覆盖写入失败 | 回滚事务，原目标表保持不变或保持上一次完整状态 |
| 内存表覆盖替换失败 | 不切换目标引用，原内存表仍可用 |
| 覆盖完成但预览刷新失败 | 运行结果已生效，预览层提示刷新失败并允许手动刷新 |
| 覆盖目标不可写 | 节点失败并给出明确错误，不自动改成新建表 |

### 10.4 主程序第一版不内置

| 能力 | 不进主程序的原因 |
| --- | --- |
| 自动多表 join / merge | 这是节点或插件业务，不属于主程序通用调度 |
| 自动保留覆盖前版本 | 会显著增加存储和预览复杂度，先由用户另存控制 |
| 输出表自动改名避冲突 | 选择已有表必须交给节点明确处理；新建同名冲突需要明确提示 |
| 追加、更新、合并 | 需要主键、匹配列和冲突策略，应作为节点业务能力单独实现 |
| 配置字控制表生成和删除 | 会让配置字从反馈过滤变成数据产物开关 |

## 11. 推荐短期落点

```text
多表输入：节点定义声明槽位，用户给槽位选表。
多表输出：节点定义声明输出槽位，用户选择当前表、新建表或已有表。
UI 形态：按槽位显示名称、说明和下拉菜单。
表目录：主程序提供当前表、内存表、运行内 SQL 表、外部 SQL 引用表。
可写基座：普通节点使用内存表和运行内 SQL 表；真实外部 SQL 写入只走 SQL 节点。
外部副作用：本地文件和外部资源必须由指定节点显式读写。
表操作语义：新建、覆盖、追加、按主键更新、合并都归节点。
主程序边界：只提供表映射、表状态登记和基础读写能力。
数据预览：展示当前表状态、来源节点、槽位和最近覆盖信息。
插件节点：按槽位名自行理解多张表的业务含义。
性能边界：传 TableRef，不传整表 rows；大表按需分页读取。
覆盖边界：节点先产出成功结果，再调用事务化或原子替换能力。
```

这个方案适合后续插件节点扩展，也能避免把多张业务表强行揉成一张表造成处理复杂度上升。
