# FlowWeaver 配置字推荐默认值（临时）

> 文档状态：历史默认值讨论稿，当前字段已收口
> 原用途：给前期节点测试、预览运行、手动运行、后台运行和 UI 配置字草稿提供默认值参考
> 核心边界：配置字只限制、降低、过滤当前已经存在的主流程记录与反馈能力；不新增业务功能；副作用节点自己的真实写入、外部资源动作、覆盖策略、路径策略等仍由节点业务配置控制

## 0. 2026-07-12 有效字段校正

当前结构化编辑器和预设只生成、修改以下字段：

```text
profile
telemetry.log_level
telemetry.event_level
telemetry.event_rate_limit_per_second
telemetry.progress_enabled
telemetry.progress_interval_seconds
diagnostics.capture_error_context
diagnostics.include_metrics
diagnostics.payload_byte_limit
diagnostics.redact_columns
diagnostics.mask_policy
```

旧工作流中的 `strict_validation` 和 `diagnostics.ttl_seconds` 仅用于兼容读取：patcher 原样保留，但不主动生成、修改或显示，运行时也没有对应执行消费者。本文后续旧表格和 JSON 中若出现这两个字段，只代表历史讨论，不代表当前产品承诺；`background_fast` 和 `diagnostic` 预设不再设置非零 TTL。

## 1. 一句话结论

配置字可以用于任何工作流运行状态。它不是专门给后台工作流用的，而是用于控制本次运行记录得多还是少、粗还是细。

当前阶段建议默认使用“普通可观测”配置：保留基础日志、运行事件、进度事件和诊断信息，方便节点测试、预览运行、手动运行和 UI 确认。

等工作流稳定并转入后台运行后，再手动切到“后台降噪”配置：减少进度、详细事件和 metrics，优先保证运行速度和记录量可控。

## 2. 硬边界

配置字只做减法，不做加法。

也就是说，配置字只负责限制、降低、过滤当前已经存在的记录和反馈能力。它不能凭空增加节点能力，也不能改变节点业务语义。

主程序和节点都需要真实按照配置字运行：

- 主程序负责统一过滤 runtime events、progress、metrics、payload、脱敏和限流。
- 节点如果会上报 progress、metrics 或诊断信息，应通过主程序统一通道，让配置字真实生效。
- 节点不得用配置字改变核心计算结果、外部写入行为或副作用开关。

## 3. 配置字只管什么

配置字管理的是主流程记录能力，也就是“主程序要记录多少、推送多少、保留多少诊断信息”。

| 范围 | 是否归配置字管理 | 说明 |
| --- | --- | --- |
| 日志等级 | 是 | 控制主流程和节点运行记录的详细程度 |
| 运行事件详细程度 | 是 | 控制 runtime events 的保留粒度 |
| 进度事件 | 是 | 控制节点进度是否记录、是否推送 |
| 事件限流 | 是 | 控制高频事件写入和推送量 |
| 错误上下文 | 是 | 控制失败时是否保留主流程诊断上下文 |
| metrics | 是 | 控制节点上报的指标是否写入事件 payload |
| payload 大小限制 | 是 | 控制单次诊断/事件 payload 的最大保留量 |
| 字段脱敏 | 是 | 控制主流程记录中的敏感字段遮蔽 |
| 生成内存表预览快照 | 否 | 会创建新的 runtime sqlite 数据产物，第一版由节点业务配置或诊断运行入口控制 |
| 文件改名是否真实执行 | 否 | 属于副作用节点业务配置 |
| 写入表是否覆盖 | 否 | 属于写入类节点业务配置 |
| 插件是否调用外部服务 | 否 | 属于插件节点业务配置 |
| 节点核心计算参数 | 否 | 属于节点 `config` |
| 当前表是否输出 | 否 | 属于工作流正确性 |
| 运行数据是否自动清理 | 否 | 当前先按用户手动清理，不由配置字删除业务输出 |

## 4. 当前后端模型默认值

当前代码模型中的默认值偏“普通可观测”，适合测试阶段和前台运行观察。

| 字段 | 当前模型默认值 | 含义 |
| --- | --- | --- |
| `version` | `1.0` | 配置字结构版本 |
| `workflow.profile` | `normal` | 普通运行配置 |
| `workflow.strict_validation` | 兼容读取 | 旧 JSON 原样保留；当前 UI 不编辑，运行时无执行消费者 |
| `telemetry.log_level` | `INFO` | 保留普通信息日志 |
| `telemetry.event_level` | `progress` | 保留关键事件和进度事件 |
| `telemetry.event_rate_limit_per_second` | `0` | 不限制事件频率 |
| `telemetry.progress_enabled` | `true` | 开启进度记录 |
| `telemetry.progress_interval_seconds` | `0` | 不设置进度最小间隔 |
| `diagnostics.capture_error_context` | `true` | 失败时保留错误上下文 |
| `diagnostics.include_metrics` | `true` | 保留节点 metrics |
| `diagnostics.payload_byte_limit` | `0` | 不限制 payload 大小 |
| `diagnostics.ttl_seconds` | 兼容读取 | 旧 JSON 原样保留；当前 UI 和预设不生成，清理由独立生命周期能力负责 |
| `diagnostics.redact_columns` | `[]` | 默认不额外脱敏字段 |
| `diagnostics.mask_policy` | `none` | 默认不遮蔽 |

说明：当前模型中 `0` 表示“不限制”或“不启用该限制”，不是关闭核心运行机制。

## 5. 推荐默认值：前期测试/普通运行

这是当前阶段建议作为新工作流默认显示的配置。它和当前后端模型默认值保持一致，优先方便调试、确认节点结果和观察 UI 数据预览。

| 字段 | 推荐值 | 原因 |
| --- | --- | --- |
| `profile` | `normal` | 作为测试和普通运行默认档位 |
| `strict_validation` | 不生成 | 仅兼容旧 JSON，不属于当前可编辑字段 |
| `telemetry.log_level` | `INFO` | 保留必要运行信息 |
| `telemetry.event_level` | `progress` | 能看到节点开始、结束、失败和进度 |
| `telemetry.event_rate_limit_per_second` | `0` | 测试阶段先不隐藏事件 |
| `telemetry.progress_enabled` | `true` | 方便观察长耗时节点 |
| `telemetry.progress_interval_seconds` | `0` | 先保持当前行为 |
| `diagnostics.capture_error_context` | `true` | 失败时便于定位 |
| `diagnostics.include_metrics` | `true` | 测试阶段保留更多观察信息 |
| `diagnostics.payload_byte_limit` | `0` | 先不截断，避免调试信息丢失 |
| `diagnostics.ttl_seconds` | 不生成 | 仅兼容旧 JSON，不绑定清理策略 |
| `diagnostics.redact_columns` | `[]` | 默认由用户按需添加 |
| `diagnostics.mask_policy` | `none` | 未配置敏感列时不遮蔽 |

推荐 JSON：

```json
{
  "version": "1.0",
  "workflow": {
    "profile": "normal",
    "telemetry": {
      "log_level": "INFO",
      "event_level": "progress",
      "event_rate_limit_per_second": 0,
      "progress_enabled": true,
      "progress_interval_seconds": 0
    },
    "diagnostics": {
      "capture_error_context": true,
      "include_metrics": true,
      "payload_byte_limit": 0,
      "redact_columns": [],
      "mask_policy": "none"
    }
  },
  "node_overrides": {}
}
```

## 6. 推荐预设：后台稳定运行

当工作流已经稳定、主要用于后台批量运行时，建议切换到降噪配置。这个配置减少主流程记录压力，但关键开始、结束、失败事件仍必须保留。

| 字段 | 推荐值 | 原因 |
| --- | --- | --- |
| `profile` | `background_fast` | 表达后台快速运行意图 |
| `strict_validation` | 不生成 | 仅兼容旧 JSON，不属于当前预设 |
| `telemetry.log_level` | `WARN` | 只保留警告和错误 |
| `telemetry.event_level` | `basic` | 保留关键生命周期事件 |
| `telemetry.event_rate_limit_per_second` | `10` | 限制非关键高频事件 |
| `telemetry.progress_enabled` | `false` | 默认不记录进度细节 |
| `telemetry.progress_interval_seconds` | `5` | 如节点覆盖开启进度，至少降频 |
| `diagnostics.capture_error_context` | `true` | 失败仍要能定位 |
| `diagnostics.include_metrics` | `false` | 默认不保留详细指标 |
| `diagnostics.payload_byte_limit` | `65536` | 限制单次 payload 体积 |
| `diagnostics.ttl_seconds` | 不生成 | 当前预设不表达 TTL，生命周期由独立清理能力管理 |
| `diagnostics.redact_columns` | `[]` | 按工作流实际敏感字段填写 |
| `diagnostics.mask_policy` | `partial` | 后台记录默认建议部分遮蔽敏感字段 |

推荐 JSON：

```json
{
  "version": "1.0",
  "workflow": {
    "profile": "background_fast",
    "telemetry": {
      "log_level": "WARN",
      "event_level": "basic",
      "event_rate_limit_per_second": 10,
      "progress_enabled": false,
      "progress_interval_seconds": 5
    },
    "diagnostics": {
      "capture_error_context": true,
      "include_metrics": false,
      "payload_byte_limit": 65536,
      "redact_columns": [],
      "mask_policy": "partial"
    }
  },
  "node_overrides": {}
}
```

## 7. 推荐预设：问题诊断

当某个节点或某条工作流需要定位问题时，建议只对目标节点使用节点覆盖，不建议把整条工作流长期切到诊断档。

| 字段 | 推荐值 | 原因 |
| --- | --- | --- |
| `profile` | `diagnostic` | 明确当前是定位问题 |
| `strict_validation` | 不生成 | 仅兼容旧 JSON，不属于当前预设 |
| `telemetry.log_level` | `DEBUG` | 保留更详细日志 |
| `telemetry.event_level` | `verbose` | 保留详细事件 |
| `telemetry.event_rate_limit_per_second` | `0` | 诊断时先不隐藏事件 |
| `telemetry.progress_enabled` | `true` | 观察节点执行过程 |
| `telemetry.progress_interval_seconds` | `0` | 保留完整进度 |
| `diagnostics.capture_error_context` | `true` | 必须保留错误上下文 |
| `diagnostics.include_metrics` | `true` | 保留指标辅助定位 |
| `diagnostics.payload_byte_limit` | `262144` | 允许更大的诊断 payload |
| `diagnostics.ttl_seconds` | 不生成 | 当前预设不表达 TTL |
| `diagnostics.redact_columns` | 按需填写 | 避免调试时泄露敏感字段 |
| `diagnostics.mask_policy` | `partial` | 默认建议部分遮蔽 |

节点覆盖示例：

```json
{
  "node_overrides": {
    "node_001": {
      "profile": "diagnostic",
      "telemetry": {
        "log_level": "DEBUG",
        "event_level": "verbose",
        "progress_enabled": true
      },
      "diagnostics": {
        "include_metrics": true,
        "payload_byte_limit": 262144,
        "mask_policy": "partial"
      }
    }
  }
}
```

## 8. 不可关闭的基础机制

以下内容属于主程序正确性，不应被配置字关闭：

| 机制 | 原因 |
| --- | --- |
| 工作流运行记录 | 没有它就无法判断运行是否开始、结束或失败 |
| 节点运行记录 | 没有它就无法定位哪个节点执行过 |
| 节点状态更新 | 这是调度正确性的基础 |
| 失败、取消、超时收口 | 这是运行闭环，不是可选日志 |
| `TableRef` 传递 | 这是数据流基础，不属于反馈开关 |
| 当前表输出 | 下游节点和数据预览依赖它 |
| runtime sqlite 业务输出表 | 当前先持久化记录，按用户手动清理 |
| 内存表预览快照生成 | 会创建新的数据产物，不属于反馈记录开关 |
| 循环运行与迭代运行记录 | 这是循环真实执行的结构基础 |

配置字可以降低“记录细节”，但不能关闭“运行正确性记录”。

## 9. 临时建议

1. 新建工作流和当前测试阶段：使用 `normal`。
2. 稳定后台运行：手动切到 `background_fast`。
3. 单节点排查问题：优先使用 `node_overrides`，只提高目标节点的详细程度。
4. 副作用节点：是否真实写入、是否覆盖、目标位置、外部调用等继续放在节点业务配置中。
5. 业务输出数据：当前按持久化记录给 UI 查询，先由用户手动清理，不由配置字自动删除。
6. UI 展示：左侧中文解释，右侧英文枚举/字段值，避免用户只看到技术字段。
