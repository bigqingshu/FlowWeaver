## 0. 目标

后台工作流和手动工作流共用持久化运行机制。分页与筛选只能降低界面读取量，不能阻止运行记录、事件、节点、循环和配置字记录持续累积。

第一版增加一个明确的管理动作：

```text
删除终态运行记录
```

它与已有的“取消运行”和“清理运行表”保持分离：

| 操作 | 真实语义 |
| --- | --- |
| 取消运行 | 请求活跃 workflow process 停止 |
| 清理运行表 | 释放终态运行持有的内部表数据，保留运行历史 |
| 删除运行记录 | 在内部表完成释放后，删除该运行及其内部关联元数据 |

## 1. 执行前代码现状

当前后端已经存在：

- `GET /api/v1/runs`：分页查询运行。
- `GET /api/v1/runs/{workflow_run_id}`：读取运行详情。
- `POST /api/v1/runs/{workflow_run_id}/retry`：基于原修订创建新运行。
- `POST /api/v1/runs/{workflow_run_id}/cancel`：请求取消活跃运行。
- `DELETE /api/v1/runs/{workflow_run_id}/table-refs`：分批释放内部表。

当前不存在：

```text
DELETE /api/v1/runs/{workflow_run_id}
```

一次运行当前可能关联以下内部记录：

- workflow run 与 workflow process。
- node run、node task、node task result、输出槽位绑定。
- runtime event。
- loop run、iteration、轮次节点和轮次表绑定。
- workflow run runtime options。
- input snapshot、read lease。
- DataRef、table lease、共享发布及成员记录。

其中多组关系没有数据库级联删除，不能直接删除 `workflow_runs` 主记录。

## 2. 第一版固定规则

### 2.1 只允许删除终态运行

允许：

- `SUCCEEDED`
- `FAILED`
- `CANCELLED`
- `ABORTED`

拒绝：

- `PENDING`
- `RUNNING`
- 任何仍有 `STARTING`、`RUNNING`、`CANCEL_REQUESTED` 进程记录的运行

### 2.2 内部表必须先完成清理

删除前检查该运行的 `MEMORY` 和 `RUNTIME_SQL` DataRef。

只有以下生命周期状态视为已经脱离可用数据：

- `RELEASED`
- `RETIRED`
- `ORPHANED`

仍为 `STAGING`、`ACTIVE`、`PUBLISHED` 或 `RELEASABLE` 时拒绝删除，并提示用户先执行“清理运行表”。

### 2.3 外部 SQL 实体不属于删除范围

`EXTERNAL_SQL` DataRef 不要求调用内部表清理，也不阻止运行记录删除。删除事务只能移除 FlowWeaver 保存的引用元数据，不得：

- 删除外部 SQL 表。
- 清空外部 SQL 数据。
- 修改外部数据库文件。
- 调用 SQL 节点或节点执行器。

### 2.4 共享数据和跨运行引用必须先解除

以下情况拒绝删除：

- 该运行生产的共享发布仍为 `PUBLISHED` 或 `RELEASING`。
- 其他运行仍持有该发布的有效读取租约。
- 其他运行的循环记录仍通过外键直接引用该运行的 DataRef。
- 该运行的任意 DataRef 仍有有效表租约。

已经释放的共享发布及其历史成员、租约元数据可以随生产运行一并删除。

### 2.5 第一版只做手动单条删除

本轮不增加自动保留期限、定时清理或批量删除。删除动作必须由用户选择一条运行并二次确认后触发。

## 3. 后端方案

### 3.1 API

新增：

```http
DELETE /api/v1/runs/{workflow_run_id}
```

成功响应：

```json
{
  "workflow_run_id": "run-id",
  "deleted": true
}
```

### 3.2 错误边界

| 情况 | HTTP | error_code |
| --- | --- | --- |
| 运行不存在 | 404 | `WORKFLOW_RUN_NOT_FOUND` |
| 运行不是终态 | 409 | `WORKFLOW_RUN_NOT_TERMINAL` |
| 仍有活跃进程 | 409 | `WORKFLOW_RUN_PROCESS_ACTIVE` |
| 内部表未清理完成 | 409 | `WORKFLOW_RUN_TABLES_NOT_CLEANED` |
| 共享发布或有效租约仍在使用 | 409 | `WORKFLOW_RUN_DELETE_BLOCKED` |
| 其他运行仍直接引用表 | 409 | `WORKFLOW_RUN_DELETE_BLOCKED` |

错误详情只返回有界数量的阻塞 ID 和总数，不把整张目录一次返回给 UI。

### 3.3 事务与删除顺序

后端使用一个 `BEGIN IMMEDIATE` 事务重新检查运行状态和阻塞条件，然后按依赖顺序删除内部元数据：

1. 循环轮次表绑定、循环轮次节点绑定。
2. 循环轮次、循环运行。
3. 节点任务结果输出绑定、节点任务结果、节点任务。
4. 节点运行。
5. 运行事件。
6. 运行配置字快照。
7. 已解除的读取租约、共享发布成员、共享发布。
8. 表租约和该运行的 DataRef 元数据。
9. 输入快照。
10. workflow process。
11. workflow run 主记录。

任一步失败时整个内部元数据事务回滚，不允许出现只删一半的运行记录。

表数据释放仍由既有“清理运行表”负责，不合并进元数据删除事务，避免跨 TableProvider 做伪原子操作。

### 3.4 代码边界

- 删除规则放在独立的后端运行删除模块。
- API 路由只负责参数、错误响应和结果响应。
- `RuntimeStore` 继续作为内部持久化基座。
- 不修改节点注册、节点 schema 或节点执行逻辑。
- 不让 Avalonia 了解数据库表删除顺序。

## 4. Avalonia 方案

运行列表操作区增加“删除运行记录”按钮：

- 仅终态运行可用。
- 运行管理忙碌时不可用。
- 使用 Flyout 二次确认，明确不可恢复且外部 SQL 不受影响。
- 调用后端删除 API，不在本地模拟删除成功。
- 成功后使对应运行缓存失效，刷新当前分页并选择剩余第一条运行。
- 当前页删除最后一条且不是第一页时，回到上一页再刷新。
- 后端返回“表未清理”时展示真实错误，不自动偷偷执行清理。

UI 与后端保持一一对应：

```text
清理运行表按钮 -> table-refs cleanup API
删除运行记录按钮 -> run delete API
```

## 5. 性能与耦合约束

1. 删除前检查使用索引条件和计数，不加载完整运行对象图。
2. 阻塞 ID 最多返回少量样本，响应大小有界。
3. 删除事务不访问 TableProvider，不执行节点，不连接外部数据库。
4. 第一版单条删除，避免一次大批量删除长时间占用 SQLite 写锁。
5. UI 成功后只刷新当前运行页，不全量刷新所有共享页面。
6. 删除逻辑集中在后端运行管理模块，前端只依赖稳定 API。
