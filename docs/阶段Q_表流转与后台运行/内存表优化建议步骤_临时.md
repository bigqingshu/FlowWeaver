# 内存表优化建议步骤（临时）

> 文档状态：临时思路梳理
> 当前用途：记录内存表后续优化方向，便于后续拆成正式执行计划
> 核心结论：内存表继续作为运行内临时快路径；需要预览、回看、后台记录时，按需生成稳定的 runtime sqlite 预览快照。

## 1. 当前问题

内存表当前适合临时中转，但不适合直接作为长期预览记录。

主要原因：

| 问题 | 大白话说明 | 影响 |
| --- | --- | --- |
| rows 在进程内存里 | 表数据存在 workflow process 或 EngineHost 当前进程内 | 进程退出、跨进程、重启后不稳定 |
| TableRef 可能还在，但 rows 不一定在 | 元数据和真实内存 rows 生命周期不完全一致 | UI 下拉可能看到表，但读取 rows 失败 |
| 大表读取会放大内存压力 | 分页、排序、复制时可能产生额外内存占用 | 大表预览或后台运行时风险更高 |
| 当前缺少生命周期策略 | 内存表什么时候释放、保留多久还没有明确机制 | 长时间运行可能堆积内存 |
| 当前不支持完整查询能力 | 例如过滤能力暂不支持 | UI 和数据 API 不能把它当成普通 SQL 表使用 |

## 2. 优化原则

1. 内存表不要伪装成持久表。
2. 内存表只承诺运行内临时使用，不承诺长期回看。
3. 当前表和运行内 SQL 表继续作为数据预览的稳定来源。
4. 需要预览内存表时，在内存表生成时同步生成 runtime sqlite 快照。
5. 快照默认不开启，避免抵消内存表的轻量优势。
6. 不修改 `TableRef` 核心模型来专门服务内存表快照。
7. 第一版优先用 `NodeTaskResult.output_refs` 和 `summary` 记录内存表与快照的关系。
8. 快照失败只记录 warning，不影响原节点主输出和内存表生成。

## 3. 建议步骤

### 步骤 1：先收口表状态说明

目标：让 UI、文档和后端语义先统一。

建议：

- 内存表标记为“临时内存表”。
- 数据预览下拉可以显示内存表，但要提示“不保证可回看”。
- 读取失败时返回友好错误，不让用户误以为工作流主结果损坏。

### 步骤 2：优化内存 provider 的基础性能

目标：先减少现有内存表的额外内存放大。

建议：

- `count_rows` 尽量直接返回行数，不复制整表。
- `read_rows` 尽量只复制当前页 rows。
- 只有用户请求排序时才做排序。
- 对大表读取、排序增加 warning 或保护边界。

### 步骤 3：增加内存表生命周期保护

目标：避免长期运行后内存表无限堆积。

建议：

- 支持释放不再需要的内存表。
- 给单表 row 数或估算大小设置软上限。
- 超出软上限时记录 warning。
- 工作流结束后，根据后续运行策略决定是否释放临时内存表。

### 步骤 4：提供通用预览快照 helper

目标：把“可读 TableRef 复制成 runtime sqlite 快照”做成后端通用能力。

建议流程：

```text
可读 TableRef
-> 按批读取 rows
-> 写入 runtime sqlite staging 表
-> publish 成新的 runtime sqlite TableRef
-> 返回 preview snapshot TableRef
```

注意：

- 必须按批读取和写入。
- 不改变原内存表。
- 不要求 UI 理解节点内部实现。
- 可复用于后续外部 SQL 引用表稳定预览、诊断快照等场景。

### 步骤 5：节点按需接入快照

目标：只有明确需要预览的内存表才生成快照。

建议优先接入：

| 节点或场景 | 建议 |
| --- | --- |
| 保存内存表节点 | 增加可选“生成预览快照”配置，默认关闭 |
| 写入到 `memory_table` 的运行内写入节点 | 增加可选“生成预览快照”配置，默认关闭 |
| 诊断运行 | 可以临时开启目标节点的快照 |
| 普通后台运行 | 默认不开启内存表快照 |

### 步骤 6：记录内存表与快照关系

目标：让 UI 和调试记录能解释“哪个快照来自哪个内存表”。

第一版建议：

| 位置 | 记录内容 |
| --- | --- |
| `NodeTaskResult.output_refs` | 同时包含 memory TableRef 和 snapshot TableRef |
| `NodeTaskResult.summary` | 记录 `memory_table_ref_id`、`preview_snapshot_table_ref_id`、`snapshot_warning` |
| 状态表 | 只记录摘要，不放大保存 rows |

暂不建议第一版新增专用关联表，也不建议修改 `TableRef` 核心字段。

### 步骤 7：数据预览体验收口

目标：用户在预览页面能自然理解内存表状态。

建议：

- 下拉菜单显示 storage kind。
- 内存表显示“临时表”标识。
- 如果存在快照，优先提供“查看快照”。
- 如果没有快照，提示“该内存表只用于运行内临时中转，不保证可预览或回看”。
- 后台运行记录优先展示 runtime sqlite 表和快照表。

### 步骤 8：补充测试

目标：保证内存表优化不会影响主流程。

建议测试：

| 测试点 | 预期 |
| --- | --- |
| 空内存表 | 可生成 TableRef，必要时可生成空快照 |
| 小表分页读取 | 返回正确页数据 |
| 大表分页读取 | 不一次性复制全表 |
| 内存表 rows 不可用 | 返回友好错误 |
| 快照成功 | runtime sqlite 快照可被数据预览读取 |
| 快照失败 | 原节点主输出不失败，只记录 warning |
| 内存表释放 | 释放后读取返回明确不可用 |

## 4. 性能与耦合边界

| 边界 | 说明 |
| --- | --- |
| 默认不全量快照 | 避免内存表失去轻量优势 |
| 快照必须分批 | 避免一次性复制大表 |
| 不改变主输出 | 快照属于预览辅助能力，不改变节点计算结果 |
| 不绑定 UI | UI 只根据 TableRef、summary 和 provider 能力展示 |
| 不改核心模型 | 不为了内存表快照改造 TableRef 通用结构 |
| 不依赖配置字第一版控制 | 配置字只限制已有观测记录；内存表快照会创建新的 runtime sqlite 数据产物，第一版只由节点业务配置或诊断运行入口触发 |

## 5. 推荐实施顺序

```text
1. 内存表状态展示和友好错误
2. MemoryTableProvider 基础性能优化
3. 生命周期和软上限保护
4. 通用 TableRef 快照 helper
5. 保存内存表节点接入可选预览快照
6. 运行内写入到 memory_table 的节点接入可选预览快照
7. UI 下拉菜单优先打开稳定快照
8. 补齐边界测试
```

短期最稳落点：

```text
内存表继续用于运行内临时中转。
需要给用户看的表，尽量使用 runtime sqlite。
需要查看内存表时，在生成时按需快照成 runtime sqlite。
```
