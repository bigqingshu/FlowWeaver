# 工作流表状态规划：内存表、SQL 表、当前表

> 文档状态：前置规划
> 当前用途：梳理工作流运行中不同表状态的定位、预览能力和后续实现方向
> 核心目标：内存表继续保持低成本临时中转；需要预览时，按需快照成稳定的 runtime sqlite 表

## 1. 一句话结论

工作流运行时建议把表分成三类看：

```text
当前表：节点主链路正在传递的表，优先使用 runtime sqlite，默认可预览。
运行内 SQL 表：FlowWeaver 本次运行生成的 runtime sqlite 表，默认适合预览和回看。
外部 SQL 引用表：由 SQL 节点映射进工作流的外部数据库表引用，回看稳定性取决于外部连接和外部数据状态。
内存表：运行内临时中转表，不适合作为长期预览记录。
```

后续如果用户需要在数据预览界面查看内存表，不建议直接依赖内存读取，而是在内存表生成时同步生成一个 runtime sqlite 预览快照。

## 2. 三类表的定位

| 表类型 | 大白话定位 | 当前后端形态 | 是否适合数据预览 | 是否适合长期回看 |
| --- | --- | --- | --- | --- |
| 当前表 | 工作流主链路上传给下一个节点的表 | 通常是 `TableRef(role=CURRENT, storage_kind=RUNTIME_SQL)` | 适合 | 适合，取决于运行数据保留策略 |
| 运行内 SQL 表 | 本次运行生成的辅助表、命名表、快照表 | `sqlite_runtime` provider，落到 `runtime/workflow_runs/{workflow_run_id}.db` | 适合 | 适合，取决于运行数据保留策略 |
| 外部 SQL 引用表 | 外部数据库或外部 sqlite 里的表引用 | `external_sql` provider 或后续外部连接 provider | 视连接状态而定 | 不由 FlowWeaver 保证，除非生成快照 |
| 内存表 | 只给运行中临时用的表 | `memory` provider，rows 在进程内存 | 当前不稳定 | 不适合 |

当前表、运行内 SQL 表、内存表都属于 FlowWeaver 内部数据流转。节点可以通过 `TableRef` / provider 读取、增加、修改、覆盖这些内部表状态，但这不等于操作用户本地文件。

真正读取、写入、覆盖、删除或重命名本地文件时，必须由指定文件类节点或插件节点显式处理。运行内 SQL 表虽然可能落在 FlowWeaver 的 runtime sqlite 文件里，但它是内部运行产物，不是用户选择的业务文件输出。

## 3. 当前事实

### 当前表

当前表是普通节点最重要的输出。它用于：

- 传给下游节点继续处理。
- 被 UI 数据预览读取。
- 被运行记录追溯。

当前表应该优先保持为 runtime sqlite 表，因为它有稳定的 provider，可通过：

```text
GET /api/v1/data/{table_ref_id}/rows
```

分页读取 rows。

### 运行内 SQL 表与外部 SQL 引用表

文档里需要把两类 SQL 形态分开：

| 类型 | 说明 |
| --- | --- |
| 运行内 SQL 表 | FlowWeaver 自己生成的 runtime sqlite 表，稳定、可读、可预览 |
| 外部 SQL 引用表 | 外部数据库或外部 sqlite 连接里的表引用，FlowWeaver 只保存引用，不默认复制全部 rows |

运行内 SQL 表是当前最适合做数据预览、运行回看和调试定位的表形态。

外部 SQL 引用表如果需要稳定预览，也建议生成运行内快照，避免后续连接变化、权限变化或外部数据变化导致回看结果不一致。

真实 SQL 数据库的写入、覆盖、追加、修改等副作用，不进入主程序通用表保存策略，继续由专门 SQL 节点负责。

真实本地文件的读取、写入、覆盖、删除、重命名等副作用，也不进入主程序通用表保存策略，继续由文件读取、文件写入、导出、插件等指定节点负责。

### 内存表

内存表适合：

- 临时中转。
- 运行内轻量辅助结果。
- 不需要长期查看的中间数据。

内存表不适合：

- 后台运行完成后回看。
- EngineHost 重启后回看。
- 跨进程稳定预览。
- 作为用户认为“已经保存”的结果。

原因是内存表 rows 在进程内存里。当前工作流运行通常由 Supervisor 拉起独立 workflow process，数据预览 API 所在的 EngineHost 进程不一定拥有同一份内存。

## 4. 数据预览下拉菜单建议

数据预览下拉菜单可以显示所有有 `TableRef` 元数据的表，但不同表的可加载能力要区分。

| 下拉显示项 | 建议展示 | 点击加载 rows 的行为 |
| --- | --- | --- |
| 当前表 | 当前表 / runtime sqlite / 可预览 | 直接加载 |
| 运行内 SQL 表 | 运行内 SQL / 可预览 | 直接加载 |
| 外部 SQL 引用表 | 外部 SQL / 依赖连接 | 尝试加载，失败时提示连接或 provider 问题 |
| 内存表 | 内存临时表 / 不保证预览 | 如果有快照则引导加载快照；没有快照则提示不可长期预览 |
| 内存表快照 | 内存表预览快照 / runtime sqlite | 直接加载 |

这能避免用户看到内存表后误以为它一定能像普通当前表一样稳定打开。

## 5. 内存表预览快照方案

### 5.1 目标

当用户或运行策略明确需要预览内存表时，把内存表 rows 复制成一个新的 runtime sqlite `TableRef`。

复制后的快照表用于 UI 数据预览，不改变原内存表的业务定位。

```text
memory TableRef
  -> preview snapshot
  -> runtime sqlite TableRef
  -> UI load rows
```

### 5.2 最稳触发时机

最稳触发时机是：

```text
内存表生成时
```

原因是这时节点或 workflow process 一定还拿得到内存表 rows。

不建议第一版只做：

```text
用户后来在 UI 点击内存表
-> API 再尝试从内存 provider 生成快照
```

因为那时 rows 可能已经不在 API 进程里，或者 workflow process 已经退出。

### 5.3 触发方式建议

第一版建议支持两种触发方式，但默认都不要强制全量开启：

| 触发方式 | 建议 | 说明 |
| --- | --- | --- |
| 节点业务配置 | 可选 | 例如内存表相关节点提供“生成预览快照”开关 |
| 诊断运行入口 | 可选 | 排查问题时可只对目标节点生成快照 |

默认不应对所有内存表自动快照，否则会把内存表的性能优势抵消掉。

第一版明确不把“是否生成内存表预览快照”交给配置字控制。配置字只限制日志、事件、进度、metrics、payload 等反馈记录；内存表预览快照会创建新的 runtime sqlite 数据产物，应归节点业务配置或诊断运行入口控制。

### 5.4 后端执行流程

建议流程：

```text
1. 节点生成 memory TableRef。
2. 如果当前策略要求预览快照，立即读取 memory rows。
3. 使用 SQLiteRuntimeTableProvider 创建 staging table。
4. 分批写入 rows。
5. publish 成 runtime sqlite TableRef。
6. 把 memory TableRef 和 snapshot TableRef 都登记到 RuntimeStore。
7. 在 NodeTaskResult summary 或状态表中记录二者关系。
8. UI 下拉菜单显示内存表和它的预览快照。
```

关系记录建议先放轻：

| 记录位置 | 第一版建议 | 原因 |
| --- | --- | --- |
| `NodeTaskResult.output_refs` | 可以同时包含内存表和快照表 | 不需要新增数据库表 |
| `NodeTaskResult.summary` | 记录 `memory_table_ref_id -> preview_snapshot_table_ref_id` | UI 和调试容易解释 |
| 新增专用关联表 | 暂不建议第一版做 | 会扩大主程序存储耦合 |
| 修改 `TableRef` 核心字段 | 暂不建议 | 避免污染通用表引用模型 |

### 5.5 快照命名建议

快照表建议使用清晰的 logical name：

```text
{memory_logical_table_id}__preview_snapshot
```

如果需要多次生成，可追加版本或短 id：

```text
{memory_logical_table_id}__preview_snapshot_v1
```

UI 展示可以用中文：

```text
内存表快照：xxx
```

## 6. 状态流转规划

### 6.1 普通主链路

```text
节点输入当前表
-> 节点处理
-> 输出新的当前表
-> runtime sqlite TableRef
-> 可预览
```

这是默认主链路，应保持稳定。

### 6.2 当前表保存为运行内 SQL 表

```text
当前表
-> 保存中转数据 / 运行内写入节点
-> run_table
-> runtime sqlite TableRef
-> 可预览、可回看
```

适合用户明确想保留的中间结果。

### 6.3 当前表保存为内存表

```text
当前表
-> 保存内存表 / 内存目标写入
-> memory TableRef
-> 运行内临时使用
```

默认不承诺长期预览。

如果需要预览：

```text
memory TableRef
-> 同步生成 preview snapshot
-> runtime sqlite TableRef
-> UI 读取快照
```

### 6.4 外部 SQL 引用表进入工作流

```text
external SQL reference
-> 读取或映射
-> 当前表 / runtime sqlite snapshot
-> 后续节点处理
```

如果只是引用外部 SQL 引用表，回看时可能受外部数据变化影响。

如果要稳定回看，应在进入工作流时生成 runtime sqlite 快照。

## 7. 配置字关系

配置字可以用于任何工作流运行状态下的调试和观测，但内存表快照会复制真实 rows，成本比普通日志、事件和 payload 控制更高。

建议边界：

| 项 | 是否建议归配置字控制 | 说明 |
| --- | --- | --- |
| 是否显示更多事件 | 是 | 纯观测行为 |
| 是否保留 metrics | 是 | 诊断记录 |
| 是否生成内存表预览快照 | 否 | 会创建新的 runtime sqlite 数据产物，不属于配置字的反馈过滤范围 |
| 节点是否写外部 SQL | 否 | 属于节点业务配置或副作用配置 |
| 当前表是否输出 | 否 | 属于工作流正确性 |

第一版把“生成内存表预览快照”放在相关节点业务配置或诊断运行入口里，不放进配置字。后续如果需要工作流级调试开关，也应作为独立运行策略评估，不能混入“只做减法”的配置字模型。

## 8. 性能与耦合约束

1. 默认不对所有内存表自动快照。
2. 快照必须分批写入，避免一次性把大表塞进内存。
3. 快照只生成新的 runtime sqlite `TableRef`，不改变原内存表。
4. 不修改 `TableRef` 核心模型来专门服务内存表快照。
5. 第一版不新增专用关联表，优先用 `output_refs` 和 summary 表达关系。
6. UI 只按 `TableRef` 和 provider 能力展示，不直接理解节点内部实现。
7. 失败时不影响原节点主输出，最多记录“快照生成失败”的 warning。
8. 后台运行默认保持少记录，只有用户明确需要预览或诊断时才生成快照。
9. 内存表预览快照第一版不归配置字控制，避免配置字变成数据产物生成开关。
10. 当前表、运行内 SQL 表、内存表只作为内部数据流转，不自动写入用户本地文件。
11. 本地文件操作必须由指定节点显式配置路径、格式、覆盖策略和失败处理。

## 9. 分阶段实施建议

### 阶段 1：状态展示收口

目标：UI 和文档先把表状态讲清楚。

任务：

- 数据预览下拉显示 storage kind。
- 内存表显示“临时内存表”提示。
- 内存表无法读取 rows 时给出友好提示。

### 阶段 2：后端快照 helper

目标：提供通用的“任意可读 TableRef -> runtime sqlite 快照”能力。

任务：

- 增加按批读取 provider rows 的 helper。
- 增加写入 runtime sqlite staging/published 表的 helper。
- 输出新的 snapshot TableRef。
- 覆盖小表、空表、大表分批、读取失败测试。

### 阶段 3：内存表生成时按需快照

目标：内存表需要预览时，同步生成 runtime sqlite 快照。

任务：

- 保存内存表节点支持可选预览快照。
- 写入类节点的 `memory_table` 目标支持可选预览快照。
- `NodeTaskResult.summary` 记录内存表和快照表关系。
- 快照失败只给 warning，不改变内存表本身是否生成。

### 阶段 4：数据预览体验完善

目标：用户可以从下拉菜单自然找到稳定快照。

任务：

- 内存表旁显示是否有预览快照。
- 有快照时提供“查看快照”入口。
- 没有快照时说明该表是运行内临时表。
- 后台运行列表中优先展示可稳定读取的 runtime sqlite 表。

## 10. 当前推荐落点

短期推荐：

```text
当前表和运行内 SQL 表继续作为数据预览主来源。
内存表可以出现在下拉菜单，但标记为临时表。
需要预览的内存表，在生成时同步生成 runtime sqlite 快照。
UI 实际加载快照表，不直接依赖内存表 rows。
```

这样可以同时保留内存表的轻量优势，又不破坏数据预览和后台回看的稳定性。
