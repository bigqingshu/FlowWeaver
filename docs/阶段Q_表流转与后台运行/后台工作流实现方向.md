# 后台工作流实现方向

更新时间：2026-07-08

## 当前状态是什么？

回答：

当前还没有完整的“后台工作流功能”。

更准确地说，当前已经有后端运行工作流的基础链路，但还没有面向用户的后台工作流管理能力。

当前手动运行工作流时，已经不是 UI 自己执行节点，而是走后端统一链路：

```text
UI 发起运行
-> POST /api/v1/workflows/{workflow_id}/runs
-> 创建 WorkflowRun
-> Supervisor 拉起 workflow_process
-> workflow_process 执行节点
-> 写入 NodeRun / RuntimeEvent / TableRef / runtime sqlite rows
```

所以目前缺的不是“后端能不能跑”，而是：

```text
后台触发、后台列表、后台运行策略、后台通知、后台清理入口
```

## 后台工作流能不能另起一套执行机制？

回答：

不建议。

后台工作流必须复用现有主链路：

```text
WorkflowRun
Supervisor
workflow_process
NodeRun
NodeTask
NodeTaskResult
RuntimeEvent
TableRef
runtime sqlite
```

如果后台工作流另起一套执行机制，会带来几个问题：

- 配置字不会自然生效。
- 数据预览要重做一套读取逻辑。
- 运行状态、错误、取消、恢复都要重复实现。
- 节点执行行为可能和手动运行不一致。
- 后续调试会分裂成两套系统。

所以后台工作流只能是“不同入口、同一执行层”。

## 后台工作流应该新增什么？

回答：

后台工作流主要新增的是运行入口和管理能力，不是新增节点执行内核。

建议新增能力如下：

| 能力 | 说明 |
| --- | --- |
| 后台触发入口 | 定时、手动加入后台、后续可扩展事件触发 |
| 后台运行列表 | 查看后台 run 的状态、开始时间、结束时间、结果 |
| 后台运行筛选 | 按工作流、状态、时间、run mode、触发来源筛选 |
| 后台通知 | 运行完成、失败、取消后通知 UI 刷新 |
| 后台取消 | 复用当前 workflow process cancel 机制 |
| 后台重试 | 基于已有 workflow revision 重新创建 run |
| 手动清理 | 用户手动删除旧运行记录和运行输出数据 |

这些能力都应围绕现有 `WorkflowRun` 记录展开。

## 配置字怎么接入后台工作流？

回答：

后台工作流不需要一套新的配置字机制。

后台运行创建 `WorkflowRun` 后，进入同一个 `workflow_process`。配置字会在 workflow process 启动时从 workflow revision 中解析：

```text
WorkflowDefinition.runtime_options
-> system defaults + workflow + node overrides
-> resolved runtime options
-> RuntimeOptionsEventSink
```

所以只要后台运行复用当前执行链路，配置字就能自然控制：

- runtime events
- progress
- metrics
- payload 限制
- 脱敏
- 事件限流
- 后续日志等级

关键约束：

```text
后台入口不能绕过 workflow_process。
```

## 数据记录是否和手动运行一致？

回答：

应该一致。

后台运行也应该写入同一套运行产物：

```text
WorkflowRun
NodeRun
NodeTaskResult
RuntimeEvent
TableRef
runtime sqlite rows
```

UI 不需要自己保存后台数据。UI 只需要查询后端记录：

```text
GET /api/v1/runs
GET /api/v1/runs/{workflow_run_id}/table-refs
GET /api/v1/data/{table_ref_id}/rows
```

当前普通输出表是持久化到 runtime sqlite 的，不是运行完成后立刻销毁。

当前清理策略先按用户手动清理，不由配置字自动删除业务输出表。

## run_mode 应该怎么设计？

回答：

现有 `run_mode` 已有：

```text
full
preview_to_node
```

后台运行不一定需要新增一个完全不同的 `run_mode`。

更稳的做法是：

```text
run_mode 仍表达执行范围；
trigger_source 或 run_source 表达是谁触发的。
```

例如：

| 字段 | 用途 |
| --- | --- |
| `run_mode=full` | 执行完整工作流 |
| `run_mode=preview_to_node` | 执行到指定节点 |
| `trigger_source=manual` | 用户手动运行 |
| `trigger_source=background_manual` | 用户加入后台运行 |
| `trigger_source=schedule` | 定时触发 |
| `trigger_source=api` | API 触发 |

这样不会把“运行范围”和“触发来源”混在一起。

第一版如果不想改数据库字段，也可以先在 API/UI 层按入口区分，后续再补 `trigger_source`。

## 后台运行和预览运行有什么区别？

回答：

区别不在执行内核，而在入口和使用目的。

| 类型 | 执行链路 | 主要目的 | 数据记录 |
| --- | --- | --- | --- |
| 手动完整运行 | 同一套 workflow process | 用户立即运行完整流程 | 持久化记录 |
| 预览到节点 | 同一套 workflow process | 查看某个节点的上游结果 | 持久化记录 |
| 后台运行 | 同一套 workflow process | 不盯着 UI，后台执行 | 持久化记录 |

所以后台运行不是“低配版运行”，也不是“只记录日志不保存表”。

后台运行只是触发方式和 UI 展示方式不同。

## 第一版后台工作流建议范围

回答：

第一版建议只做最小闭环：

```text
用户选择工作流
-> 加入后台运行
-> 创建正常 WorkflowRun
-> Supervisor 启动 workflow_process
-> UI 可在后台运行列表查看状态
-> 运行完成后 UI 可查看 TableRef 和数据预览
-> 用户可手动清理旧运行
```

第一版不建议做：

- 复杂调度规则。
- 分布式执行。
- 独立后台执行器。
- 配置字之外的新诊断体系。
- 自动删除业务输出表。
- 后台专用数据预览表。

## 分阶段实现方向

### 阶段 1：后台运行入口

目标：

让用户可以把一次完整工作流运行放到后台执行。

实现方向：

- 复用现有 `POST /api/v1/workflows/{workflow_id}/runs`。
- 后端仍创建普通 `WorkflowRun`。
- 仍由 `Supervisor.start_workflow_process()` 启动。
- UI 不阻塞等待完成，只记录 run id 并刷新后台列表。

验收：

- 后台运行能产生正常 `WorkflowRun`。
- 配置字按 workflow revision 生效。
- 能查到 RuntimeEvent、NodeRun、TableRef。

### 阶段 2：后台运行列表

目标：

让用户能查到过去和正在运行的后台任务。

实现方向：

- 复用 `WorkflowRun` 查询。
- UI 增加后台运行列表。
- 支持按状态筛选：运行中、成功、失败、取消。
- 显示 workflow 名称、run id、开始时间、结束时间、状态。

验收：

- UI 不需要保存后台运行结果。
- 关闭 UI 后重新打开，仍能从后端记录恢复列表。

### 阶段 3：结果查看与数据预览

目标：

后台运行完成后，用户可以查看输出表。

实现方向：

- 复用现有 TableRef 列表接口。
- 复用现有 data rows 接口。
- 优先展示 runtime sqlite 表。
- 内存表只提示临时表；如需查看，后续走预览快照方案。

验收：

- 后台 run 的输出表能在数据预览中打开。
- 不新增第二套数据预览记录机制。

### 阶段 4：取消、重试、手动清理

目标：

后台运行具备基本运维操作。

实现方向：

- 取消复用现有 workflow process cancel。
- 重试基于原 workflow revision 创建新的 run。
- 清理由用户手动触发。
- 清理时同时处理运行记录、TableRef 元数据和 runtime sqlite 文件，具体实现需单独方案。

验收：

- 取消后状态一致。
- 重试不会覆盖旧 run。
- 用户能主动清理旧运行产物。

### 阶段 5：定时触发

目标：

在最小后台运行稳定后，再增加定时能力。

实现方向：

- 增加 schedule 定义。
- 到点后创建普通 `WorkflowRun`。
- 仍由 Supervisor 启动 workflow process。
- 定时触发只改变触发来源，不改变执行内核。

验收：

- 定时创建的 run 和手动 run 使用同一套记录、配置字和数据预览机制。

## 性能与耦合约束

1. 后台运行不能绕过 `WorkflowRun -> Supervisor -> workflow_process`。
2. 后台运行不能新增第二套节点执行器。
3. 后台运行不能新增第二套数据预览记录。
4. 配置字只限制已有反馈，不控制业务输出清理。
5. runtime sqlite 输出表默认可被 UI 查询。
6. 清理先按用户手动触发，不自动删除业务输出。
7. 后台运行列表查询应分页，避免一次加载大量历史 run。
8. 后台通知只提示状态变化，不携带大表数据。
9. 内存表不作为长期后台预览来源。
10. 定时触发只能创建普通 run，不能绕开主执行链路。

## 当前一句话结论

后台工作流目前还没有作为独立功能完整实现。

但现有手动运行已经具备后端进程执行、运行记录、事件、TableRef、runtime sqlite 持久化和配置字解析基础。

后续后台工作流应只补“触发和管理层”，不要重做执行层。
