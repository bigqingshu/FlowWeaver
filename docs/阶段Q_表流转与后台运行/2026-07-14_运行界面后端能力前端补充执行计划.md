## 5. 性能与耦合硬边界

### 5.1 禁止选中运行后一次加载所有数据

选择运行时不得同时无条件加载：

```text
review
+ 全部 NodeRun
+ 全部 TableRef
+ 全部 RuntimeEvent
+ 全部循环轮次
+ 表 rows
```

推荐加载规则：

| 数据 | 加载时机 |
| --- | --- |
| 运行基础信息 | 当前运行被选中后，可轻量刷新 |
| review | 概览页签首次激活或用户点击刷新时 |
| NodeRun | 第一页按需加载，后续分页 |
| 循环 | 循环页签激活后加载第一页 |
| TableRef | 用户进入 Data/DataPreview 或点击查看表后加载 |
| RuntimeEvent | 用户进入 Logs 或点击查看日志后加载 |
| 表 rows | 用户明确选择可读表后分页加载 |

### 5.2 所有异步选择链必须防止旧响应覆盖

运行、节点、循环和轮次切换必须同时具备：

- `CancellationTokenSource` 替换。
- request version 检查。
- 返回前再次核对当前 ID。
- 被取消请求不得覆盖 Message、Error、Items 或 SelectedItem。

### 5.3 节点运行必须使用后端分页

不得继续依赖当前非分页 `ListNodeRunsAsync()` 作为运行页正式数据源。

第一版建议：

```text
page size = 100
previous / next
status filter
保留原始 status 作为 API 参数
本地化 status 只用于显示
```

### 5.4 不复制共享页面状态

运行页跳转表或日志时应复用：

- 当前 `TableRefs` / DataPreview 状态链。
- 当前 Logs 过滤字段和刷新命令。
- 当前 AppShell 页面导航状态。

不得新建第二套运行专用表预览缓存或第二套运行专用日志集合。

### 5.5 运行审查必须按需加载

当前 review 响应会包含节点和表目录信息。第一版不得在用户快速划过运行列表时连续请求 review。

必须满足：

- 只对稳定选中的运行加载。
- 快速切换时取消上一请求。
- 同一个终态运行可以缓存一次结果。
- 活跃运行只在明确刷新或相关事件触发后失效。
- 不用定时轮询弥补事件链。

### 5.6 子 ViewModel 只依赖窄服务

新状态模型优先遵循现有 `RunLoopMonitorViewModel` 模式：

```text
RunOverviewViewModel -> IRunReviewService
NodeRunMonitorViewModel -> IRunTableDirectoryService
MainWindow bridge -> 只同步连接、selected run、导航和顶层通知
```

不得让新的子 ViewModel 直接依赖完整 `MainWindowViewModel`，也不在本文中启动 page-level ViewModel 全量迁移。
