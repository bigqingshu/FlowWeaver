## 1. 当前结论

阶段 Q 的第一轮后端基础已经落地，后续重点不再是“能不能表达表流转”，而是把已落地能力收成稳定底座，再逐步推广到更多节点和后台运行管理。

后续主线建议为：

```text
后端表流转 helper 收口
-> 普通纯表节点批量接入
-> 数据预览表目录能力增强
-> 后台运行最小闭环
-> 配置字真实执行加固
```

这样可以保持低耦合：主程序继续负责 TableRef、槽位绑定、表目录、分页查询和 provider 基础能力；节点继续负责具体表操作语义。

## 3. 前端 UI 当前边界

当前前端 UI 仍在重构阶段，本计划后续执行时默认不直接修改前端 UI 代码。

前端相关内容在本文中只作为后续契约保留，作用是让后端实现时提前稳定字段、接口和表目录含义，方便前端重构完成后按契约接入。

当前执行边界如下：

| 边界 | 说明 |
| --- | --- |
| 不同步改 UI | 后端阶段执行时不修改 Avalonia / UI 文件，避免和前端重构产生竞态 |
| 保留 UI 计划 | 输入槽位、输出目标、表目录、后台列表、配置字窗口等 UI 内容只记录为后续计划 |
| 后端先稳定契约 | 后端先稳定 API、TableRef、slot binding、summary、分页 rows 等契约 |
| 前端后接入 | 等前端重构稳定后，再按后端契约渲染表下拉、输出目标、数据预览和后台列表 |

因此，本文后续所有提到“UI 应显示”“UI 可查询”“UI 后续按契约渲染”的内容，均表示未来接入要求，不表示当前阶段要修改 UI 代码。

## 4. 后续阶段一：表流转 helper 收口

### 问题

当前能力已经能跑通，但仍分散在不同节点和不同 helper 中。后续如果批量接入节点，容易出现每个节点自己判断输入表、输出目标、覆盖方式和 summary 字段的情况。

### 目标

形成稳定的后端节点表流转底座，让普通节点接入时只关心自己的业务计算。

### 动作

- 统一输入槽位 helper：
  - `require_input_slot(...)`
  - `iter_slot_batches(...)`
  - 缺槽位、错 storage kind、不可读表时给出明确错误。
- 统一输出目标 helper：
  - 当前表输出。
  - 新建内存表。
  - 新建运行内 SQL 表。
  - 已有内存表 / 运行内 SQL 表目标。
- 统一写入结果 summary：
  - `output_slot`
  - `target_type`
  - `target_table`
  - `target_table_ref_id`
  - `write_mode`
  - `affected_rows`
- 保持主程序只提供基础能力，不进入覆盖、追加、合并等节点业务语义。

### 验收

- 新节点接入表输入 / 输出时不需要重复写大量 provider 判断。
- 旧默认 CURRENT 链路保持不变。
- 单元测试覆盖 helper 成功路径和错误路径。

### 提交建议

`后端: 收口节点表流转辅助能力`

## 5. 后续阶段二：普通纯表节点批量接入

### 问题

当前只是试点节点打通了新表流转能力，大部分普通纯表节点仍主要依赖默认输入和默认当前表输出。

### 目标

让低风险普通节点逐步支持：

```text
选择输入表
-> 处理数据
-> 输出当前表或另存目标表
```

### 优先节点范围

| 优先级 | 节点类型 | 原因 |
| --- | --- | --- |
| 高 | 筛选、删除行、删除列、重命名列、复制列 | 纯表转换，行为稳定，适合批量接入 |
| 高 | 合并列、提取文本、替换文本、填充类 | 业务边界清楚，容易验证输入输出 |
| 中 | 去重、展开/转置类 | 可能涉及较多数据量，需要注意批处理 |
| 中 | 写入类节点 | 有副作用语义，继续单独加固 |
| 后置 | 控制类、循环类、子工作流类 | 调度语义更复杂，不作为第一批批量接入 |

### 动作

- 为每个低风险节点补输入槽位读取。
- 为每个低风险节点补输出目标选择。
- 默认不配置输出目标时，行为保持旧的当前表输出。
- 节点 summary 统一记录实际读取和输出的表。
- 补充节点级定向测试。

### 验收

- 普通纯表节点可选择上游内存表或运行内 SQL 表作为输入。
- 普通纯表节点可额外输出到内存表或运行内 SQL 表。
- 默认旧工作流测试不回归。

### 提交建议

`后端: 扩展纯表节点表输入输出能力`

## 6. 后续阶段三：数据预览表目录增强

### 问题

当前数据预览可以通过 run 的 TableRef 和 data rows API 查询数据，但后续 UI 需要更清楚地知道每张表是什么、来自哪个节点、属于哪个槽位、是否适合长期回看。

### 目标

增强后端表目录信息，让 UI 后续只查询后端记录，不自己保存运行数据。

### 动作

- 在 run table refs 查询中保持返回 TableRef 完整信息。
- 明确或补充表目录展示字段：
  - 表类型：当前表、内存表、运行内 SQL 表、外部 SQL 引用表。
  - 来源节点。
  - 输出槽位。
  - logical table id。
  - lifecycle status。
  - 是否可分页读取。
- 保持 rows API 分页读取。
- 不把大表 rows 放进事件、日志或 run 列表。
- 覆盖已有表后，预览只显示当前最新状态，不承诺同名表历史版本。

### 验收

- UI 能通过后端返回信息构建表下拉和数据预览列表。
- 大表预览仍通过 offset / limit 分页。
- 内存表不被误认为后台长期可回看的稳定数据源。

### 提交建议

`后端: 增强运行表目录查询契约`

## 7. 后续阶段四：后台运行最小闭环

### 问题

当前已有普通 WorkflowRun、workflow process 和 `trigger_source`，但还缺面向用户的后台运行管理层。

### 目标

后台运行只新增触发和管理能力，不新增第二套执行器，不新增第二套数据预览记录。

### 动作

- 后台运行入口：
  - 创建普通 WorkflowRun。
  - `run_mode` 继续表示执行范围：`full` 或 `preview_to_node`。
  - `trigger_source` 表示触发来源：例如 `background_manual`。
  - 仍由 supervisor 启动 workflow process。
- 后台运行列表：
  - 支持按 workflow、status、run_mode、trigger_source 查询。
  - 必须分页。
- 后台取消：
  - 复用现有 workflow process cancel。
- 后台结果查看：
  - 复用 run table refs API。
  - 复用 data rows API。
- 后台重试：
  - 基于 workflow revision 重新创建 run。
  - 不复用旧运行中的 TableRef。

### 验收

- 后台运行和手动运行走同一执行链路。
- 后台运行产生的 runtime SQL 表能通过同一套表目录和 rows API 查询。
- 后台列表不携带大表数据。
- UI 不需要自己保存后台运行结果。

### 提交建议

`后端: 增加后台运行管理入口`

## 8. 后续阶段五：配置字真实执行加固

### 问题

配置字的方向已经明确，但后续需要确认主程序和节点真实按配置字降低反馈量，而不是只保存配置。

### 目标

配置字只控制运行反馈强度，不控制表产物。

### 动作

- workflow 级配置字和 node 级配置字合并后传入运行过程。
- 主流程按配置字控制：
  - 事件详细程度。
  - 进度刷新频率。
  - metrics 记录粒度。
  - 诊断 payload 详细程度。
  - 日志等级。
- 节点按配置字控制：
  - summary 详细程度。
  - warning / diagnostic 记录量。
  - 进度与指标反馈频率。
- 明确不可被配置字关闭的基础能力：
  - 节点开始 / 结束 / 失败状态。
  - 必要错误信息。
  - TableRef 登记。
  - 表产物生成。

### 验收

- `normal`、`background_fast`、`diagnostic` 三类配置有可观察差异。
- 后台快速配置下事件和诊断 payload 减少。
- diagnostic 配置下错误上下文更完整。
- 配置字不会改变节点是否生成表、覆盖表或删除表。

### 提交建议

`后端: 加固运行配置字执行效果`

## 9. 性能与耦合硬边界

| 边界 | 要求 |
| --- | --- |
| 不传整表 | 主程序、节点、UI 之间继续传 TableRef / slot binding，不传 rows |
| 预览分页 | 数据预览 rows API 必须保持 offset / limit |
| 事件轻量 | runtime events 不携带大表数据 |
| 后台列表分页 | 后台运行列表不能一次返回大量历史 run |
| 内存表临时 | 内存表适合作为运行内中转，不作为长期后台回看主来源 |
| 表策略节点化 | 新建、覆盖、追加、更新、合并由节点决定 |
| 文件副作用节点化 | 本地文件写入、删除、重命名只由指定节点或插件节点处理 |
| 外部 SQL 写入节点化 | 真实外部 SQL 修改只由 SQL 节点处理 |
| 配置字不管表产物 | 配置字不控制表是否生成、覆盖、删除、快照 |
| 前端契约驱动 | UI 后续按节点槽位和表目录渲染，不写节点专属业务逻辑 |
| 当前不改前端 | 前端 UI 正在重构，阶段 Q 后续执行默认只改后端和文档 |

## 10. 推荐下一步

阶段 Q 第一版冻结后的独立方向是：

```text
定时/事件后台调度
外部 SQL 通用执行与事务边界
真实跳转、子工作流和复杂循环
内存表显式快照与引用存活分析
Avalonia 页面级 MVVM 继续迁移
```

这些方向不应继续追加到阶段 Q 收口提交中，应分别定义目标、测试矩阵和停止条件。
