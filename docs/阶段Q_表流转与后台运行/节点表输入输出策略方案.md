# 节点表输入输出策略方案

> 文档状态：前置方案整理
> 当前用途：收口节点可选择不同表进行处理、并可将处理结果另存为中转表的需求
> 核心目标：保持工作流按节点顺序运行，同时让复杂工作流可以在当前表、内存表、运行内 SQL 表之间进行低耦合中转
> 核心边界：节点管理表操作，主程序只提供表映射、表状态登记和基础读写能力
> 外部边界：本地文件和外部资源的真实读写必须由指定节点显式处理

## 1. 需求范围

本方案只覆盖以下 5 种情况：

| 编号 | 需求 | 方案语义 |
| --- | --- | --- |
| 1 | 输出默认当前表，当前表不可命名 | 普通节点默认输出新的当前表，继续沿工作流主链路传递；当前表名由系统管理，用户不手动命名 |
| 2 | 输出另存为内存表 | 节点处理完成后，可以额外保存一份内存表；内存表允许用户命名 |
| 3 | 输出另存为运行内 SQL 表 | 节点处理完成后，可以额外保存一份运行内 SQL 表；运行内 SQL 表允许用户命名，适合稳定预览和回看 |
| 4 | 输入源选择当前表 | 默认行为，节点继续处理上游当前表 |
| 5 | 输入源选择上游内存表 / 运行内 SQL 表 / 外部 SQL 引用表 | 节点可以从上游已经产生的内存表、运行内 SQL 表或外部 SQL 引用表中选择一张作为处理输入 |

## 2. 明确不进入本方案的边界

真实 SQL 数据库的读取和修改不由主程序通用输入输出策略直接负责。

推荐边界：

```text
真实 SQL 读取
-> 由 SQL 读取 / SQL 映射节点完成
-> 映射成工作流内 TableRef
-> 后续普通节点按 TableRef 处理

真实 SQL 修改
-> 由 SQL 保存 / SQL 修改节点完成
-> 节点自己负责连接、事务、覆盖、追加、回滚和错误提示

主程序
-> 只负责调度节点、传递 TableRef、记录运行状态
-> 不直接管理真实数据库写入
```

因此，本方案中的“输出另存为运行内 SQL 表”只指 FlowWeaver 运行内的 runtime sqlite 表，不指外部真实数据库表。

同理，本方案中的当前表、内存表、运行内 SQL 表只属于工作流内部数据流转，不代表真实本地文件。

```text
当前表 / 内存表 / 运行内 SQL 表
-> 内部表状态
-> 节点通过 TableRef / provider 自行维护、增加、修改、覆盖

本地文件 / 外部资源
-> 外部副作用
-> 必须由文件类节点、SQL 节点或插件节点显式执行
```

主程序不会因为节点输出了某张表，就自动写入、覆盖、删除或重命名用户本地文件。需要把内部表导出为 xlsx、csv、json、docx 或其他真实文件时，应使用对应导出 / 写入节点。

术语统一如下：

| 术语 | 含义 |
| --- | --- |
| 运行内 SQL 表 | FlowWeaver 本次运行生成的 runtime sqlite 表，后端 `storage_kind=RUNTIME_SQL` |
| 外部 SQL 引用表 | 由 SQL 读取 / SQL 映射节点映射进工作流的外部数据库表引用，通常 `storage_kind=EXTERNAL_SQL` |
| 真实 SQL 写入 | 对外部真实数据库执行保存、追加、覆盖、修改等副作用，只由专门 SQL 节点负责 |
| 本地文件操作 | 对用户文件执行读取、写入、覆盖、删除、重命名等副作用，只由指定文件类节点或插件节点负责 |

## 3. 基础概念

| 概念 | 大白话说明 | 后端语义 |
| --- | --- | --- |
| 当前表 | 工作流主链路正在往下传的表 | `TableRole.CURRENT` |
| 内存表 | 运行内临时中转表，轻量但不保证长期回看 | `TableStorageKind.MEMORY` |
| 运行内 SQL 表 | 当前工作流运行产生的稳定中转表 | `TableStorageKind.RUNTIME_SQL` |
| 外部 SQL 引用表 | 由 SQL 节点映射进工作流的表引用 | 通常是 `TableStorageKind.EXTERNAL_SQL` |
| 输出另存 | 在不打断当前表主链路的前提下，额外保存一份辅助表 | 通常输出 `TableRole.AUXILIARY` |

## 4. 推荐运行逻辑

### 4.1 默认运行逻辑

默认情况下，工作流保持原来的主链路语义：

```text
上游 CURRENT
-> 节点处理
-> 输出新的 CURRENT
-> 下游节点继续处理
```

这保证普通工作流不需要理解多表中转，也不会因为新增能力改变旧工作流行为。

### 4.2 选择上游辅助表作为输入

当节点配置里选择了输入源时，节点运行前先解析输入源：

```text
上游节点 output_refs
-> 根据节点配置里的稳定输入源选择器选择目标 TableRef
-> 将选中的 TableRef 作为当前节点 input_refs
-> 节点处理该输入表
-> 输出新的 CURRENT
```

选择范围建议只限于“上游已产生的表”，不建议第一版支持任意历史表。

工作流定义里不应保存运行期 `table_ref_id`。`TableRefId` 是每次运行后生成的引用，同一个工作流下次运行时会变化，不能作为稳定配置。

第一版建议使用稳定选择器：

```text
source_node_instance_id
output_role
storage_kind
logical_table_id / output_alias
```

运行时再根据上游节点已经完成的 `output_refs` 解析出真实 `TableRefId`。

原因：

- 不破坏 DAG 依赖关系。
- 不需要让主程序全局扫描所有历史表。
- 可以避免节点在目标表还没生成时提前运行。
- 后台运行和回放更容易解释。

### 4.3 输出另存为中转表

普通表处理节点处理完成后，仍然默认输出当前表；如果用户开启另存，则主程序为节点提供额外中转表的目标映射：

```text
节点处理结果
-> CURRENT：继续作为主链路输出
-> 可选 MEMORY：另存为内存表
-> 可选 RUNTIME_SQL：另存为运行内 SQL 表
```

推荐第一版采用“额外保存副本”，不要让另存目标替代当前表。

这里的“另存”不是主程序替节点执行表操作。主程序只负责把输出保存配置解析成可写目标，节点负责真正的新建、覆盖、追加、按主键更新或合并等表操作语义。

这样做的好处是：

- 不打断工作流主链路。
- 下游默认逻辑仍然简单。
- 用户需要多表中转时可以显式选择辅助表。
- 当前表、内存表、运行内 SQL 表的职责清楚。

## 5. 节点配置建议

建议在节点配置区域增加两个通用配置块，并由节点声明驱动 UI 渲染：

```text
输入表
输出目标
```

节点声明自己需要几个输入表、每个表的名称、说明和可接受类型。UI 按声明显示对应行和下拉菜单。

示例：

```text
输入表
main_table / 主数据表        [下拉选择表]
rules_table / 规则表         [下拉选择表]

输出目标
result_table / 结果表        [当前表 / 新建内存表 / 新建运行SQL表 / 已有表]
```

主程序提供表目录和可写目标，不决定节点如何操作表。

### 5.1 输入源配置

建议字段：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `input_source.type` | enum | `current` | 可选 `current`、`upstream_table` |
| `input_source.source_node_instance_id` | string | 空 | 当选择上游表时，记录来源节点实例 ID |
| `input_source.output_role` | enum | 空 | 可选 `CURRENT`、`AUXILIARY`、`SHARED`，用于缩小选择范围 |
| `input_source.storage_kind` | enum | 空 | 可选 `MEMORY`、`RUNTIME_SQL`、`EXTERNAL_SQL`，用于区分内存表、运行内 SQL 表和外部 SQL 引用表 |
| `input_source.logical_table_id` | string | 空 | 稳定逻辑表名，用于在来源节点的输出表中匹配目标表 |
| `input_source.output_alias` | string | 空 | 后续可选字段，用于节点定义声明的稳定输出别名 |

第一版 UI 可以让用户从上游已知表中选择，但写入工作流配置时应保存稳定选择器，不保存运行期 `table_ref_id`。

运行时后端最终根据稳定选择器解析出真实 `TableRefId`，再写入当前节点的 `NodeTask.input_refs`。

UI 下拉建议按类型显示：

| 类型 | 显示 |
| --- | --- |
| 当前表 | `当前表: 来源节点 / 当前状态` |
| 内存表 | `内存表: 表名` |
| 运行内 SQL 表 | `运行SQL: 表名` |
| 外部 SQL 引用表 | `SQL: 数据库名 / 表名` |

如果外部 SQL 只有连接别名，显示为 `SQL: 连接名 / 表名`。

本地文件不作为普通表目录里的可写基座展示。文件路径、覆盖策略、编码、格式、sheet 名、Word 表格位置等配置应留在指定文件类节点自己的业务配置里。

### 5.2 输出保存配置

建议字段：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `output_save.enabled` | boolean | `false` | 是否在当前表之外额外保存一份中转表 |
| `output_save.target_type` | enum | 空 | 开启另存时可选 `memory_table`、`run_table` |
| `output_save.table_name` | string | 空 | 当目标是内存表或运行内 SQL 表时可填写 |
| `output_save.sync_with_input` | boolean | `true` | 输入源变化时，输出保存类型和名称可默认跟随输入源 |

限制：

- 当前表永远是普通表处理节点的默认输出，不作为 `output_save.target_type`。
- `output_save.enabled=false` 时只输出当前表。
- `memory_table` 和 `run_table` 允许填写表名。
- `output_save.enabled=true` 时输出当前表，并额外保存一份辅助表。
- 输出目标只表达“写到哪里”，新建、覆盖、追加、按主键更新、合并等写法由节点内部决定。
- 外部真实 SQL 不作为普通输出保存目标，真实 SQL 写入继续由专门 SQL 节点负责。
- 本地文件不作为普通输出保存目标，真实文件写入继续由指定文件类节点负责。

## 6. 输入源与输出保存的同步逻辑

为了减少复杂工作流里的重复配置，建议增加一个轻量同步逻辑：

```text
用户选择输入表
-> 如果 output_save.sync_with_input=true
-> 输出保存开关、类型和表名可以默认跟随输入表
-> 输出保存表名默认跟随输入表 logical_table_id
-> 用户仍然可以手动修改输出保存类型和表名
```

示例：

| 输入源 | 默认输出保存建议 | 用户可改为 |
| --- | --- | --- |
| 当前表 | 不开启另存，只输出当前表 | 内存表、运行内 SQL 表 |
| 上游内存表 `temp_a` | 开启另存为内存表 `temp_a` | 运行内 SQL 表、改名后的内存表、不另存 |
| 上游运行内 SQL 表 `stage_a` | 开启另存为运行内 SQL 表 `stage_a` | 内存表、改名后的运行内 SQL 表、不另存 |
| 外部 SQL 引用表 `db_orders` | 开启另存为运行内 SQL 表 `db_orders` | 内存表、改名后的运行内 SQL 表、不另存 |

注意：

- 同步只是默认填充，不代表强制覆盖。
- 用户手动修改后，应停止自动覆盖该字段，避免 UI 反复改掉用户输入。
- 当前表始终不可命名。
- 另存目标只允许内存表或运行内 SQL 表。

## 7. 表名和选择器冲突规则

`logical_table_id` 是用户可理解的逻辑表名，不保证在整个 workflow run 内全局唯一。

第一版建议按以下规则收口：

| 场景 | 规则 |
| --- | --- |
| 不同节点产生同名表 | 允许；UI 展示时必须带来源节点，后端选择时必须带 `source_node_instance_id` |
| 同一节点同一次输出产生同名辅助表 | 不允许；节点或后端 helper 应返回明确错误 |
| 工作流定义中的输入源 | 使用稳定选择器，不保存运行期 `table_ref_id` |
| UI 下拉展示 | 建议显示“来源节点 / 表名 / 表类型 / 角色” |
| 后端运行解析 | 在来源节点成功输出中匹配选择器；匹配 0 个或多个都应报明确错误 |

推荐选择器最小组合：

```text
source_node_instance_id
+ logical_table_id
+ storage_kind
+ output_role
```

后续如果节点定义能声明稳定 `output_alias`，可以优先用 `output_alias` 消除同名歧义。

## 8. 后端职责划分

### 8.1 主程序负责

- 按节点顺序和 DAG 依赖运行。
- 收集上游节点已经输出的 TableRef。
- 根据节点配置里的稳定选择器解析当前节点实际输入表。
- 将解析后的输入表放入 `NodeTask.input_refs`。
- 保持普通表处理节点默认当前表传递逻辑。
- 将输出保存配置解析成节点可用的输出目标映射。
- 提供 UI 下拉所需的表目录，包括当前表、内存表、运行内 SQL 表和外部 SQL 引用表。
- 提供内存表和运行内 SQL 表的可写基座。
- 提供基础表读取、写入、事务或原子替换能力。
- 登记节点最终产出的 `TableRef` 和表状态。
- 不决定新建、覆盖、追加、按主键更新、合并等表操作语义。
- 不把内部表输出自动转换为本地文件读写。

### 8.2 节点负责

- 处理传入的输入表。
- 普通表处理节点输出处理后的当前表。
- 源节点、控制节点、专门写入节点、状态节点等特殊节点按节点定义声明输出角色。
- 根据自身业务决定新建、覆盖、追加、按主键更新或合并。
- 自行处理字段对齐、主键、冲突策略和业务校验。
- 调用底层表接口完成事务化写入或原子替换。
- 不直接关心 UI 如何选择输入表。
- 不把真实 SQL 数据库写入混进普通输出保存逻辑。
- 不把本地文件写入混进普通表输出逻辑，文件副作用应由指定文件类节点显式实现。

### 8.3 SQL 相关节点负责

- 从真实数据库读取表或查询结果。
- 将真实数据库结果映射为工作流内 TableRef。
- 对真实数据库执行保存、追加、覆盖、修改等副作用。
- 自己处理连接、事务、失败回滚和错误提示。

## 9. 分阶段落地建议

### 阶段 1：协议和文档先收口

目标：先统一输入源和输出保存的配置结构。

任务：

- 明确 `input_source` 和 `output_save` 的字段。
- 明确节点声明输入槽位和输出槽位，UI 按槽位渲染表选择区。
- 明确表目录显示规则：当前表、内存表、运行内 SQL 表、外部 SQL 引用表。
- 明确输入源保存稳定选择器，不保存运行期 `table_ref_id`。
- 明确当前表不可命名。
- 明确真实 SQL 数据库只由 SQL 节点处理。
- 明确本地文件和外部资源只由指定节点显式处理。
- 明确第一版只允许选择上游已产生的表。

### 阶段 2：后端输入源解析

目标：让节点可以选择不同上游表作为输入。

任务：

- 在任务提交前收集上游节点 output_refs。
- 默认继续选择 `TableRole.CURRENT`。
- 当配置选择上游表时，使用稳定选择器解析目标表，并校验该表确实来自上游。
- 将选中的 TableRef ID 写入当前节点 `input_refs`。
- 对找不到表、匹配多个表、表不可读、表尚未生成等情况返回明确错误。

### 阶段 3：输出另存为内存表 / 运行内 SQL 表

目标：主程序提供输出目标映射，节点处理结果可额外保存为中转表。

任务：

- 普通表处理节点继续输出当前表。
- 开启另存时，主程序把目标解析为内存表或运行内 SQL 表可写目标。
- 节点调用目标映射和基础写入能力，额外生成内存表或运行内 SQL 表。
- 另存表使用 `TableRole.AUXILIARY`。
- `NodeTaskResult.output_refs` 同时包含当前表和另存表。
- `summary` 记录当前表、另存表和保存目标。

### 阶段 4：UI 配置区域接入

目标：用户可以在节点配置里选择输入源和输出保存。

任务：

- 节点配置下方显示“输入源”。
- 节点配置下方显示“输出保存”。
- UI 根据节点声明的输入 / 输出槽位生成表选择控件。
- 输入源下拉只展示上游可用表。
- 表下拉按 `当前表`、`内存表`、`运行SQL`、`SQL` 分组或加前缀展示。
- 当前表不作为另存目标，也不显示表名输入。
- 内存表和运行内 SQL 表启用表名输入。
- 输入源变化时，按同步逻辑默认填充输出保存。

### 阶段 5：边界测试

目标：保证新能力不破坏旧工作流。

任务：

- 默认配置下旧工作流行为不变。
- 当前表输出不可命名。
- 工作流配置中不持久化运行期 `table_ref_id`。
- 可选择上游内存表作为输入。
- 可选择上游运行内 SQL 表作为输入。
- 可选择 SQL 节点映射出的外部 SQL 引用表作为输入。
- 可另存为内存表。
- 可另存为运行内 SQL 表。
- 找不到指定上游表时给出明确错误。
- 选择器匹配多个同名表时给出明确错误。

## 10. 性能与耦合约束

| 约束 | 说明 |
| --- | --- |
| 不全局扫描任意历史表 | 第一版只允许选择上游已产生表，避免调度和查询复杂化 |
| 不在工作流配置中保存 `table_ref_id` | `TableRefId` 是运行期引用，必须通过稳定选择器解析 |
| 不改变默认当前表链路 | 普通工作流不需要额外配置 |
| 不让另存替代当前表 | 另存只作为辅助副本，避免下游断链 |
| 不把真实 SQL 写入放进主程序 | 真实 SQL 副作用继续由专门 SQL 节点处理 |
| 不把本地文件操作放进主程序 | 文件读取、写入、覆盖、删除、重命名继续由指定节点处理 |
| 内部表不等于本地文件 | 当前表、内存表、运行内 SQL 表只作为内部数据流转 |
| 不把表操作语义放进主程序 | 新建、覆盖、追加、按主键更新、合并都由节点决定 |
| UI 不理解节点业务 | UI 只按节点声明的槽位渲染名称、说明和表下拉 |
| 不要求每个节点手写重复逻辑 | 输入源解析和输出另存应尽量做成统一 helper 或运行层能力 |
| 内存表不承诺长期回看 | 需要稳定预览时优先另存为运行内 SQL 表 |

## 11. 推荐短期落点

```text
默认仍然处理当前表。
节点可从上游输出表中选择输入源，但配置保存稳定选择器。
普通表处理节点处理后继续输出当前表。
特殊节点按节点定义声明输出角色。
需要中转时，额外另存为内存表或运行内 SQL 表。
主程序只负责输入源解析、输出目标映射、表状态登记和基础读写能力。
主程序给 UI 提供表目录，给节点提供内存表和运行内 SQL 表的可写基座。
UI 按节点输入 / 输出槽位显示名称、说明和表下拉菜单。
新建、覆盖、追加、按主键更新、合并等表操作语义由节点内部执行。
真实 SQL 数据库读写继续交给专门 SQL 节点。
本地文件读写、覆盖、删除、重命名继续交给指定文件类节点或插件节点。
```

这个落点能满足复杂工作流的多表中转需求，同时把主程序改动控制在输入源解析、输出目标映射、表状态登记、基础表接口和 UI 配置展示这几块。
