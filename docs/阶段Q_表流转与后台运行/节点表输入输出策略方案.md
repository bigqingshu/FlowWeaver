# 节点表输入输出策略方案

> 文档状态：前置方案整理
> 当前用途：收口节点可选择不同表进行处理、并可将处理结果另存为中转表的需求
> 核心目标：保持工作流按节点顺序运行，同时让复杂工作流可以在当前表、内存表、运行内 SQL 表之间进行低耦合中转

## 1. 需求范围

本方案只覆盖以下 5 种情况：

| 编号 | 需求 | 方案语义 |
| --- | --- | --- |
| 1 | 输出默认当前表，当前表不可命名 | 普通节点默认输出新的当前表，继续沿工作流主链路传递；当前表名由系统管理，用户不手动命名 |
| 2 | 输出另存为内存表 | 节点处理完成后，可以额外保存一份内存表；内存表允许用户命名 |
| 3 | 输出另存为运行内 SQL 表 | 节点处理完成后，可以额外保存一份运行内 SQL 表；运行内 SQL 表允许用户命名，适合稳定预览和回看 |
| 4 | 输入源选择当前表 | 默认行为，节点继续处理上游当前表 |
| 5 | 输入源选择上游内存表 / SQL 表 | 节点可以从上游已经产生的内存表、运行内 SQL 表或已映射进工作流的 SQL 表中选择一张作为处理输入 |

## 2. 明确不进入本方案的边界

真实 SQL 数据库的读取和修改不由主程序通用输入输出策略直接负责。

推荐边界：

```text
真实 SQL 读取
-> 由 SQL 读取 / SQL 映射节点完成
-> 映射成工作流内 TableRef
-> 后续普通节点按 TableRef 处理

真实 SQL 修改
-> 由 SQL 保存 / SQL 修改节点完成
-> 节点自己负责连接、事务、覆盖、追加、回滚和错误提示

主程序
-> 只负责调度节点、传递 TableRef、记录运行状态
-> 不直接管理真实数据库写入
```

因此，本方案中的“输出另存为运行内 SQL 表”只指 FlowWeaver 运行内的 runtime sqlite 表，不指外部真实数据库表。

## 3. 基础概念

| 概念 | 大白话说明 | 后端语义 |
| --- | --- | --- |
| 当前表 | 工作流主链路正在往下传的表 | `TableRole.CURRENT` |
| 内存表 | 运行内临时中转表，轻量但不保证长期回看 | `TableStorageKind.MEMORY` |
| 运行内 SQL 表 | 当前工作流运行产生的稳定中转表 | `TableStorageKind.RUNTIME_SQL` |
| 上游 SQL 表 | 由 SQL 节点映射进工作流的表引用 | 通常是 `EXTERNAL_SQL` 或后续 SQL provider 的 TableRef |
| 输出另存 | 在不打断当前表主链路的前提下，额外保存一份辅助表 | 通常输出 `TableRole.AUXILIARY` |

## 4. 推荐运行逻辑

### 4.1 默认运行逻辑

默认情况下，工作流保持原来的主链路语义：

```text
上游 CURRENT
-> 节点处理
-> 输出新的 CURRENT
-> 下游节点继续处理
```

这保证普通工作流不需要理解多表中转，也不会因为新增能力改变旧工作流行为。

### 4.2 选择上游辅助表作为输入

当节点配置里选择了输入源时，节点运行前先解析输入源：

```text
上游节点 output_refs
-> 根据节点配置选择目标 TableRef
-> 将选中的 TableRef 作为当前节点 input_refs
-> 节点处理该输入表
-> 输出新的 CURRENT
```

选择范围建议只限于“上游已产生的表”，不建议第一版支持任意历史表。

原因：

- 不破坏 DAG 依赖关系。
- 不需要让主程序全局扫描所有历史表。
- 可以避免节点在目标表还没生成时提前运行。
- 后台运行和回放更容易解释。

### 4.3 输出另存为中转表

节点处理完成后，仍然默认输出当前表；如果用户开启另存，则额外保存一份中转表：

```text
节点处理结果
-> CURRENT：继续作为主链路输出
-> 可选 MEMORY：另存为内存表
-> 可选 RUNTIME_SQL：另存为运行内 SQL 表
```

推荐第一版采用“额外保存副本”，不要让另存目标替代当前表。

这样做的好处是：

- 不打断工作流主链路。
- 下游默认逻辑仍然简单。
- 用户需要多表中转时可以显式选择辅助表。
- 当前表、内存表、运行内 SQL 表的职责清楚。

## 5. 节点配置建议

建议在节点配置区域增加两个通用配置块：

```text
输入源
输出保存
```

### 5.1 输入源配置

建议字段：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `input_source.type` | enum | `current` | 可选 `current`、`upstream_table` |
| `input_source.table_ref_id` | string | 空 | 当选择上游表时记录目标表引用 |
| `input_source.logical_table_id` | string | 空 | 可选辅助显示，用于 UI 展示表名 |
| `input_source.storage_kind` | enum | 空 | 可选辅助显示，用于 UI 标识内存表、运行内 SQL 表、SQL 映射表 |

第一版 UI 可以让用户从上游已知表中选择，后端最终以 `table_ref_id` 为准。

### 5.2 输出保存配置

建议字段：

| 字段 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `output_save.mode` | enum | `current_only` | 可选 `current_only`、`save_copy` |
| `output_save.target_type` | enum | `current` | 可选 `current`、`memory_table`、`run_table` |
| `output_save.table_name` | string | 空 | 当目标是内存表或运行内 SQL 表时可填写 |
| `output_save.sync_with_input` | boolean | `true` | 输入源变化时，输出保存类型和名称可默认跟随输入源 |

限制：

- `target_type=current` 时不允许填写表名。
- `memory_table` 和 `run_table` 允许填写表名。
- `current_only` 时只输出当前表。
- `save_copy` 时输出当前表，并额外保存一份辅助表。

## 6. 输入源与输出保存的同步逻辑

为了减少复杂工作流里的重复配置，建议增加一个轻量同步逻辑：

```text
用户选择输入表
-> 如果 output_save.sync_with_input=true
-> 输出保存类型默认跟随输入表类型
-> 输出保存表名默认跟随输入表 logical_table_id
-> 用户仍然可以手动修改输出保存类型和表名
```

示例：

| 输入源 | 默认输出保存建议 | 用户可改为 |
| --- | --- | --- |
| 当前表 | 当前表 | 内存表、运行内 SQL 表 |
| 上游内存表 `temp_a` | 内存表 `temp_a` | 当前表、运行内 SQL 表、改名后的内存表 |
| 上游运行内 SQL 表 `stage_a` | 运行内 SQL 表 `stage_a` | 当前表、内存表、改名后的运行内 SQL 表 |
| 上游 SQL 映射表 `db_orders` | 运行内 SQL 表 `db_orders` | 当前表、内存表、改名后的运行内 SQL 表 |

注意：

- 同步只是默认填充，不代表强制覆盖。
- 用户手动修改后，应停止自动覆盖该字段，避免 UI 反复改掉用户输入。
- 如果输出目标是当前表，则当前表仍然不可命名。

## 7. 后端职责划分

### 7.1 主程序负责

- 按节点顺序和 DAG 依赖运行。
- 收集上游节点已经输出的 TableRef。
- 根据节点配置解析当前节点实际输入表。
- 将解析后的输入表放入 `NodeTask.input_refs`。
- 保持默认当前表传递逻辑。
- 对节点输出结果做通用另存处理，或提供统一 helper 给节点调用。

### 7.2 节点负责

- 处理传入的输入表。
- 输出节点处理后的当前表。
- 不直接关心 UI 如何选择输入表。
- 不把真实 SQL 数据库写入混进普通输出保存逻辑。

### 7.3 SQL 相关节点负责

- 从真实数据库读取表或查询结果。
- 将真实数据库结果映射为工作流内 TableRef。
- 对真实数据库执行保存、追加、覆盖、修改等副作用。
- 自己处理连接、事务、失败回滚和错误提示。

## 8. 分阶段落地建议

### 阶段 1：协议和文档先收口

目标：先统一输入源和输出保存的配置结构。

任务：

- 明确 `input_source` 和 `output_save` 的字段。
- 明确当前表不可命名。
- 明确真实 SQL 数据库只由 SQL 节点处理。
- 明确第一版只允许选择上游已产生的表。

### 阶段 2：后端输入源解析

目标：让节点可以选择不同上游表作为输入。

任务：

- 在任务提交前收集上游节点 output_refs。
- 默认继续选择 `TableRole.CURRENT`。
- 当配置选择上游表时，校验该表确实来自上游。
- 将选中的 TableRef ID 写入当前节点 `input_refs`。
- 对找不到表、表不可读、表尚未生成等情况返回明确错误。

### 阶段 3：输出另存为内存表 / 运行内 SQL 表

目标：节点处理结果可额外保存为中转表。

任务：

- 普通节点继续输出当前表。
- 开启另存时，额外生成内存表或运行内 SQL 表。
- 另存表使用 `TableRole.AUXILIARY`。
- `NodeTaskResult.output_refs` 同时包含当前表和另存表。
- `summary` 记录当前表、另存表和保存目标。

### 阶段 4：UI 配置区域接入

目标：用户可以在节点配置里选择输入源和输出保存。

任务：

- 节点配置下方显示“输入源”。
- 节点配置下方显示“输出保存”。
- 输入源下拉只展示上游可用表。
- 当前表目标禁用表名输入。
- 内存表和运行内 SQL 表启用表名输入。
- 输入源变化时，按同步逻辑默认填充输出保存。

### 阶段 5：边界测试

目标：保证新能力不破坏旧工作流。

任务：

- 默认配置下旧工作流行为不变。
- 当前表输出不可命名。
- 可选择上游内存表作为输入。
- 可选择上游运行内 SQL 表作为输入。
- 可选择 SQL 节点映射出的上游 SQL 表作为输入。
- 可另存为内存表。
- 可另存为运行内 SQL 表。
- 找不到指定上游表时给出明确错误。

## 9. 性能与耦合约束

| 约束 | 说明 |
| --- | --- |
| 不全局扫描任意历史表 | 第一版只允许选择上游已产生表，避免调度和查询复杂化 |
| 不改变默认当前表链路 | 普通工作流不需要额外配置 |
| 不让另存替代当前表 | 另存只作为辅助副本，避免下游断链 |
| 不把真实 SQL 写入放进主程序 | 真实 SQL 副作用继续由专门 SQL 节点处理 |
| 不要求每个节点手写重复逻辑 | 输入源解析和输出另存应尽量做成统一 helper 或运行层能力 |
| 内存表不承诺长期回看 | 需要稳定预览时优先另存为运行内 SQL 表 |

## 10. 推荐短期落点

```text
默认仍然处理当前表。
节点可从上游输出表中选择输入源。
节点处理后永远继续输出当前表。
需要中转时，额外另存为内存表或运行内 SQL 表。
真实 SQL 数据库读写继续交给专门 SQL 节点。
```

这个落点能满足复杂工作流的多表中转需求，同时把主程序改动控制在输入源解析、输出另存和 UI 配置展示这三块。
