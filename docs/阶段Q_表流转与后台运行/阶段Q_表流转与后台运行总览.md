# 阶段Q：表流转与后台运行总览

> 文档状态：阶段 Q 第一版冻结总控
> 当前用途：统一本阶段多个 MD 的实现顺序、依赖关系和硬边界
> 核心目标：先收口节点处理哪张表、输出目标如何映射，再处理内存表、配置字和后台工作流

## 0. 2026-07-12 当前状态

阶段 Q 第一版代码边界已经冻结。当前已完成：稳定输入选择器、节点表槽位、通用输出目标、显式写入结果、结果 binding 目录、分页数据预览、真实串行循环、手动后台运行管理、配置字动态反馈控制、共享表生命周期、有界 run 清理、内存表分页优化和软阈值告警。

本文后续 Q0 至 Q7 章节保留阶段设计顺序，其中以下内容仍属于独立后续完整化，不应误读为第一版缺陷：

- 内存表显式 RUNTIME_SQL 预览快照。
- 定时或事件触发的后台调度。
- 普通节点通用外部 SQL 读取和真实外部写入。
- 真实跳转、真实子工作流、嵌套或并行循环。
- Avalonia 页面级 MVVM 继续迁移。

本轮执行与验证记录以 `2026-07-12_阶段Q当前未收口项源码审计与执行方案.md` 第 0 节为准。

## 1. 阶段主线

阶段 Q 的主线是：

```text
节点表输入输出策略
-> 表状态和内存表边界
-> 配置字反馈控制
-> 后台工作流触发与管理
```

这里的顺序很重要。节点如何选择输入表、如何拿到输出目标映射、如何登记输出表状态，是后续内存表预览、配置字降噪、后台运行回看的前提。

## 3. 推荐实施顺序

### Q0：术语和选择器收口

目标：先避免后续实现用错概念。

任务：

- 明确“运行内 SQL 表”只指 runtime sqlite。
- 明确“外部 SQL 引用表”只指 SQL 节点映射进工作流的外部表引用。
- 明确真实 SQL 写入只由专门 SQL 节点处理。
- 明确本地文件和外部资源读写只由指定节点处理。
- 明确当前表、内存表、运行内 SQL 表只属于内部数据流转。
- 明确工作流配置不保存运行期 `table_ref_id`。
- 明确输入源使用稳定选择器。

验收：

- 文档和代码字段不再把运行内 SQL 表、外部 SQL 引用表、真实 SQL 写入混在一起。
- 文档和代码字段不把内部表输出误认为本地文件写入。

### Q1：节点输入源稳定选择器

目标：让节点可以选择上游已产生的目标表作为输入。

任务：

- 增加输入源配置结构。
- 使用 `source_node_instance_id + logical_table_id + storage_kind + output_role` 解析目标表。
- 默认继续使用上游 `TableRole.CURRENT`。
- 找不到表、匹配多个表、表不可读时返回明确错误。

验收：

- 旧工作流默认行为不变。
- 节点可选择上游内存表、运行内 SQL 表、外部 SQL 引用表作为输入。
- 工作流定义中不持久化运行期 `table_ref_id`。

### Q2：输出目标映射与基础表接口

目标：普通表处理节点继续输出当前表，同时主程序为节点提供可选中转表目标映射和基础表读写接口。

任务：

- 使用 `output_save.enabled` 控制是否另存。
- `output_save.target_type` 只允许 `memory_table`、`run_table`。
- 当前表不作为另存目标，也不可命名。
- 另存表使用 `TableRole.AUXILIARY`。
- 处理表名冲突，尤其是同一节点同一次输出的重复逻辑表名。
- 为多表输出预留输出槽位模型。
- 明确主程序只提供输出目标映射和基础读写能力，不替节点决定表操作语义。
- 明确主程序提供表目录：当前表、内存表、运行内 SQL 表、外部 SQL 引用表。
- 明确主程序提供可写基座：内存表和运行内 SQL 表。

验收：

- 普通表处理节点默认继续输出当前表。
- 开启另存后，节点可通过目标映射产出当前表和另存表。
- `NodeTaskResult.output_refs` 能登记当前表和另存表。
- `summary` 能解释当前表和另存表的关系。
- UI 能按表类型显示 `当前表`、`内存表`、`运行SQL`、`SQL`。

### Q2.5：多表输入输出与覆盖表

目标：支持插件节点和复杂节点按槽位接收多张表、输出多张表。

任务：

- 节点定义声明输入槽位和输出槽位。
- UI 按节点槽位声明显示名称、说明和表下拉菜单。
- 用户给每个输入槽位选择当前表、内存表、运行内 SQL 表或外部 SQL 引用表。
- 用户给每个输出槽位选择当前表、新建表或已有表，主程序生成目标映射。
- 输出到已有表时表名不可改。
- 覆盖、追加、按主键更新、合并等表操作语义由节点内部执行。
- 覆盖已有表时，节点应通过事务或原子替换能力完成“清空旧数据，再写入新数据”。
- 规划删除表节点，用于删除或释放运行内中转表，删除语义也归节点。

验收：

- 插件节点可以收到按槽位命名的多个 TableRef。
- 多输出节点可以生成多个输出槽位。
- UI 不需要理解插件业务，只按槽位声明渲染输入表和输出目标。
- 覆盖表不会混入旧数据，且失败时不留下空表或半张表。
- 数据预览能展示表当前状态、来源节点、输出槽位和最近覆盖信息。

### Q3：内存表基础优化

目标：先让内存表作为临时中转更稳。

任务：

- 优化 `count_rows` 和分页读取，减少额外复制。
- 增加内存表释放和软上限建议。
- 内存表不可用时返回友好错误。
- UI 明确标记“临时内存表”。

验收：

- 内存表不再被误认为长期持久表。
- 大表读取不会无意义放大内存压力。

### Q4：内存表预览快照

目标：需要预览内存表时，生成稳定 runtime sqlite 快照。

任务：

- 增加通用 `可读 TableRef -> runtime sqlite 快照` helper。
- 按批读取和写入。
- 在 `NodeTaskResult.output_refs` 和 `summary` 记录内存表与快照关系。
- 快照失败只记录 warning，不影响主输出。
- 快照第一版不归配置字控制，避免配置字变成数据产物生成开关。

验收：

- 有快照时 UI 优先打开 runtime sqlite 快照。
- 没有快照时 UI 提示内存表不保证长期预览。

## 4. 配置字阶段

### Q5：配置字反馈控制

目标：让配置字真实控制主流程记录和反馈量。

任务：

- 工作流定义保存 `runtime_options`。
- UI 支持工作流级配置字和节点覆盖。
- 运行时合并 `system defaults + workflow + node override`。
- 控制 runtime events、progress、metrics、payload、脱敏和限流。

验收：

- 配置字只限制已有反馈，不改变节点核心计算。
- 配置字不关闭当前表输出。
- 配置字不控制真实 SQL 写入。
- 配置字不生成内存表预览快照。
- 配置字不自动清理业务输出表。

## 5. 后台工作流阶段

### Q6：后台运行入口和列表

目标：补后台运行的触发和管理层，不重做执行层。

任务：

- 后台入口继续创建普通 `WorkflowRun`。
- 继续由 `Supervisor -> workflow_process` 执行。
- 后台列表查询后端运行记录。
- 结果查看复用 TableRef 和 data rows API。

验收：

- 后台运行、手动运行、预览运行复用同一执行链路。
- 后台运行产生的 runtime sqlite 表可被 UI 查询。
- 内存表不作为后台长期预览来源。

### Q7：清理和长期维护

目标：为运行记录和运行输出提供用户可控的维护入口。

任务：

- 用户手动清理旧运行记录。
- 清理时同步处理 TableRef 元数据和 runtime sqlite 文件。
- 后台列表分页，避免加载大量历史记录。
- 后台通知只携带状态变化，不携带大表数据。

验收：

- 清理不由配置字自动触发。
- 清理操作不影响仍在运行的 workflow process。

## 6. 硬边界

| 边界 | 说明 |
| --- | --- |
| 不保存运行期 `table_ref_id` 到工作流定义 | 输入源必须使用稳定选择器 |
| 不让另存替代当前表 | 另存只作为辅助副本 |
| 不把真实 SQL 写入放进主程序通用保存策略 | 真实 SQL 副作用继续由专门 SQL 节点负责 |
| 不把本地文件操作放进主程序通用表策略 | 文件读取、写入、覆盖、删除、重命名必须由指定节点显式处理 |
| 内部表只服务工作流数据流转 | 当前表、内存表、运行内 SQL 表由节点通过 TableRef / provider 自行维护 |
| 主程序不维护表操作语义 | 新建、覆盖、追加、按主键更新、合并都由节点决定 |
| 主程序只提供表映射和基础读写能力 | 主程序负责把槽位和输出目标解析为 TableRef 或可写目标 |
| 主程序提供表目录和可写基座 | 表目录给 UI 选择；内存表和运行内 SQL 表给节点创建或写入 |
| UI 由节点槽位声明驱动 | UI 只渲染槽位名称、说明、下拉菜单和输出目标，不理解节点业务 |
| 不把配置字变成业务功能开关 | 配置字只限制记录和反馈 |
| 不用配置字生成内存表预览快照 | 快照会创建新的 runtime sqlite 数据产物，应归节点业务配置或诊断运行入口 |
| 不让后台运行绕过主执行链路 | 后台只新增触发和管理层 |
| 不新增第二套数据预览记录机制 | UI 继续查询后端 `TableRef` 和 provider rows |
| 不把内存表承诺为长期可回看表 | 稳定预览优先 runtime sqlite 或快照 |
| 多表输入不传整表 rows | 主程序只传槽位名和 `TableRef`，节点或插件按需分页读取 |
| 多表选择器不全局扫历史表 | 第一版只解析上游已完成节点输出，按来源节点缩小范围 |
| 覆盖已有表必须原子化 | 节点调用事务或原子替换能力，避免空表和半张表 |
| 配置字不控制多表数据产物 | 多表输入、覆盖、新建、删除和快照不属于配置字范围 |

## 7. 当前推荐落点

阶段 Q 第一版当前落点：

```text
节点声明、编辑配置、真实执行和结果目录一致
-> 大表分页、清理和告警有界
-> 手动后台、配置字和共享表复用同一运行链
-> 第一版边界冻结
```

下一轮不再继续扩大“阶段 Q 收口”范围。定时后台、外部 SQL、复杂控制流、内存快照和页面级 MVVM 应分别建立新目标，避免重新耦合已经稳定的表流转主线。
