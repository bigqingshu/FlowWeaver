# FlowWeaver WORKFLOW-MGMT-4：阶段验收复核

## 范围

本复核只覆盖 Avalonia 前端工作流管理入口：

- 删除工作流。
- 导出工作流。
- 导入为新工作流。
- 三类管理操作与工作流详情、草稿 dirty、revision 冲突状态的交互。

本阶段不修改后端 API、不新增后端导入导出协议、不处理运行数据或表数据导入导出。

## 完成矩阵

| 能力 | 当前状态 | 说明 |
| --- | --- | --- |
| 删除工作流 | 已完成 | 使用现有 `DeleteWorkflowAsync`，删除前有确认弹窗，成功后清理选中项和详情区域。 |
| 导出工作流 | 已完成 | 使用现有 `GetWorkflowAsync` 读取已保存定义，生成 `flowweaver.workflow.v1` JSON 并通过桌面保存入口写出。 |
| 导入工作流 | 已完成 | 通过桌面打开文件入口读取 JSON，校验 `export_format` 后使用 `CreateWorkflowAsync` 创建新工作流。 |
| 导入身份边界 | 已完成 | 不复用原 `workflow_id`、`revision_id`、`version`，只复用 `workflow.name` 和 `workflow.definition`。 |
| 按钮状态 | 已完成 | 导入只依赖 EngineHost 可用和工作流区非忙碌；导出、删除还要求选中 ACTIVE 工作流。 |
| 错误提示 | 已完成 | 覆盖取消、文件不可读、非法 JSON、不支持格式、缺字段、后端失败。 |
| 本地化 | 已完成 | 简体中文和 English 均有导入、导出、删除相关状态文案。 |

## 交互复核

### 工作流详情自动加载

- 选中已有工作流时仍会触发既有详情自动加载逻辑。
- 导入成功后刷新列表并选中新工作流。
- 导入成功后可手动刷新详情并继续进入编辑、保存、预览和运行入口。

### 草稿 dirty 状态

- 草稿 dirty 不阻断导入、导出、删除。
- 运行和预览仍遵循既有规则：存在 dirty 草稿时会先保存草稿再运行或预览。
- 导出语义保持为导出当前已保存的工作流详情，不导出未保存草稿。

### Revision 冲突状态

- revision 冲突会继续阻断草稿保存、节点编辑、运行和预览。
- revision 冲突不阻断导入、导出、删除这三类工作流管理动作。
- 删除当前选中工作流后，详情与节点选择会被清理，避免继续显示旧工作流内容。

## 测试复核

新增或补强的前端测试覆盖：

- 管理按钮在 EngineHost 未连接、未选中工作流、非 ACTIVE 工作流下的启用状态和禁用原因。
- 导入、导出、删除在 dirty 草稿和 revision 冲突下的独立状态。
- 导入成功后新工作流可以加载详情，并继续具备编辑、保存、预览和运行入口。
- 导入文件解析支持当前导出格式，并拒绝非法 JSON、不支持格式和缺少 definition 的文件。
- 简体中文和英文导入、导出核心文案可加载和格式化。

## 明确不在本阶段支持

- 覆盖导入到已有工作流。
- 导入/导出 WorkflowRun、NodeRun、RuntimeEvent。
- 导入/导出 TableRef、SharedPublication、外部 SQL 文件或运行数据库。
- 导入前对目标环境节点注册表兼容性做完整预检查。
- 导入时合并或重写节点配置中的外部资源路径。

## 下一步建议

进入工作流管理后的体验复核或小步修正：

1. 手动 smoke：导出一个真实工作流，再导入为新工作流，确认新工作流可加载详情、预览选中节点并完整运行。
2. 若真实使用中导入失败信息仍偏技术化，再补一层用户友好错误映射。
3. 若后续要支持覆盖导入，先单独设计 `WORKFLOW-MGMT-5`，明确覆盖策略、revision 冲突和草稿保留规则。
